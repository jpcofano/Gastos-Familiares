# F6.9.10 — Fix lectura `entrantes`: `resource == null` en chequeo de existencia (dependiente puede subir)

Bug encontrado en la prueba en vivo de F6.9.9: el dependiente (Federico) no puede
subir un archivo nuevo. Tira `evaluation error at L99:24 for 'get', Null value error`.
El admin (Juan) sube bien.

**No es regresión de F6.9.9** (que tocó solo `comprobantes:92`). La regla de lectura
de `entrantes` (L99) ya estaba scopeada de antes; el bug es latente y recién aparece
con la **primera subida como dependiente**.

## Causa raíz
`src/datos/entrantes.ts:44` — `subirEntrante` hace dedup con `getDoc(entrantes/{hash})`
**antes** de crear. Para un archivo nuevo el doc no existe ⇒ en la regla de read
`resource` es `null` ⇒ `resource.data.creadoPor` revienta ("Null value error").
- **Admin:** `esAdmin()` es `true` ⇒ el `||` cortocircuita ⇒ nunca evalúa `resource.data` ⇒ funciona.
- **Dependiente:** `esAdmin()` es `false` ⇒ evalúa `resource.data.creadoPor` sobre `resource` null ⇒ error.

## Regla de oro (hacer primero)
Clone fresco de `origin`. Confirmar que L99 sigue siendo la línea de `allow read` de
`entrantes` (con F6.9.9 aplicado localmente; si todavía no commiteaste F6.9.9, validá
el número de línea real antes de aplicar — buscá `match /entrantes`).

## Cambio 1 — `firestore.rules`: permitir read de doc inexistente

BUSCAR:
```
    match /entrantes/{hash} {
      allow read:   if esMiembro() && (esAdmin() || resource.data.creadoPor == miMemberId());
```
REEMPLAZAR:
```
    match /entrantes/{hash} {
      allow read:   if esMiembro() && (esAdmin() || resource == null || resource.data.creadoPor == miMemberId());
```

Por qué es seguro:
- `resource == null` solo aplica al **get de un doc inexistente** (el chequeo de dedup).
  Devuelve "not found" limpio en vez de tirar; `subirEntrante` sigue al create.
- Para docs que **sí existen**, el scoping sigue intacto: el entrante de otro miembro se
  sigue denegando (`resource.data.creadoPor == miMemberId()` falso).
- La **query de lista** (`suscribirEntrantes`, `where('creadoPor','==',memberId)`) evalúa
  la regla por doc matcheado; esos siempre existen ⇒ `resource` nunca es null ⇒ no se afloja nada.
- El `create` (L100-103) ya exige `request.resource.data.creadoPor == miMemberId()`, así
  que un dependiente no puede crear a nombre de otro. Sin cambios ahí.

## Cambio 2 — `docs/CLAUDE.md`: changelog

Sumar línea: fix de lectura de `entrantes` — `resource == null` agregado a la rama no-admin
para que el chequeo de dedup (`getDoc` previo al create en `subirEntrante`) sobre un hash
nuevo no reviente con "Null value error"; el scoping de docs existentes se mantiene.

## Nota de clase (no es cambio, es para no repetir el patrón)
Toda regla de `read` scopeada que derefencia `resource.data.X` es frágil ante un `getDoc`
de un doc inexistente por un no-admin (resource null ⇒ throw, no deny). Revisar si hay
otros `getDoc` de docs puntuales en rutas de dependiente:
- `movimientos:49` (`resource.data.persona`) — OK si solo se leen por query; revisar si
  algún flujo hace `getDoc` de un movimiento puntual antes de existir.
- `comprobantes:92` (F6.9.9, `resource.data.subidoPor`) — OK por ahora: se lee por query
  (el hook). Si se agrega una vista detalle con `getDoc(comprobantes/{id})` para no-admin,
  aplicar el mismo `resource == null`.

## Edge a tener en cuenta (no bloquea; decisión de diseño aparte)
Si un dependiente sube un archivo **idéntico** a uno que ya subió otro miembro (mismo hash),
el `getDoc` encuentra el doc de otro ⇒ read denegado ⇒ sale como "Error al subir", no como
"duplicado". El dedup es content-addressed (un doc por hash). Si querés que en ese caso diga
"ya cargado", hay que repensar el chequeo de dedup fuera del read scopeado (p. ej. una
callable o un campo de existencia público). Lo dejo anotado, no lo toco acá.
