## Problema

Un comprobante de **transferencia** (MercadoPago) hacia un beneficiario que está configurado como
**ítem esperado con `matchTexto`** se carga como movimiento con `itemEsperadoId: null`, quedando en **rama 3**
(solo prefill de categoría). El vínculo con el ítem esperado nunca se formaliza.

### Evidencia (caso real)

Comprobante `hashPdf = 059972b29eddaeeea5116ea6ae24442412da07d1636766d2c49af60df06e685d`:

```
datosExtraidos:
  tipoDocumento:       "transferencia"
  comercioRazonSocial: "Mercado Pago"          ← la PLATAFORMA, no el payee
  destinoNombre:       "Baggini Juan Francisco" ← el payee real
  destinoCuit:         "20222002908"
  destinoCbu:          "3108100900010000137703"

propuestaMatch:
  rama:          3
  origenDestino: true
  itemEsperadoId: (ausente)
```

Ítem esperado configurado: `matchTexto.incluye = ["rugby", "baggini"]`.

## Causa raíz

`functions/src/matchLogica.ts` → `matchConEsperados()` evalúa el `matchTexto`
**únicamente contra `datos.comercioRazonSocial`**. En una transferencia ese campo es la plataforma
(`"Mercado Pago"`), no el beneficiario. El texto que contiene `"baggini"` vive en `destinoNombre`,
que la función nunca mira → `matchTexto` no engancha → `matchConEsperados` devuelve `[]` → **rama 3**.

### Consecuencia (ciclo de aprendizaje roto)

Al confirmar en rama 3, `aprenderDestino` graba el destino **sin `itemEsperadoId`**. La próxima transferencia
al mismo payee pega en `matchPorDestino` y vuelve a devolver **rama 3** (solo categoría, `matchLogica.ts` ~454-462),
porque el destino aprendido no tiene ítem. El ciclo virtuoso
(`texto → itemEsperadoId en movimiento → aprenderDestino con ítem → futuras transferencias en rama 2 por destino directo`)
**nunca arranca**.

## Fix

`functions/src/matchLogica.ts` → función `matchConEsperados`.

Componer el texto a evaluar con `comercioRazonSocial` **+** `destinoNombre`.

### BUSCAR

```ts
  if (!datos.comercioRazonSocial) return [];
  const comercio = datos.comercioRazonSocial.toLowerCase();

  return items.filter(item => {
    if (!item.activo)   return false;
    if (item.moneda !== datos.moneda) return false;
    if (!item.matchTexto || item.matchTexto.incluye.length === 0) return false;
    return evaluarMatchTexto(comercio, item.matchTexto);
  });
```

### REEMPLAZAR

```ts
  // F9.60 — el texto a matchear combina emisor + beneficiario. En transferencias el
  // comercioRazonSocial es la plataforma ("Mercado Pago") y el payee real (contra el
  // que se configura matchTexto) viaja en destinoNombre. Evaluar solo comercioRazonSocial
  // dejaba itemEsperadoId sin poblar (rama 3) → el destino se aprendía sin item y el
  // ciclo virtuoso nunca arrancaba.
  const texto = [datos.comercioRazonSocial, datos.destinoNombre]
    .filter((s): s is string => !!s)
    .join(' ');
  if (!texto) return [];

  return items.filter(item => {
    if (!item.activo)   return false;
    if (item.moneda !== datos.moneda) return false;
    if (!item.matchTexto || item.matchTexto.incluye.length === 0) return false;
    return evaluarMatchTexto(texto, item.matchTexto);
  });
```

Notas:

- `evaluarMatchTexto` ya hace `toLowerCase()` internamente — no hace falta normalizar en el `join`.
- La firma de la función y sus tipos (`DatosExtractosMin` ya expone `destinoNombre`) no cambian.

## Por qué NO toca el frontend

Existen **dos matchers espejo** que usan la misma semántica de `incluye`/`excluye` (`evaluarMatchTexto`)
pero operan sobre campos distintos según el contexto:

- **Backend — `matchConEsperados`** (este fix): corre al **extraer el comprobante**, sobre `datosExtraidos`.
  Antes de que exista un movimiento, el único texto disponible del payee es `comercioRazonSocial + destinoNombre`.
- **Frontend — `movimientosDelItem`** (`src/datos/checklist.ts`): corre en el Resumen, sobre movimientos **ya creados**,
  evaluando `matchTexto` contra `m.descripcion` (ej. `"Micro Rugby Baggini Juan Francisco"`, que ya contiene el payee).

Por eso el ítem **ya aparece matcheado en el Resumen** aunque `itemEsperadoId` sea null: el front lo asocia por
descripción. Este fix alinea el matcher del backend para que **formalice** el vínculo (`itemEsperadoId`) y cierre
el aprendizaje del destino. **No modificar `checklist.ts`.** El comentario "espeja `movimientosDelItem`" en
`matchLogica.ts` sigue siendo válido a nivel de semántica de `incluye`/`excluye`.

## Fuera de alcance (próximo addendum — F9.61)

Este addendum **no toca la fecha del movimiento ni `pagado`/`confirmadoPago`**. Ambos temas van juntos en F9.61
porque comparten el concepto de **fecha final del movimiento**:

1. **Fecha del movimiento desde el vencimiento.** `Comprobantes.tsx` prellena `fecha` con `d.fecha` (emisión),
   por lo que facturas de servicio caen en el mes equivocado (ej. Edenor emitida 24/06 → junio, cuando debería
   entrar en julio por su primer vencimiento 08/07). Regla objetivo:
   `fecha = d.vencimientos?.[0]?.fecha ?? d.fecha ?? undefined`.
   - Con vencimiento (servicios, cupones) → primer vencimiento.
   - Sin vencimiento (transferencias, pagos hechos) → fecha de emisión (= fecha real del pago).
2. **Regla `pagado`/`confirmadoPago` por fecha.** `pagado = fechaFinal ≤ hoyAR`, mismo booleano para ambos flags,
   calculado sobre la fecha **final** del form (post-edición) y para **todo** comprobante (hoy `confirmadoPago`
   solo se deriva por fecha si `esPago` y usa `d.fecha`; `pagado` está hardcodeado en `true`).

No incluir nada de esto en F9.60.
