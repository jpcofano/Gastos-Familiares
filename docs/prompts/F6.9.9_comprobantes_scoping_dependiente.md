# F6.9.9 — Scoping de lectura de comprobantes (dependiente ve solo lo suyo)

Cierra la fuga detectada en auditoría y reconfirmada contra `origin` HEAD `2e49836`:
hoy cualquier miembro lee **todos** los comprobantes en los tres niveles —
regla abierta (`firestore.rules:92`), query sin filtro (`useComprobantes`),
y render del Historial sin gate (`Comprobantes.tsx`). Replica el patrón ya
probado en `movimientos` (regla scoped) y en `entrantes` (query condicional).

Confirmar que HEAD sigue en `2e49836` y que las líneas referidas abajo siguen vigentes.

- `subidoPor` = `memberId`. Cliente: `src/datos/comprobantes.ts:78,90`. Servidor:
  `functions/src/index.ts:1173,1201` (`entrante.creadoPor`, que traza a `memberId`

- Helpers de reglas presentes: `esAdmin()` (8), `esMiembro()` (12), `miMemberId()` (16).

## Cambio 1 — `firestore.rules`: scopear lectura

BUSCAR:
```
    match /comprobantes/{hash} {
      allow read:   if esMiembro();
```
REEMPLAZAR:
```
    match /comprobantes/{hash} {
      allow read:   if esAdmin() || (esMiembro() && resource.data.subidoPor == miMemberId());
```
`create` (subida) y `update/delete` (admin) no se tocan. El server (Admin SDK) bypassa reglas.

## Cambio 2 — `src/hooks/useComprobantes.ts`: query condicional

Con la regla scoped, una query sin `where('subidoPor')` la rechaza Firestore entera
(la regla es un gate, no un filtro). Mismo patrón que `src/datos/entrantes.ts:87-89`.

BUSCAR:
```
import { collection, query, orderBy, onSnapshot } from 'firebase/firestore';
```
REEMPLAZAR:
```
import { collection, query, where, orderBy, onSnapshot } from 'firebase/firestore';
```

BUSCAR:
```
export function useComprobantes() {
```
REEMPLAZAR:
```
export function useComprobantes(memberId: string, esAdmin: boolean) {
```

BUSCAR:
```
  useEffect(() => {
    const q = query(collection(db, 'comprobantes'), orderBy('subidoEn', 'desc'));
```
REEMPLAZAR:
```
  useEffect(() => {
    const q = esAdmin
      ? query(collection(db, 'comprobantes'), orderBy('subidoEn', 'desc'))
      : query(collection(db, 'comprobantes'), where('subidoPor', '==', memberId), orderBy('subidoEn', 'desc'));
```

BUSCAR:
```
    return unsub;
  }, []);
```
REEMPLAZAR:
```
    return unsub;
  }, [memberId, esAdmin]);
```

## Criterios de cierre
- Reglas en emulador: dependiente lee solo comprobantes con `subidoPor == su memberId`;
  admin lee todos.
- `useComprobantes` filtra cuando `!esAdmin`; la query no tira `permission-denied`
  ni `failed-precondition` (índice desplegado/seedeado en emulador).
- `npm run build` (front) verde.
- **Prueba en vivo con datos cruzados** (clave — el bug no se ve sin esto): que existan
  comprobantes de al menos dos personas. Logueado como dependiente, en Carga el Historial
  muestra **solo los propios**; como admin, todos. Antes la prueba no exponía la fuga porque
  no había comprobantes de otros.
