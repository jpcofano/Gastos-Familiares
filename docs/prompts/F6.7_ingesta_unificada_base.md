# F6.7 — Ingesta unificada (base prompt)

> Una sola superficie de subida (botón in-app + página + share-target del celular) que detecta el tipo
> y rutea: resumen de tarjeta → flujo tarjeta; todo lo demás (comprobante/factura) → flujo comprobante.
> Los flujos de extracción existentes NO se tocan: se agrega un router adelante.

## Objetivo
Hoy hay dos puntos de subida separados (Comprobantes y /tarjetas) y el share-target del celular sube
fijo a Comprobantes. Unificar en una sola entrada que clasifique y derive al flujo correcto, sin que
el usuario elija tipo a mano.

## Principio rector
Una sola superficie de ingesta; **la detección de tipo es una etapa barata y separada de la
extracción**; ante baja confianza, NO se adivina → queda en una bandeja para resolución manual.
Es la misma filosofía "propone → confirmás" del resto del sistema.

## Decisiones fijadas (defaults — Juan confirma si alguna no le cierra)

**D1 — Detección (determinística, sin LLM extra).**
- Imagen (`image/*`) → siempre `comprobante` (nadie fotografía un resumen de 30 páginas).
- PDF → extraer **texto de la página 1** y contar marcadores de resumen (normalizados, uppercase, sin
  acentos): `TOTAL A PAGAR`, `VENCIMIENTO`, `CIERRE`, `SALDO PENDIENTE`, `PAGO MINIMO`, número de
  tarjeta enmascarado (`XXXX...####`). Regla: **≥2 marcadores → `resumen`**; **0 marcadores →
  `comprobante`**; **exactamente 1 → `ambiguo`**.
- Si el PDF no tiene capa de texto (escaneado, page-1 vacía) → **fallback de visión barato** (una sola
  imagen de la página 1 a un modelo liviano que responde solo el tipo). Si la visión tampoco decide →
  `ambiguo`.

**D2 — Quién sube y qué pasa con el rol.**
- `entrantes` permite `create` a **cualquier miembro** (los chicos suben sus comprobantes).
- El router corre server-side (admin SDK) y rutea por tipo sin importar quién subió.
- Un `comprobante` detectado → se crea con `creadoPor = uploader` (sirve para admin y dependiente).
- Un `resumen` detectado se crea en `resumenesTarjeta` (colección admin-only de lectura): si lo subió
  un dependiente, simplemente no lo verá después (correcto, /tarjetas es admin-only); el admin lo ve.
- **Prior por rol en el caso `ambiguo`:** si el uploader es dependiente, un `ambiguo` se trata como
  `comprobante` (es lo único que puede poseer; un chico no sube resúmenes). Si el uploader es admin,
  `ambiguo` va a la bandeja para que elija a mano. *(Decisión: usar el rol como desempate; flipear si
  preferís bandeja siempre.)*

**D3 — Colección `entrantes`: no efímera, con trazabilidad.**
- No se borra al rutear: queda con `estado` y `destino` (colección + id del doc creado) para depurar
  misroutes y para idempotencia. La limpieza es un tema posterior (no de esta fase).

## Modelo de datos

`entrantes/{hash}` (hash = SHA-256 del archivo, igual criterio que comprobantes/resumenesTarjeta):
```ts
interface Entrante {
  hash: string;
  rutaStorage: string;            // ver nota de Storage abajo
  mimeType: string;
  creadoPor: string;              // memberId del uploader
  origen: 'app' | 'share_target';
  estado: 'pendiente' | 'ruteado' | 'ambiguo' | 'error';
  tipoDetectado?: 'comprobante' | 'resumen' | 'ambiguo';
  destino?: { coleccion: 'comprobantes' | 'resumenesTarjeta'; id: string };
  motivoDeteccion?: string;       // p.ej. "2 marcadores: TOTAL A PAGAR, VENCIMIENTO" — trazabilidad
  creadoEn: Timestamp;
  actualizadoEn: Timestamp;
}
```

## Cambios

### 1. Punto de entrada único (UI)
- Reemplazar el botón "Subir" de Comprobantes y el de /tarjetas por **una sola acción** que sube a
  `entrantes` (Storage + doc), sin selector de tipo.
- Una **bandeja** (puede vivir en Comprobantes o vista propia) que muestra: items `ruteados` (con un
  link al doc destino) y items `ambiguo` esperando un toque del admin → botones
  "Es comprobante" / "Es resumen de tarjeta" que promueven el entrante al flujo elegido.

### 2. Router (Cloud Function)
- Nueva CF `routearEntrante` = `onDocumentCreated('entrantes/{hash}')`.
- Lógica: idempotencia primero (si ya existe `comprobantes/{hash}` o `resumenesTarjeta/{hash}` → marca
  `estado:'ruteado'` con el destino existente y corta). Luego detección (D1). Luego, según tipo:
  - `comprobante` → crear el doc en `comprobantes/{hash}` **con la misma forma que produce hoy la
    subida de comprobantes** (mirar el flujo actual y replicar campos/estado) → `extraerComprobante`
    dispara solo, sin cambios.
  - `resumen` → crear `resumenesTarjeta/{hash}` en `estado:'subido'` **con la misma forma que produce
    hoy `subirResumenTarjeta`** → `extraerResumenTarjeta` dispara solo.
  - `ambiguo` → aplicar prior por rol (D2); si queda ambiguo, `estado:'ambiguo'` (espera la bandeja).
- **No tocar** `extraerComprobante` ni `extraerResumenTarjeta`: el router solo crea el doc de entrada
  que cada trigger ya espera.

### 3. Share-target (ajuste de F6.6)
- El handler del service worker que hoy sube a Comprobantes pasa a subir a `entrantes` (con
  `origen:'share_target'`). El resto del mecanismo (IDB, auto-subida) igual.

### 4. Reglas de Firestore (`entrantes`)
```
match /entrantes/{hash} {
  allow read:   if esMiembro() && (esAdmin() || resource.data.creadoPor == miMemberId());
  allow create: if esMiembro()
    && request.resource.data.creadoPor == miMemberId()
    && request.resource.data.estado == 'pendiente'
    && request.resource.data.hash is string;
  allow update, delete: if false;   // solo el router (admin SDK) escribe estado/destino
}
```
La promoción de un `ambiguo` desde la bandeja la hace una CF callable (admin), no un write directo del
cliente, para no abrir `update`.

## Nota de Storage (importante)
Verificar **dónde leen el archivo** `extraerComprobante` y `extraerResumenTarjeta` hoy (campo de ruta
en el doc, o path convención). El router debe dejar el doc destino apuntando a esa misma ubicación: lo
más simple es subir el archivo una vez a `entrantes/{hash}` y que el doc destino referencie ese mismo
objeto (reusar el path), evitando copiar. Si los extractores hoy asumen un path fijo distinto, el
router copia el objeto a ese path antes de crear el doc. Elegir lo que cambie menos los triggers.

## No tocar
- `extraerComprobante`, `extraerResumenTarjeta`, el cuadre, la confirmación, el clasificador.
- La separación de colecciones `comprobantes` vs `resumenesTarjeta` (cada una con su trigger y sus
  reglas admin-only/miembro existentes).

## Criterios de cierre
- Subir una **imagen** de comprobante por la entrada única → `entrantes` → `comprobantes/{hash}` →
  `extraerComprobante` dispara → flujo normal.
- Subir un **PDF de resumen** → detección ≥2 marcadores → `resumenesTarjeta/{hash}` estado subido →
  `extraerResumenTarjeta` dispara → preview de tarjeta.
- Subir un **PDF ambiguo** como admin → bandeja con los dos botones → elegir → rutea bien.
- Subir el **mismo archivo dos veces** (o compartir + subir) → segundo no duplica (idempotencia por hash).
- **Share-target** desde el celular → cae en `entrantes` y rutea igual que la subida in-app.
- Un **dependiente** sube un comprobante → funciona; sube un PDF de resumen → no lo ve después (queda
  para admin), sin error.
