# F9.101 — Actualización integral post-corrida ("Actualizar todo")

## Contexto

Hoy, después de confirmar una corrida nueva, el usuario tiene que regenerar a mano
cuatro cosas en Research: análisis de lote, panorama sectorial, agenda macro y
(aparte, en Config) el valor de las posiciones manuales. Además la sincronización
CAFCI es otro botón separado. Son 4-5 interacciones que siempre van juntas.

Este spec agrega **un solo punto de entrada "Actualizar todo"** con dos variantes,
reutilizando la infraestructura existente:

- `generarPromptIA` / `importarAnalisisIA` (functions/src/index.ts ~3466 y ~3548),
  onCall con `modo: 'posicion' | 'sectorial' | 'agenda' | 'lote'`.
- `analizarConIA` (functions/src/index.ts ~2999), API con `claude-sonnet-4-6` y
  tool `web_search_20250305`.
- `guardarPosicionManual` / colección `posicionesManuales`
  (src/datos/patrimonio.ts:81), tipo `PosicionManual` en src/types/patrimonio.ts:52.
- `handleSincronizarCafci` (src/vistas/Patrimonio.tsx:3074).

**Auditar antes de tocar**: verificar en el código real las firmas y rutas citadas;
si algo difiere, reportarlo antes de implementar.

## Alcance

### 1. Backend — nuevo modo `'completo'`

#### 1a. `generarPromptIA` acepta `modo: 'completo'`

Nueva función `buildPromptCompleto(contexto)` que compone en un único prompt:

1. El bloque del **lote** (reutilizar `buildPromptLote` — ya trae sus instrucciones
   de formato embebidas; ajustar para que el JSON de salida quede anidado, ver 1c).
2. El bloque **sectorial** (reutilizar `buildPromptSectorial`).
3. El bloque **agenda** (reutilizar `buildPromptAgenda`).
4. Un bloque nuevo de **posiciones manuales**: recibe en `contexto.manuales` un
   array `{ id, ticker, nombre, cantidad, valorUsd, fechaValuacion }` y pide:
   buscar el precio de mercado actual de cada ticker (cierre más reciente),
   calcular `valorUsd = cantidad × precioUnitarioUsd`, y devolver `null` en
   `precioUnitarioUsd` si no se puede verificar (no inventar).

Instrucciones de formato del modo completo (reemplazan a las individuales):

- Usar búsqueda web para verificar datos actuales antes de responder.
- Responder ÚNICAMENTE con **un solo bloque ```json** con esta forma exacta:

```json
{
  "lote": { "analisis": [ { "ticker": "...", "resultado": { ... } } ] },
  "sectorial": "## <Nombre> [driver: <driver>]\n...",
  "agenda": { "eventos": [ { "fecha": "...", "evento": "...", "driver": "...", "porQueImporta": "..." } ] },
  "manuales": [ { "id": "...", "ticker": "...", "precioUnitarioUsd": 143.21, "valorUsd": 7160.50, "fechaPrecio": "YYYY-MM-DD", "fuente": "..." } ]
}
```

- `sectorial` es el markdown completo como string JSON (headers
  `## Nombre [driver: xxx]` intactos, drivers válidos: energia_ar, cer_pesos,
  soberano, cripto, tech_global).
- Sin texto fuera del bloque; sin comentarios dentro del JSON.

#### 1b. `importarAnalisisIA` acepta `modo: 'completo'`

- Extraer el JSON con `extraerResultado` como hoy.
- **Fail-soft por sección** (mismo espíritu que el lote actual): cada sección se
  valida e importa de forma independiente; una sección inválida no aborta las
  demás. Resumen compuesto, ej.:
  `"lote: 18 importados, 1 con error · sectorial: ok · agenda: 12 eventos · manuales: 1 actualizada (ACN), 1 sin precio (GLOB)"`.
- Refactor necesario: extraer la lógica de importación existente de cada modo a
  helpers internos (`importarLote(items)`, `importarSectorial(texto)`,
  `importarAgenda(obj)`) y que tanto los modos individuales como `'completo'`
  deleguen en ellos. Reusar `validarResultadoImportado` tal cual por sección.
- **Sección `manuales`** (nueva): para cada item, cargar el doc de
  `posicionesManuales` por `id`; validar que el doc exista, que
  `precioUnitarioUsd > 0` y que `|valorUsd − cantidad_del_doc × precioUnitarioUsd| < 1`.
  Si valida: actualizar **solo** `valorUsd` y `fechaValuacion` (= `fechaPrecio`).
  Nunca tocar `cantidad` ni crear docs nuevos. `precioUnitarioUsd: null` → skip
  con nota en el resumen.

#### 1c. Compatibilidad del lote

`buildPromptLote` hoy pide `{ "analisis": [...] }` a nivel raíz. En modo completo
la misma estructura va anidada bajo `"lote"`. Ajustar la redacción del bloque para
que quede claro, sin romper el modo `'lote'` standalone (que sigue igual).

### 2. Backend — modo API

`analizarConIA` (o un wrapper nuevo si queda más limpio) acepta la etapa
`'manuales'`: mismo bloque de prompt de manuales, con web search, devolviendo el
array `manuales` del esquema de 1a. La importación reutiliza el helper de 1b.

**No** hacer una única llamada API gigante con el prompt completo: el riesgo de
truncamiento con lote + sectorial + agenda en una sola respuesta es alto. El modo
API se orquesta del lado cliente por etapas (ver 3b).

### 3. Frontend (src/vistas/Patrimonio.tsx)

#### 3a. Botón "Actualizar todo — Chat"

En Research, arriba de las cards actuales, una card "Actualización integral" con
dos botones. El de **Chat**:

1. Llama `generarPromptIA('completo', contexto)` con el contexto combinado que hoy
   arman los flujos individuales + `manuales` (de `cargarPosicionesManuales()`).
2. Muestra el prompt con el mismo modal copiar/pegar existente.
3. El paso 2 del modal importa con `importarAnalisisIA('completo', contenido)` y
   muestra el resumen compuesto.
4. Al cerrar con éxito, dispara `handleSincronizarCafci()` (determinístico, no
   necesita chat) y refresca los estados locales (analisisCache, sectorial,
   agenda, manuales).

#### 3b. Botón "Actualizar todo — API"

Orquestador cliente secuencial con progreso por etapa
(`Etapa 2/5: sectorial…`):

1. Lote (flujo API existente, con su progreso por ticker)
2. Sectorial (`analizarConIA` existente)
3. Agenda (existente)
4. Manuales (etapa nueva de §2)
5. Sincronizar CAFCI (`handleSincronizarCafci`)

Fail-soft: si una etapa falla, se registra y se continúa; al final, resumen con
etapas ok/fallidas. Deshabilitado si `!configIA.habilitado` (igual que "Analizar
lote" hoy).

#### 3c. CTA post-ingesta

Al confirmar una corrida en PatrimonioIngesta, además del mensaje de éxito,
ofrecer un link/botón "Actualizar análisis con la cartera nueva" que navegue a
Research con la card de actualización integral visible. Sin lógica nueva: es solo
navegación.

### 4. Fuera de alcance

- No tocar los botones individuales existentes (lote, sectorial, agenda,
  posición): siguen funcionando igual.
- No modificar el esquema de la corrida ni la ingesta.
- No crear posiciones manuales nuevas desde el import.

## Criterios de aceptación

1. `generarPromptIA('completo', ctx)` devuelve un único prompt con las cuatro
   secciones y las instrucciones del esquema JSON unificado.
2. Pegar una respuesta válida en el import de modo completo puebla
   `analisisPosiciones`, `analisisSectorial`, `agendaMacro` y actualiza
   `valorUsd`/`fechaValuacion` de las manuales, con resumen compuesto.
3. Una sección inválida (p. ej. sectorial sin headers) no impide importar las
   otras tres; el resumen lo refleja.
4. Import de manuales: nunca modifica `cantidad`, nunca crea docs, skip limpio
   con `precioUnitarioUsd: null`.
5. El botón API corre las 5 etapas con progreso visible y fail-soft por etapa.
6. Los modos individuales (`posicion`, `sectorial`, `agenda`, `lote`) siguen
   pasando sus flujos actuales sin cambios de comportamiento.
7. `npx tsc --noEmit` limpio (frontend y functions — recordar que el tsconfig
   cubre `src/**/*.tsx`).
