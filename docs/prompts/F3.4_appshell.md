# F3.4 — AppShell con 4 estados

**Fase:** F3 · **Sub-paso:** 3.4
**Prerequisito:** F3.3 cerrado (useMiembro resolviendo los 4 estados correctamente).
**Scope:** solo emulador.

---

## Contexto

`useMiembro()` ya expone `estado: 'cargando' | 'noAutenticado' | 'noAutorizado' | 'autenticado'` con `memberId` y `miembro`. Este paso reemplaza la UI provisoria de `App.tsx` por el AppShell real, siguiendo el mismo blueprint que la app de comidas.

---

## Qué hace F3.4

1. **`src/AppShell.tsx`** — componente que renderiza según `estado`:
   - `cargando` → pantalla de carga simple (spinner o texto centrado). Sin parpadeo: es lo primero que se ve siempre hasta resolver auth.
   - `noAutenticado` → pantalla de bienvenida: nombre de la app + botón "Ingresar con Google".
   - `noAutorizado` → mensaje claro ("Tu cuenta {email} no pertenece a la familia") + botón "Salir".
   - `autenticado` → layout base de la app: header con nombre de la app, "{miembro.nombre}" y botón de salir; cuerpo vacío con placeholder ("Próximamente: Dashboard") donde F4 montará las vistas.

2. **Estructura para crecer:** el cuerpo del estado `autenticado` debe quedar listo para recibir rutas/vistas en F4 (aunque ahora sea un placeholder). Si el blueprint de comidas usa algún patrón de layout (header fijo + contenido), replicarlo.

3. **Estilo:** coherente con el blueprint de comidas. Sin agregar librerías de UI nuevas. CSS simple o el approach que ya use el proyecto.

4. **El rol disponible para F4:** dejar accesible `miembro.rol` desde el contexto del shell (prop, context o el propio hook) — F4 lo va a usar para diferenciar la vista de admin vs dependiente. NO implementar todavía ninguna diferencia de vista.

---

## Restricciones

- **NO** agregar router todavía si comidas no lo usa en el shell (las rutas llegan con las vistas de F4; si conviene meter el router acá, proponerlo en el reporte y decido).
- **NO** fetchear movements ni ninguna colección de datos (es F4).
- **NO** tocar `scripts/seed/`, Rules, índices.

---

## Criterio de cierre

- Los 4 estados renderizan correctamente sus pantallas (probar con cuenta whitelisteada, cuenta ajena, y deslogueado).
- Refresh en estado autenticado: pasa por `cargando` y aterriza en `autenticado` sin flashear `noAutenticado` en el medio.
- `npm run validate` sigue en 8 OK / 0 FAIL.
