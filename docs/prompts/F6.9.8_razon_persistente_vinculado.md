# F6.9.8 — Razón del match persistente en el card resuelto (post-vinculado)

## Problema
Los badges de F6.9.6 ("Pagó una factura" / "Cumple un gasto esperado" / "Movimiento nuevo") viven dentro de
`PropuestaCard`, que **solo se renderiza mientras `comp.estado === 'extraido' && comp.propuestaMatch`**
(`Comprobantes.tsx`, card shell). Apenas el comprobante se resuelve y pasa a `vinculado`, la `PropuestaCard`
desaparece y el card queda solo con el `BadgeEstado` "VINCULADO" genérico — se pierde el *por qué*.

Caso más visible: un pago que reconcilia por payee (rama 1 `origenReconciliacion`) se auto-vincula en
background, salta de `extraido` a `vinculado` casi instantáneo, y nunca llegás a ver el verde "Pagó una
factura". El card resuelto no distingue una reconciliación de un alta nueva ni de un cumplimiento de esperado.

El dato está disponible: `confirmarRama1` y `marcarVinculado` (`src/datos/comprobantes.ts`) **solo actualizan
`estado`** — NO borran `propuestaMatch`. Así que `comp.propuestaMatch` (rama, `origenReconciliacion`,
`esAdicional`) sigue legible en estado `vinculado`. Esto es puro cliente, sin tocar lógica de match.

> **Depende de F6.9.6 commiteado** (reusa las clases CSS `cmp-propuesta-tipo` / `cmp-tipo--*` y el campo
> `origenReconciliacion` tipado en el `PropuestaMatch` cliente). Aplicar después de los commits de F6.9.6 + F6.9.7.

## Cambio 1 — helper `RazonVinculado` (`src/vistas/Comprobantes.tsx`)
Insertar el helper justo después de `BadgeEstado`.

BUSCAR:
```
function BadgeEstado({ estado }: { estado: string }) {
  return (
    <span className={`cmp-estado-badge cmp-estado--${estado}`}>{estado}</span>
  );
}
```
REEMPLAZAR:
```
function BadgeEstado({ estado }: { estado: string }) {
  return (
    <span className={`cmp-estado-badge cmp-estado--${estado}`}>{estado}</span>
  );
}

// F6.9.8 — etiqueta persistente de la razón del match en el card ya resuelto.
// Lee propuestaMatch (sobrevive al estado vinculado: confirmarRama1/marcarVinculado
// solo tocan `estado`) para conservar el "por qué" después de resolver.
function RazonVinculado({ pm }: { pm: Comprobante['propuestaMatch'] }) {
  if (!pm) return null;
  let texto: string;
  let clase: string;
  switch (pm.rama) {
    case 0:
      texto = 'Ya cargado';
      clase = 'cmp-tipo--dedup';
      break;
    case 1:
      texto = pm.origenReconciliacion ? 'Pagó una factura' : 'Vinculado a un movimiento';
      clase = pm.origenReconciliacion ? 'cmp-tipo--reconc' : 'cmp-tipo--dedup';
      break;
    case 2:
      texto = pm.esAdicional ? 'Pago adicional' : 'Cumplió un gasto esperado';
      clase = 'cmp-tipo--nuevo';
      break;
    case 3:
      texto = 'Cargado como nuevo';
      clase = 'cmp-tipo--nuevo';
      break;
    default:
      return null;
  }
  return <span className={`cmp-propuesta-tipo ${clase}`}>{texto}</span>;
}
```

## Cambio 2 — render en el card resuelto (`src/vistas/Comprobantes.tsx`)
En el card shell, agregar el bloque `vinculado` después de los condicionales de `extraido`.

BUSCAR:
```
      {comp.estado === 'extraido' && !comp.propuestaMatch && (
        <p className="cmp-nota">Calculando match…</p>
      )}
    </div>
  );
}
```
REEMPLAZAR:
```
      {comp.estado === 'extraido' && !comp.propuestaMatch && (
        <p className="cmp-nota">Calculando match…</p>
      )}
      {comp.estado === 'vinculado' && comp.propuestaMatch && (
        <div className="cmp-card-razon">
          <RazonVinculado pm={comp.propuestaMatch} />
        </div>
      )}
    </div>
  );
}
```
