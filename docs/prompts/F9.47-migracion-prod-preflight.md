# F9.47 — Etapa B (arranque): preflight + guardrails + runbook de migración a prod

> Reglas: doc-first. **Toca `.firebaserc`, `scripts/seed/*` (guardrails), agrega un script
> `migrate:preflight` y un runbook.** NO importa datos a prod por su cuenta: deja el repo
> **seguro** para que el import lo corra el usuario con red de contención. Cero cambios de UI,
> cero cambios de modelo.

## Contexto
El cableado a datos reales está cerrado (F9.21→F9.46). El seed (`scripts/seed/seed.ts`),
el validador (`validators/runValidations.ts`) y el backfill (`backfillPersonaMemberId.ts`)
**ya aceptan `--target=production --i-am-sure`** y escriben con Admin SDK (saltean Rules).
La infra de prod (Rules + índices) ya existe en `firestore.rules` / `firestore.indexes.json`.

Antes de escribir un solo doc en prod hay que tapar 4 agujeros que hoy harían perder datos o
apuntar al proyecto equivocado. Eso es esta tarea.

## Diagnóstico (auditado contra el repo)
1. **`.firebaserc` apunta al proyecto que NO existe.**
   `default: "gastos-familiares"` → el proyecto real es **`gastos-familiares-e6415`**.
   `firebase deploy` / `emulators` resuelven el id desde acá → todo va al lugar equivocado o
   falla.
2. **El emulador hardcodea el otro id.**
   `scripts/seed/utils/firestore.ts` (branch `emulator`) hace
   `initializeApp({ projectId: 'gastos-familiares' })`. Con `singleProjectMode:true`
   (firebase.json) el projectId del emulador **debe coincidir** con el `default` del
   `.firebaserc`, si no la data sembrada no aparece en la Emulator UI / Auth. Si se corrige
   (1), hay que alinear esto a `gastos-familiares-e6415` también.
3. **`seedConfig` hace `.set()` (overwrite total) de `config/familia`.**
   En el primer import a prod (vacío) está perfecto. Pero `config/familia` es **el doc que
   editan las callables admin** (F9.36–41: medios, miembros, categorías, tarjetas, TC…). Si
   alguien re-corre el seed contra prod **después** de editar configs, `.set()` **borra esas
   ediciones**. Hay que impedir el clobber accidental.
4. **No hay preflight.** Hoy `seed --target=production` escribe a ciegas: no dice qué hay en
   prod antes de tocar, ni avisa si ya tiene datos. Para una migración irreversible eso es
   inaceptable.

## Tarea

### 1. Corregir el project id (los dos lugares)
- `.firebaserc` → `"default": "gastos-familiares-e6415"`.
- `scripts/seed/utils/firestore.ts`, branch `emulator`:
  `initializeApp({ projectId: 'gastos-familiares-e6415' })`.
- `grep -ri "gastos-familiares\b"` (sin el `-e6415`) en todo el repo y corregir cualquier
  otra referencia colgada al id viejo (configs, scripts, docs de Code). Loguear las que
  encuentres.

### 2. Guardrail anti-clobber de `config/familia`
En `transformers/config.ts` (`seedConfig`), antes del `.set()`:
- Leer `config/familia`. Si **existe y `target === 'production'`**, NO pisar salvo flag
  explícito **`--force-config`**. Sin el flag: loguear
  `SKIP config/familia (ya existe en prod — usá --force-config para sobrescribir)` y seguir
  con el resto del seed (no abortar).
- Con `--force-config` (o si no existe, o target emulador): comportamiento actual (`.set()`).
- Propagar el flag desde `parseFlags()` en `seed.ts` (`forceConfig: boolean`) hasta
  `seedConfig(db, data, dryRun, forceConfig)`.
- Mismo criterio recomendado para cualquier transformer que haga overwrite de docs de config
  editables; los catálogos de solo lectura (tcDiario, subcategorias, etiquetas, reglas) se
  pueden re-sembrar sin riesgo (no los edita nadie).

### 3. Script de preflight (`scripts/seed/preflight.ts`)
Nuevo script, **solo lectura**, que se conecta al target y reporta el estado ANTES de migrar.
Agregar a `package.json`: `"migrate:preflight": "tsx scripts/seed/preflight.ts"`.
- Acepta `--target=emulator|production` (default emulator). Para prod **no** requiere
  `--i-am-sure` (es read-only) pero sí necesita el service account.
- Imprime, para cada colección que toca el seed
  (`config`, `tcDiario`, `subcategorias`, `etiquetas`, `reglasNormalizacion`, `diccionario`,
  `itemsEsperados`, `resumenesTarjeta`, `autorizados`, `movimientos`):
  el **count actual** de docs en el target.
- Imprime el **count esperado** del Excel (lo que el seed escribiría) leyendo el snapshot con
  los mismos transformers en `dryRun` (o contando las filas equivalentes): debe matchear los
  números conocidos — **1143 movimientos, 18 resúmenes, 177 tcDiario** + config/autorizados.
- Veredicto claro:
  - prod **vacío** (todas en 0) → `LISTO PARA IMPORT LIMPIO`.
  - prod **con datos** → `prod NO está vacío — re-import sobrescribe por id (idempotente) pero
    config/familia queda protegida; revisá antes de seguir`.
- Verifica que exista el service account (`GOOGLE_APPLICATION_CREDENTIALS` o
  `./secrets/serviceAccountKey.json`) y, si falta, falla con un mensaje que diga exactamente
  qué bajar de la consola de Firebase y dónde ponerlo. Confirmar que `secrets/` y
  `serviceAccountKey.json` están en `.gitignore` (agregarlos si no).

### 4. Runbook (`docs/RUNBOOK-migracion-prod.md`)
Doc operativo con los comandos EXACTOS en orden, para que el usuario ejecute la migración.
Estructura (los pasos los corre el usuario, no Code):

```
0. Preflight local (emulador): npm run seed && npm run validate  → 21/21
1. Service account: bajar de consola Firebase (gastos-familiares-e6415) →
   ./secrets/serviceAccountKey.json  (NUNCA a git)
2. Deploy de índices PRIMERO (tardan minutos en construir):
   firebase deploy --only firestore:indexes
3. Deploy de rules:        firebase deploy --only firestore:rules
4. Deploy de functions:    firebase deploy --only functions   (MEP F9.30 + callables + Calendar)
5. Preflight prod:         npm run migrate:preflight -- --target=production   → debe decir LISTO
6. Import a prod:          npm run seed -- --target=production --i-am-sure
7. Backfill persona:       tsx scripts/seed/backfillPersonaMemberId.ts --target=production           (dry-run)
                           tsx scripts/seed/backfillPersonaMemberId.ts --target=production --apply --i-am-sure   (si dry-run > 0)
8. Cuadre post-import:     npm run validate -- --target=production   → 21/21 contra prod
9. Sign-off F9.31/F9.32 + abrir la app apuntando a prod y verificar lectura sin errores de permisos/índices.
```
Incluí en el runbook: qué hacer si el deploy de índices queda "Building" (esperar antes del
paso 5), y que `--force-config` solo se usa si se quiere repisar config en un re-import
deliberado.
