# F9.75 — Obligaciones no se marcan pagadas por vencimiento

> Aplicar **después** de F9.74.

## Problema

La regla "pagado por fecha" (`fecha ≤ hoyAR` → `pagado`/`confirmadoPago`) se aplica hoy a **todo**
comprobante (decisión F9.63: *"la regla de pago aplica a TODO comprobante, no solo esPago"*).

Para una **obligación** (recibo de servicio, factura) eso es incorrecto: el movimiento se fecha en
el **vencimiento** (F9.63), y cuando ese vencimiento pasa, la regla marca el gasto como **pagado
aunque nadie lo haya pagado**. Ejemplo real: expensas "Cons. Prop. Pje del Signo 4032/42",
vencimiento `2026-07-15`, `montoTotal` 660.833,43 — si se confirma después del 15/07 (o si algún
recalculador re-evalúa), queda `confirmadoPago=true` sin pago real.

El pago de una obligación **viene después** (transferencia / comprobante de pago) y lo confirma la
reconciliación por payee (`reconciliarPorPayee` → rama 1 → `confirmadoPago=true`). El recibo solo
**define la obligación** del mes.

## Comportamiento correcto

- **Obligaciones** (`recibo_servicio`, `factura_a/b/c`): el movimiento se crea con
  `confirmadoPago=false` **siempre**, sin importar la fecha. `estadoItem` lo muestra como
  `por_confirmar` en el mes vigente (checklist.ts:44-50 filtra por `m.confirmadoPago`). El pago
  real reconcilia luego y lo pasa a `pagado`. Ciclo verificado end-to-end.
- **Pagos y tickets** (`transferencia`, `comprobante_pago`, `ticket`, `otro`): sin cambios, siguen
  con la regla de fecha (un ticket es prueba de pago contemporáneo).
- **`resumen_tarjeta`**: fuera de alcance — tiene su propia lógica en `src/datos/resumenesTarjeta.ts`.

## Punto autoritativo

`cargarMovimientoDesdeComprobante` (server) **recalcula** `pagado`/`confirmadoPago` e ignora lo que
manda el cliente (index.ts:1524). La callable **ya carga** el comprobante (`comp = compSnap.data()`),
así que tiene `comp.datosExtraidos.tipoDocumento` disponible. El gate va ahí. Los diffs de cliente
son para coherencia de display (el server manda igual) y para el path rama-1 que sí escribe directo.

## NO tocar (a propósito)

- **`AltaMovimiento.tsx`** (`pagadoPorFecha = fecha <= hoyISO()`): para altas **manuales** la regla
  es correcta (gasto fechado = pagado). En el flujo de comprobante, el server recalcula y descarta
  este valor. No agregar el gate acá.
- **`resumenesTarjeta.ts`**: dominio de resúmenes de tarjeta, lógica propia.

## Limitación conocida (no se aborda acá)

`estadoItem` fuerza `pagado` cuando `mes < mesActual` aunque `confirmadoPago=false`
(checklist.ts:43 — "meses viejos se asumen saldados"). Es decir: una obligación impaga de un mes ya
cerrado igual figura como pagada al pasar el mes. Cambiar eso toca la vista histórica y va aparte si
se decide. Este prompt solo corta el auto-pago **dentro del mes vigente**, que es el caso del recibo.

## Regression tests (post-deploy)

1. **Recibo de expensas (caso real):** cargar/confirmar el recibo "Cons. Prop. Pje del Signo 4032/42"
   (venc. `2026-07-15`). El movimiento queda `confirmadoPago=false` / `pagado=false`. El ítem
   Expensas del mes vigente muestra **`por_confirmar`** (no `pagado`). Verificar en Firestore.
2. **Confirmar una obligación con vencimiento YA pasado** (fecha del mov ≤ hoy): igual queda
   `confirmadoPago=false`. Antes quedaba `true`. Este es el bug que se corrige.
3. **Cierre del ciclo:** cargar luego el pago real (transferencia al CBU `0070127920000005316566`,
   mismo monto). `reconciliarPorPayee` → rama 1 → el movimiento pasa a `confirmadoPago=true` → el
   ítem pasa a **`pagado`**.
4. **No regresión — ticket/pago:** un `comprobante_pago` o `ticket` con fecha ≤ hoy sigue quedando
   `confirmadoPago=true` (regla de fecha intacta).
5. **No regresión — alta manual:** un Gasto manual fechado hoy sigue `pagado=true`
   (`AltaMovimiento` no cambió).
