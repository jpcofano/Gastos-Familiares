# F9.25 — Agregados derivados, no copiados (Dashboard / Resumen)

> Reglas: doc-first · una PR por fase. Auditoría + (si hace falta) refactor. **No** debe
> introducir copias persistidas de agregados.

## Principio rector
**`movimientos` = verdad. Dashboard / Resumen / Tarjetas = vistas derivadas.**
Cualquier total, promedio, ranking por categoría/medio/persona, o serie temporal se
**calcula** desde `movimientos` (+ `tcDiario` + medios), nunca se guarda como copia. Es el
mismo riesgo que F9.21 cerró en tarjetas: si se persiste el agregado, deja de cuadrar con
las líneas.

## Tarea
1. **Auditar** dónde la app obtiene los números de Dashboard y Resumen:
   - ¿Se computan en cliente/Functions desde `movimientos`? ✓ ok.
   - ¿Hay alguna colección/doc que **guarde** totales por mes, por categoría, por medio,
     o KPIs (balance, ingresos, salidas, top categorías, pico día, etc.)? → marcar como
     riesgo.
   - Revisar especialmente cualquier "cache" de dashboard (en legacy existía
     `62_DashCache.gs` / `60_Dash.gs`): si se portó como copia persistida, documentarlo.
2. **Regla de derivación**: todo KPI/aggregado se deriva on-read (o cache **invalidable**
   y reconstruible desde `movimientos`, nunca fuente de verdad).
3. **TC y medios en la derivación**: los montos se convierten con `tcDiario` (fuente
   única) y se agrupan con `medioCanonico()` (Efectivo→MP, F9.23). Confirmar que el cálculo
   los usa.
4. **Si hay copias persistidas**: plan para reemplazarlas por cálculo derivado (o por un
   cache claramente marcado como derivado + cómo se invalida). No romper performance:
   si el cálculo on-read es caro, cache derivado es válido **mientras** se reconstruya
   desde movimientos y no se edite a mano.

## Cierre de Etapa A — todo con datos reales (parte de esta PR)
Además de la auditoría de agregados, **F9.25 es el milestone donde el repo queda 100%
sobre datos reales**. Antes de cerrar:
- **Purgar mocks/fallbacks**: ningún dato hardcodeado ni de ejemplo en la ruta de
  render (KPIs de muestra, arrays placeholder, valores por defecto que tapen un fetch
  vacío). Estado vacío real → mostrar "sin datos", no un mock.
- **Una sola fuente por dato**: `movimientos` (verdad) + `tcDiario` (TC) + medios
  (con alias canónico F9.23). Sin copias persistidas (lo de arriba).
- **Confirmar que cada pantalla** (Inicio, Resumen, Tarjetas, Perfil + sub) lee de
  Firestore/Functions, no de constantes locales.

## Aceptación
- Documentado (en el repo) de dónde sale cada número de Dashboard y Resumen.
- Ningún agregado es fuente de verdad: todos reconstruibles desde `movimientos`.
- La suma de las partes cuadra con el detalle (mismo invariante que tarjetas en F9.21).
- Conversión vía `tcDiario`; agrupación de medios vía alias canónico.
- **Sin mocks**: la app corre con datos reales en todas las pantallas; estados vacíos
  reales en vez de datos de ejemplo. → fin de la Etapa A.
