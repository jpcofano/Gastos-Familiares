# F6.4.5 — Addendum 2 (descripción limpia + normalización on-write)

> Modifica el base + addendum 1. Leer los tres. No reescribir. Cierra dos huecos vs el viejo.

## Qué modifica
La salida del clasificador y la clave de aprendizaje del addendum 1: agrega **normalización on-write** y
**descripción limpia**, como hacía `gf_dict_normalizar_()` del viejo.

## 1. Normalización on-write (NO negociable — learning documentado)
- El viejo normalizaba la descripción cruda con las reglas de `Diccionario_Normalizacion` ANTES de
  guardar/buscar. Tu modelo nuevo ya tiene la colección **`reglasNormalizacion`**.
- **El `patron` del diccionario (y la clave del lookup/upsert) es SIEMPRE la versión NORMALIZADA**, no la
  cruda. Tanto el lookup como el aprendizaje normalizan el texto primero con `reglasNormalizacion`.
  Razón (tu propio learning): normalizar solo on-read es insuficiente — los `patron` crudos acumulan
  entradas que nunca matchean lookups futuros.
- **Corrige la clave del addendum 1**: `id = SHA256(normalizar(textoRaw) + banco + tarjeta)`. El `patron`
  dentro del hash es el normalizado, no el crudo.
- Preservar el texto crudo en `patronOriginal` (trazabilidad), igual que el viejo.
- El clasificador (`useDiccionario`) carga `reglasNormalizacion` junto con el diccionario y expone la
  normalización; reusar el mismo normalizador en cliente y en el trigger de aprendizaje (no dos versiones).

## 2. Descripción limpia como salida
- `clasificar(texto, { banco?, tarjeta? })` devuelve ahora también **`descripcionLimpia`** (el
  `descripcionLimpia` del doc del diccionario, o la versión normalizada si no hay alias explícito).
- En la propuesta/alta, el movimiento guarda **`descripcionLimpia`** como descripción visible (ej.
  "MERCPAGO*MINGMINGYU 13/06" → "Mingming Yu"), preservando la cruda para trazabilidad.
- Al aprender, si el usuario editó la descripción, se actualiza `descripcionLimpia` del mapeo.

## 3. Salida completa del clasificador (actualiza la firma del addendum 1)
```
clasificar(texto, { banco?, tarjeta? }) → {
  descripcionLimpia: string | null,
  categoria: string | null,        // valor canónico del catálogo
  subcategoria: string | null,     // validada contra subcategorias/categoriaPadre
  personaDefault: string | null,   // si el mapeo lo trae
  monedaDefault: 'ARS'|'USD'|null,
  confianza: number
} | null
```

## Cierre (suma a lo del addendum 1)
- Prueba de normalización: dos descripciones crudas que normalizan al mismo `patron` (ej. con/ sin fecha
  pegada) → caen en el MISMO mapeo del diccionario (no crean dos entradas).
- Prueba de descripción limpia: importar un texto crudo feo → el movimiento queda con la descripción limpia.
