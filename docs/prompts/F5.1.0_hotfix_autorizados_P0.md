1. `firestore.rules` → `match /autorizados/{uid} { allow read, write: if isSignedIn() && request.auth.uid == uid; }`
   (el comentario propio dice "escrito por el cliente al resolver memberId").

3. `src/hooks/useMiembro.ts` (~línea 49):
   `setDoc(doc(db,'autorizados',user.uid), { memberId, rol })`.

Explotación: cualquier cuenta de Google logueada hace por SDK directo
`setDoc('autorizados/'+uid, { rol:'admin', memberId:'Juan' })` → admin total.

## Endurecimientos opcionales (se ven ahora que se leyeron las Rules enteras)
- **Cerrar la enumeración de memberIds**: una vez que `/autorizados` se siembra por email,
  `esMiembro()` ya es true al loguear un email sembrado SIN depender de que el cliente lea config.
  Se puede subir `config/{doc}` de `read: isSignedIn()` a `read: esMiembro()`. Verificar que no
  rompa el bootstrap (el cliente necesita config para categorías/bancos/tarjetas, no para autorizar).
- **`mes` vs `fecha` en Rules**: la regla de `create` de `movimientos` valida `mes.matches('^[0-9]{4}-[0-9]{2}$')`
  (solo formato), no que `mes` derive de `fecha`. Firestore Rules no formatea timestamps, así que la
  invariante "mes se deriva de fecha" sigue dependiendo del cliente + Validator 10. Dejar registrado
  como limitación conocida (no se puede cerrar a nivel Rules de forma limpia).
