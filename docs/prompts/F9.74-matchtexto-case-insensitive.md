# F9.74 — matchTexto case-insensitive (incluye/excluye)

## Contexto / síntoma

Un `recibo_servicio` de Expensas (`comercioRazonSocial` = `destinoNombre` =
`"Cons. Prop. Pje del Signo 4032/42"`, hash
`d895a5612e673507d95ff4442b9e7774ffdca67be3770ca2dac8fa05de8248ce`) cae en **rama 3**
a pesar de que el ítem esperado de Expensas tiene el `matchTexto.incluye`
poblado con `"Cons. Prop. Pje del Signo"`.

No es falta de texto ni problema de destino. Verificado en `index.ts`:
- rama 0 (dedup) no aplica, rama 1 (reconciliación por payee) no aplica —el pago aún no existe—,
- `matchPorDestino` cae por fall-through F6.9.7 (destino en frío no cortocircuita),
- el pipeline **sí** llega a `matchConEsperados` → `evaluarMatchTexto`, y ahí muere.

## Causa raíz

`evaluarMatchTexto` (server) y `checklist.ts` (cliente) bajan a minúscula el **texto objetivo**
pero **no el patrón** de `incluye`/`excluye`. `String.includes` es *case-sensitive*: un patrón
guardado con mayúsculas (`"Cons. Prop. Pje del Signo"`) nunca matchea contra el objetivo en
minúscula (`"cons. prop. pje del signo 4032/42"`) → filtro descarta el ítem → rama 3.

## Alcance

Dos sitios de evaluación con el bug idéntico (deuda gemela cliente/server):

1. `functions/src/matchLogica.ts` → `evaluarMatchTexto` — rama 2 del pipeline de comprobantes.
2. `src/datos/checklist.ts` → arma "movimientos del ítem" (alimenta la card PENDIENTE y el
   conteo `porRevisar`). Si se arregla solo el server, el display del cliente queda desincronizado.

**NO tocar** `src/datos/clasificador.ts`: es el diccionario legacy (`patron`/`tipoMatch`), otro
store, y ya baja ambos lados (`patronNorm ... toLowerCase()`).

## Diffs

### 1) `functions/src/matchLogica.ts`

```
BUSCAR:
  return mt.incluye.some(p => t.includes(p)) && !mt.excluye.some(p => t.includes(p));
REEMPLAZAR:
  return mt.incluye.some(p => t.includes(p.trim().toLowerCase())) && !mt.excluye.some(p => t.includes(p.trim().toLowerCase()));
```

### 2) `src/datos/checklist.ts`

```
BUSCAR:
      const inc = item.matchTexto.incluye.some(t => desc.includes(t));
      const exc = item.matchTexto.excluye.some(t => desc.includes(t));
REEMPLAZAR:
      const inc = item.matchTexto.incluye.some(t => desc.includes(t.trim().toLowerCase()));
      const exc = item.matchTexto.excluye.some(t => desc.includes(t.trim().toLowerCase()));
```

`.trim()` come espacios sueltos de los chips del editor (`ConfigEsperados.tsx`).

## Regression tests (post-deploy)

1. **Caso que disparó el bug:** re-correr el match del comprobante hash
   `d895a56…de8248ce` ("Cons. Prop. Pje del Signo 4032/42"). Esperado:
   `propuestaMatch.rama === 2` con `itemEsperadoId` = id del ítem Expensas vigente en
   `itemsEsperados` (ya no rama 3).
2. **Ciclo de aprendizaje:** al confirmar/vincular ese comprobante, `aprenderDestino` persiste
   el destino (CBU `0070127920000005316566`) → `itemEsperadoId`. Verificar en Firestore que
   queda el doc en `destinos` con el FK. Próximo comprobante del mismo consorcio: rama 2 vía
   `matchPorDestino` (origenDestino).
3. **Checklist cliente coherente:** en Gastos Fijos, el ítem Expensas ahora lista ese movimiento
   como match (antes lo ignoraba por el mismo bug). El conteo `porRevisar` refleja el cambio.
4. **No regresión — patrones ya en minúscula:** un ítem cuyo `matchTexto.incluye` ya estaba en
   minúscula (p.ej. algún colegio) sigue matcheando igual (lowercasing es no-op sobre minúscula).
5. **`excluye` con mayúsculas (ver caveat):** si existe algún `excluye` con mayúsculas, ahora
   **empieza a excluir** (antes estaba silenciosamente roto). Verificar que ningún match legítimo
   se pierda por esto.

## Caveat a revisar ANTES de deployar

Para `incluye` el fix es seguro: solo puede *empezar* a matchear lo intencionado, nunca rompe un
match que hoy anda. El único cambio de comportamiento está en `excluye`: un patrón de exclusión con
mayúsculas hoy no excluye nada; con el fix pasa a excluir. Es lo correcto, pero revisá los `excluye`
de `itemsEsperados` por si alguno con mayúsculas empieza a filtrar un movimiento que hoy sí entra.

## Fuera de alcance (a propósito)

Normalizar acentos/puntuación cambiaría la semántica de *todos* los ítems y puede romper otros
matches — va aparte con su propio test si se decide. Este prompt solo corrige case + trim.
