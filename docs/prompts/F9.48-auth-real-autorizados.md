# F9.48 — Etapa B: auth real en dispositivo + reconciliar `/autorizados`

> Reglas: doc-first. **Toca `src/firebase.ts`, `src/hooks/useMiembro.ts` (+ `AppShell`
> estado nuevo), y agrega un check de reconciliación al gate (`validators/checks.ts`).**
> Cierra el acceso real (login Google + familia) para la app en prod. Cero cambios de modelo.
> Corre DESPUÉS de que el import de F9.47 cierre 21/21 contra prod.

## Contexto
Con los datos en Firestore prod (F9.47), falta que la familia entre de verdad: login Google,
gate admin/dependiente, y que nadie de afuera lea nada. La auth ya está cableada
(`signInConGoogle`, `useMiembro`, `AppShell` con estados
`cargando|noAutenticado|noAutorizado|autenticado`). Pero hay **3 agujeros** que en prod dan
"entré pero no veo nada" o "apunto al proyecto equivocado".

## Diagnóstico (auditado contra `src/`)
1. **`firebase.ts` hardcodea el proyecto viejo.**
   `projectId: 'gastos-familiares'` (no `-e6415`) está **hardcodeado** (no detrás de env), y
   `authDomain` default = `gastos-familiares.firebaseapp.com`. En un build de prod esto
   apunta auth + Firestore al proyecto equivocado → login y lecturas fallan. El resto del
   config sí sale de `VITE_FIREBASE_*`, pero `projectId` no.
2. **Dos fuentes de verdad para el acceso, que pueden driftar.**
   - **Cliente (UI gate):** `useMiembro` → `resolverMiembro(email, config)` matchea contra
     `config/familia.miembros[*].emails`.
   - **Servidor (Rules):** `firestore.rules` gatea por `/autorizados/{email.lower()}`
     (`memberId` + `rol`) **y** `request.auth.token.email_verified == true`.
   Ambas se siembran de `data.usuarios`, pero el CRUD de miembros (F9.37) y de Mis datos
   (F9.36) editan una sin la otra → drift. Si un email está en `miembros` pero no en
   `/autorizados`, el cliente lo deja entrar y **toda lectura la rechazan las Rules** →
   "entré pero no veo nada". Hay que reconciliarlas y dejar un check que lo vigile.
3. **El cliente no mira `email_verified`; las Rules sí.**
   `resolverMiembro` no chequea `user.emailVerified`. Una cuenta con email no verificado pasa
   el UI gate (`autenticado`) pero las Rules deniegan todo → pantalla rota sin explicación.
   (Las cuentas Google suelen venir verificadas, pero el gate tiene que ser defensivo y dar
   un mensaje claro, no permission-denied silencioso.)

## Tarea

### 1. `firebase.ts` — config de prod correcta y por env
- `projectId` debe salir de `import.meta.env.VITE_FIREBASE_PROJECT_ID ?? 'gastos-familiares-e6415'`
  (no hardcodear el viejo). Igual `authDomain` →
  `?? 'gastos-familiares-e6415.firebaseapp.com'`, `storageBucket` →
  `?? 'gastos-familiares-e6415.appspot.com'`.
- Dejar documentado (comentario + el runbook de F9.47) las vars de `.env.production`:
  `VITE_FIREBASE_API_KEY`, `VITE_FIREBASE_AUTH_DOMAIN`, `VITE_FIREBASE_PROJECT_ID`,
  `VITE_FIREBASE_APP_ID`, `VITE_FIREBASE_MESSAGING_SENDER_ID` (de la consola Firebase del
  proyecto `-e6415`, Web App config).
- El branch `import.meta.env.DEV` que conecta al emulador queda igual; solo asegurate que un
  build de prod (`vite build`) NO entre a ese branch.

### 2. `useMiembro` — chequear `email_verified` y dar estado propio
- Agregar estado `'emailNoVerificado'` al union `Estado`.
- En el efecto: si `user && !user.emailVerified` → `setEstado('emailNoVerificado')` **antes**
  de cargar la config (no tiene sentido resolver miembro si las Rules lo van a rechazar).
- En `AppShell`, branch nuevo para `emailNoVerificado`: mensaje claro
  ("Verificá tu email con Google para entrar") + botón Salir. NO mandarlo a `noAutorizado`
  (es otra causa).

### 3. Reconciliación `/autorizados` ↔ `config/familia.miembros` (check de gate)
Nuevo check en `validators/checks.ts` (suma al `npm run validate`, sirve emulador y prod):
- Para cada email en `config/familia.miembros[*].emails`: debe existir
  `/autorizados/{email.lower()}` con el **mismo `memberId`** (la key del miembro) y el
  **mismo `rol`**. Y a la inversa: cada doc de `/autorizados` debe corresponder a un email
  presente en `miembros`. Cualquier mismatch = **FAIL** con el detalle del email y el lado
  que falta.
- Caso **María con 2 emails** (personal + Accenture): los **dos** emails tienen que estar en
  `miembros.María.emails` **y** tener su doc en `/autorizados` con `memberId: 'María'`.
  Verificarlo explícitamente (es el caso que más fácil drifta).
- Caso **`activo:false`**: recordá la decisión (CLAUDE.md F9.37) — desactivar un miembro
  **borra sus `/autorizados`** (eso corta el acceso); `activo:false` por sí solo no revoca.
  El check debe tolerar un miembro `activo:false` SIN doc en `/autorizados` (es lo correcto),
  pero FAIL si está `activo:true` y le falta el `/autorizados`.

### 4. Verificación operativa en dispositivo (al runbook de F9.47)
Agregar al `docs/RUNBOOK-migracion-prod.md` un bloque "Auth real (F9.48)", a correr por el
usuario con la app build de prod apuntando a `-e6415`:

- Una cuenta con **email sin verificar** cae en `emailNoVerificado`.
- Los **2 emails de María** resuelven ambos a María (sin warning de "matchea más de un
  miembro").
