# F5.1 — Rules de escritura + alta manual de movimientos

**Fase:** F5 (flujos de escritura) · **Sub-paso:** 5.1
**Prerequisito:** F4.3 cerrado (scope por persona probado en ambas direcciones, 10 OK / 0 FAIL).
**Scope:** Rules + formulario de alta + ajuste de validators. Solo emulador.

---

## Contexto

Primer flujo de escritura del sistema. Hasta acá Firestore era read-only para el cliente; este paso abre la creación de movimientos con validación en Rules (la UI valida por usabilidad, las Rules validan por seguridad).

Decisiones cerradas que aplican:
- Admins: crean para cualquier persona. Dependientes: crean SOLO con su propia persona; no editan ni borran (la edición/borrado de admins llega en F5.2 — en este paso NADIE edita ni borra).
- `mes` se deriva SIEMPRE de `fecha` (decisión registrada). El form no pide mes.
- Audit: `creadoEn`, `actualizadoEn` (= request.time), `creadoPor` (= memberId del autor).

## Qué hace F5.1

### 1. Rules — `movimientos` create

```
allow create: if esMiembro()
  && request.resource.data.creadoPor == miMemberId()
  && (esAdmin() || request.resource.data.persona == miMemberId())
  && request.resource.data.tipo in ['Gasto','Ingreso']
  && request.resource.data.moneda in ['ARS','USD']
  && request.resource.data.monto is number && request.resource.data.monto > 0
  && request.resource.data.fecha is timestamp
  && request.resource.data.mes is string
  && request.resource.data.mes.matches('^[0-9]{4}-[0-9]{2}$')
  && request.resource.data.creadoEn == request.time
  && request.resource.data.actualizadoEn == request.time
  && request.resource.data.descripcion is string
  && request.resource.data.descripcion.size() > 0;
```
(Adaptar sintaxis; lo importante: autor honesto, scope de dependiente, campos mínimos válidos, audit no falsificable. `update`/`delete` siguen DENEGADOS para todos en este paso.)

### 2. Alta manual — UI

- Botón "+" visible en Dashboard (admin y dependiente).
- Form: fecha (default hoy), tipo (Gasto/Ingreso), descripción, monto, moneda, categoría + subcategoría (desde catálogos `subcategorias`), etiqueta (opcional, de `etiquetas`), banco (opcional, de `config`), persona (admin: selector con los 4; dependiente: fijo en sí mismo, no editable), toggle "incluir en resumen del mes" (default ON — un gasto manual es caja real; ajustable).
- Derivados automáticos (NO editables): `mes` desde fecha; `subtipo='Manual'`; `excluirDash=false`; `tcUsdArs` por lookup en `tcDiario` (TC exacto de la fecha o el último anterior — replicar la lógica de `tcForDate` en la capa de datos del cliente); `pagado=true`; resto de campos null.
- Post-guardado: re-fetch one-shot del mes visible (optimistic/realtime es F5.3). Mensaje de éxito sobrio.

### 3. Validators — scope a documentos del seed

Con escrituras habilitadas, los checks de conteo (`firestore == excel`) romperían ante el primer alta manual. Ajustar: los validators que comparan contra el Excel filtran por `idLegacy != null` (solo documentos sembrados). Los movimientos creados por el cliente quedan fuera del scope del validator, EXCEPTO el invariante `mes == YYYY-MM(fecha)` (validator 10), que aplica a TODOS los documentos.

### 4. Pruebas

- Admin crea un gasto para María → aparece en Dashboard y suma en totales (con TC correcto del día).
- Dependiente (Federico) crea un gasto propio → ok; intenta crear con `persona='Juan'` (manipulando desde consola) → **denied**.
- Cualquiera intenta `update` o `delete` desde consola → **denied**.
- Falsificar `creadoPor` o `creadoEn` desde consola → **denied**.
- Alta con fecha de un día sin TC en `tcDiario` → toma el TC anterior, no falla.
- `npm run validate` después de crear movimientos de prueba: 10 OK / 0 FAIL (validators scopeados).
