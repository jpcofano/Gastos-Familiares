# F9.36 — CRUD configs: Medios de pago + Mis datos (patrón base de edición)

> Reglas: doc-first · una PR. Primera del bloque "configs editables" (Opción 1: editar
> antes de migrar). Establece el **patrón reutilizable de edición** (leer→editar→guardar en
> Firestore) con las dos configs más simples, para que Miembros/Categorías lo reusen.

## Contexto
F9.32 encontró que las configs **leen Firestore real** (post F9.26) pero **no editan**
(salvo Pagos esperados). Esta serie agrega el CRUD. Empezamos por las dos más directas.

## Patrón base (reusable)
- Form controlado + estado dirty + acción guardar/cancelar; persistir con la capa de datos
  existente (la misma que ya lee). Optimistic update + rollback si falla.
- Permisos: las configs de familia son **admin**; dependiente las ve read-only (respetar
  `useMiembroCtx` / Firestore Rules — no romper el scoping ya existente).
- Estados: guardando / ok / error; validación inline. Sin `#fff` hardcodeado (tokens kit).

## 1. Medios de pago (`Perfil › Medios`)
- **Editar**: nombre, color, tipo (Banco/Billetera/Efectivo), dominio (logo Brandfetch).
- **Alta / baja** de medio.
- Mantener la regla F9.23: **Efectivo** es alias cosmético de MP (`aliasDe:'mp'`, `oculto`)
  — no se puede "des-aliasar" rompiendo el modelo; el canónico se respeta al agrupar.
- No duplicar Efectivo en la lista editable.

## 2. Mis datos (`Perfil › Mis datos`)
- **Editar** el perfil del miembro logueado: nombre visible, email (si aplica), apariencia/
  tema. Identidad base ya es real (`useMiembroCtx`) — acá se vuelve editable.
- Un dependiente edita **lo suyo**; un admin puede editar miembros desde F9.37 (no acá).

## Aceptación
- Medios y Mis datos se **editan y persisten** en Firestore (no solo lectura).
- Patrón de edición reutilizable extraído (hook/scaffold) para las siguientes configs.
- Scoping admin/dependiente respetado; alias Efectivo intacto; build verde.

## Notas
- Base para F9.37 (Miembros) y F9.38 (Categorías). Ver `CLAUDE.md` (F9.36 split).
