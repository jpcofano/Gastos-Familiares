# F6.9 — Reconciliación pago↔obligación por payee (no por monto)

## Contexto / problema

Flujo objetivo: una **factura** registra un movimiento (la obligación, vencimiento ~10 días,
`confirmadoPago: false`). Después entra el **pago** y debe **confirmar ese mismo movimiento**
(`confirmadoPago → true`), **sin crear un segundo movimiento** (no doble conteo).

Hoy no pasa: en `matchComprobante`, cuando el pago llega a un payee ya aprendido, `matchPorDestino`
corta **antes** que la reconciliación contra movimiento existente (que vive sepultada en
`calcularPropuesta`/rama 1) y devuelve rama 2 `esAdicional` o rama 3 → **crea un mov nuevo → duplica**.

La llave de reconciliación correcta es **el payee (CBU/CUIT/alias), no el monto** — porque puede
haber pagos del mismo monto a payees distintos. El monto solo **desempata** dentro del mismo payee.

Datos que lo habilitan (verificado en el prompt del extractor, `index.ts` ~62-65):
- Factura de comercio → llena `destinoCuit` con el CUIT del receptor (el comercio). El mov de la
  factura **trae `destinoCuit`**.
- Transferencia/pago → trae `destinoCbu` (cuenta receptora) y normalmente también `destinoCuit`.
- Ambos comparten identificador de payee (`destinoCuit` casi siempre; `destinoCbu`/`destinoAlias` si están).

## Decisión cerrada (separación de responsabilidades)

- **Reconciliar** (nuevo): pago ↔ obligación existente, por **payee compartido + monto exacto**, directo
  contra los campos `destino*` de los movs. **NO usa el doc `destinos/{id}` aprendido.** Confirma (rama 1).
- **Clasificar** (lo de hoy, intacto): `matchPorDestino` + `aprenderDestino` + doc `destinos/{id}` siguen
  igual, para **clasificar un mov genuinamente nuevo** (pago suelto sin obligación previa).
- Orden: **reconciliar primero, clasificar después.** Nada se elimina del aprendizaje de destino.

### 2. `functions/src/matchLogica.ts` — helper nuevo `reconciliarPorPayee`
Junto a `matchConMovimientos`. Devuelve los movs-obligación que matchean el pago entrante:
```ts
export function reconciliarPorPayee(
  datos: DatosExtractosMin,
  movs: MovimientoMin[],
): MovimientoMin[] {
  const pCuit  = datos.destinoCuit  ?? null;
  const pCbu   = datos.destinoCbu   ?? null;
  const pAlias = datos.destinoAlias ?? null;
  if (!pCuit && !pCbu && !pAlias) return [];
  if (datos.montoTotal == null) return [];

  return movs.filter(m => {
    if (m.tipo !== 'Gasto')        return false;
    if (m.moneda !== datos.moneda) return false;
    if (m.confirmadoPago)          return false;           // solo obligaciones abiertas → idempotencia
    const mismoPayee =
      (!!pCuit  && m.destinoCuit  === pCuit)  ||
      (!!pCbu   && m.destinoCbu   === pCbu)   ||
      (!!pAlias && m.destinoAlias === pAlias);
    if (!mismoPayee) return false;
    return Math.abs(m.monto - (datos.montoTotal as number)) < 0.01;  // monto exacto, desempate
  });
}
```
Notas:
- `destinoNombre` queda **fuera** de la llave (string ruidoso); solo CUIT/CBU/alias.
- **Sin gate de fecha** dentro del candidate set (payee+monto es la llave; cubre los 10 días aunque
  crucen de mes). El candidate set sigue acotado por la ventana de `mes` ±1 que ya arma `matchComprobante`.

Solo corre para comprobantes de pago:
```ts
const esPago = datos.tipoDocumento === 'transferencia'
            || datos.tipoDocumento === 'comprobante_pago';
if (esPago) {
  const reconc = reconciliarPorPayee(datos, movs);
  if (reconc.length === 1) {
    await ref.update({

        rama: 1,

    });
    console.log(`[matchComprobante] ${hashActual} → rama 1 (reconciliación por payee, mov ${reconc[0].id})`);
    return;
  }
  if (reconc.length > 1) {
    await ref.update({
      propuestaMatch: {
        rama: 1,
        candidatos: reconc.map(m => ({
          tipo: 'movimiento' as const,

          descripcion: m.descripcion, monto: m.monto, moneda: m.moneda,

```
- **Importante:** el gate `esPago` evita que una **segunda factura** del mismo payee+monto confirme la
  obligación de la primera. Una factura crea obligación, no la salda.
- El cliente ya rutea rama 1 (con `movimientoId` o con `candidatos`) a `confirmarRama1` — **no tocar el cliente.**
  `confirmarRama1` ya setea `confirmadoPago` por fecha del pago (≤ hoy → true), propaga destino y limpia `seedImport`.

## Build + test
- `cd functions && npm run build` antes de probar (el emulator corre `functions/lib`, no el TS).
- Pruebas en el emulator:
  1. **No-duplicado:** factura → mov `confirmadoPago:false`. Entra pago mismo payee+monto → **confirma ese mov**
     (`confirmadoPago:true`), **no aparece un segundo movimiento**.
  2. **Cross-mes:** factura emitida fin de mes, pago al mes siguiente (>7 días) → igual reconcilia (sin gate de fecha).
  3. **Idempotencia:** reprocesar el mismo pago → el mov ya está `confirmadoPago:true` → queda fuera del scan, no re-confirma.
  4. **Seguridad de colisión:** dos obligaciones abiertas de **distinto** payee con el **mismo monto** → el pago
     confirma solo la del payee correcto (no la otra). Dos del **mismo** payee+monto → `candidatos`, elige el usuario.
  5. **Clasificación intacta:** pago suelto a payee conocido **sin** obligación abierta → cae a `matchPorDestino`
     y prellena cat/subcat como hoy (el aprendizaje de destino no cambió).
