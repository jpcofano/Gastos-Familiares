# F6.8 follow-up — candidatos de match legibles (descripción · monto · fecha)

**Problema:** cuando el match contra movimientos devuelve varios candidatos (rama 1, lista
"Seleccioná el movimiento correspondiente"), la UI muestra el doc-id crudo (`HIS_672831ab…`).
No hay forma de elegir bien porque el id no dice nada.

**Causa:** el candidato se arma con solo `{ tipo, id, score }`. En el punto de construcción
está disponible el movimiento entero (`c.mov`) pero se descarta. La UI renderiza `c.id.slice(0,16)`.

**Fix:** denormalizar descripción/monto/moneda/fecha en el candidato (server-side, donde está
`c.mov`) y renderizarlos en la UI, reusando el `fmtMonto` que ya existe.

> ⚠️ El tipo `candidatos` vive en DOS copias sincronizadas a mano
> (`functions/src/matchLogica.ts` y `src/types/index.ts`). Tienen que quedar idénticas,
> misma disciplina que el normalizador.

---

## Sitio 1 — `functions/src/matchLogica.ts`, tipo (≈ línea 46)

Reemplazar:
```ts
  candidatos?: Array<{ tipo: 'movimiento' | 'esperado'; id: string; score?: number }>;
```
por:
```ts
  candidatos?: Array<{
    tipo: 'movimiento' | 'esperado';
    id: string;
    score?: number;
    descripcion?: string;
    monto?: number;
    moneda?: 'ARS' | 'USD';
    fecha?: string;            // ISO YYYY-MM-DD, para mostrar al usuario
  }>;
```

## Sitio 2 — `functions/src/matchLogica.ts`, construcción rama 1 múltiple (≈ línea 195)

Reemplazar:
```ts
      candidatos: movCands.map(c => ({ tipo: 'movimiento' as const, id: c.mov.id, score: c.score })),
```
por:
```ts
      candidatos: movCands.map(c => ({
        tipo:        'movimiento' as const,
        id:          c.mov.id,
        score:       c.score,
        descripcion: c.mov.descripcion,
        monto:       c.mov.monto,
        moneda:      c.mov.moneda,
        fecha:       c.mov.fecha.toISOString().slice(0, 10),
      })),
```
(No tocar la construcción de candidatos de esperados ≈ línea 207: esa rama no se renderiza
como picker hoy.)

## Sitio 3 — `src/types/index.ts`, `PropuestaMatch.candidatos`

Aplicar EXACTAMENTE el mismo cambio de tipo que el Sitio 1 (las dos copias idénticas).

## Sitio 4 — `src/vistas/Comprobantes.tsx` (≈ línea 111)

Reemplazar:
```tsx
                <code>{c.id.slice(0, 16)}…</code>
```
por:
```tsx
                <span className="cmp-candidato-info">
                  {c.descripcion ?? `${c.id.slice(0, 16)}…`}
                  {c.monto != null && ` · ${fmtMonto(c.monto, c.moneda ?? 'ARS')}`}
                  {c.fecha && ` · ${c.fecha}`}
                </span>
```
`fmtMonto` ya está a nivel de módulo en este archivo; no importar nada. El fallback a `c.id`
cubre propuestas viejas guardadas sin los campos nuevos. (Opcional: una regla CSS mínima para
`.cmp-candidato-info`; funciona sin ella.)

## Build y prueba
1. `cd functions && npm run build` — el emulador corre el compilado.
2. Reset de `emulator-data` para no arrastrar propuestas viejas, levantar emuladores, re-seed.
3. Subir un comprobante que genere **varios** candidatos de movimiento (caso ITPA SA del test).
   Éxito: los radios muestran `descripción · $monto · fecha` en vez de `HIS_…`.

## Criterio de cierre
- Los candidatos de movimiento se leen al ojo humano.
- El tipo quedó idéntico en las dos copias.
- Cero cambios fuera de los cuatro sitios.
