# F9.60.1 — corrección de F9.60: concatenar `comercioRazonSocial` + `destinoNombre` (NO `??`)

> **Corrige** la implementación de F9.60. Sub-addendum de fix. Un solo archivo de código.
> Numeración auditada contra origin: HEAD `309b566` (F9.59b); F9.60 aplicado local (sin push aún).

---

## Qué pasó

F9.60 se implementó con un **fallback** (`comercioRazonSocial ?? destinoNombre`) en vez de la **concatenación**
que pedía el diff (`[comercioRazonSocial, destinoNombre].join(' ')`). Son cosas distintas y el fallback **no
resuelve el caso que originó el fix**.

`??` devuelve `destinoNombre` **solo si `comercioRazonSocial` es null**. Pero en el comprobante real de Baggini:

```
comercioRazonSocial: "Mercado Pago"          ← NO es null
destinoNombre:       "Baggini Juan Francisco"
```

Como "Mercado Pago" no es null, el `??` nunca llega a `destinoNombre` → `matchTexto ["rugby","baggini"]` se evalúa
contra `"mercado pago"` → no matchea → **sigue en rama 3**. El bug queda intacto.

El fallback solo funciona cuando `comercioRazonSocial` es null (ej. una transferencia a "EDESUR SA" con comercio
null). El caso difícil —y el que motivó el prompt— es justo cuando el comercio viene poblado con la **plataforma**
(Mercado Pago) y el payee real está en `destinoNombre`. Ahí hay que mirar **los dos campos juntos**.

| Caso | comercioRazonSocial | destinoNombre | `??` | `join` |
|------|---------------------|---------------|------|--------|
| Baggini (transferencia MP) | "Mercado Pago" | "Baggini Juan Francisco" | "mercado pago" → **rama 3 ✗** | "mercado pago baggini juan francisco" → **rama 2 ✓** |
| EDESUR (comercio null) | null | "EDESUR SA" | "edesur sa" → rama 2 ✓ | "edesur sa" → rama 2 ✓ |
| Edenor (factura) | "…(Edenor)" | "…(Edenor)" | "…(edenor)" → rama 2 ✓ | "…(edenor) …(edenor)" → rama 2 ✓ |

La concatenación cubre los tres; el fallback falla en el primero.

## Fix

`functions/src/matchLogica.ts` → función `matchConEsperados`.

El cuerpo actual (post-F9.60) usa un fallback de una sola fuente, del estilo:

```ts
const comercio = (datos.comercioRazonSocial ?? datos.destinoNombre)?.toLowerCase();
if (!comercio) return [];
// ...
return evaluarMatchTexto(comercio, item.matchTexto);
```

Reemplazá **todo el cuerpo** de la función (desde la primera línea después de la firma `): ItemEsperadoMin[] {`
hasta antes del `}` de cierre) por esta versión con **concatenación**:

```ts
  // F9.60.1 — evaluar matchTexto contra emisor Y beneficiario CONCATENADOS (no un fallback `??`).
  // En transferencias el comercioRazonSocial es la plataforma ("Mercado Pago") y el payee real
  // (contra el que se configura matchTexto) viaja en destinoNombre. Un `??` no sirve: como
  // comercioRazonSocial viene poblado con la plataforma, nunca se llegaría a destinoNombre y el
  // itemEsperadoId quedaría sin poblar (rama 3), rompiendo el ciclo de aprendizaje del destino.
  const texto = [datos.comercioRazonSocial, datos.destinoNombre]
    .filter((s): s is string => !!s)
    .join(' ');
  if (!texto) return [];

  return items.filter(item => {
    if (!item.activo)   return false;
    if (item.moneda !== datos.moneda) return false;
    if (!item.matchTexto || item.matchTexto.incluye.length === 0) return false;
    return evaluarMatchTexto(texto, item.matchTexto);
  });
```

Notas:

- `evaluarMatchTexto` ya hace `toLowerCase()` internamente — no normalizar en el `join`.
- No cambia la firma ni los tipos. `DatosExtractosMin` ya expone `destinoNombre`.
- No tocar el frontend (`checklist.ts` / `movimientosDelItem`).
