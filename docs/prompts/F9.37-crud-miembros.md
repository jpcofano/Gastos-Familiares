# F9.37 — CRUD configs: Miembros (alta / rol / permisos)

> Reglas: doc-first · una PR. Reusa el patrón de edición de F9.36. **Sensible**: toca roles
> y permisos → cuidar Firestore Rules. Solo **admin** edita miembros.

## Tarea (`Perfil › Miembros`)
- **Lista** de miembros real (`config/familia`) — ya se lee post F9.26.
- **Alta** de miembro: nombre, email, rol (admin / dependiente). Genera el memberId
  canónico (consistente con F9.24: `persona` referencia ese id).
- **Editar**: nombre, email, rol/permisos.
- **Baja / desactivar**: no borrar en duro si tiene movimientos asociados — **desactivar**
  (los movimientos históricos con su memberId siguen válidos). Confirmar política.
- **Permisos**: solo admin entra a esta config. Un dependiente no la ve / read-only.

## Reglas / integridad
- El memberId es la clave que usan `movimientos.persona`, el scoping de Dashboard/Resumen
  y Firestore Rules. **No** reasignar ni reciclar ids al editar.
- Revisar que dar de baja un miembro no rompa los filtros (movimientos quedan; el miembro
  queda inactivo, no se borra el id).
- Sincronizar con **Firestore Rules**: alta/cambio de rol debe respetar (y no eludir) las
  reglas de admin.

## Aceptación
- Admin puede **crear, editar y desactivar** miembros, persistido en Firestore.
- Roles aplicados: un nuevo dependiente ve solo lo suyo; admin ve todo.
- memberIds estables; movimientos históricos no se rompen.
- Dependiente no puede editar miembros (UI + Rules). Build verde.

## Notas
- Depende del patrón de F9.36. Distribución de roles la valida el gate F9.31 (validador de
  roles). Ver `CLAUDE.md`.
