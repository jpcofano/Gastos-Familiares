# F9.99.3 — Informes legibles + permeado de informes en posiciones

> Prompt de feature para Code. Va a `docs/prompts/`.
> Incluye un FIX DE CRASH LATENTE (parte 0) que bloquea la prueba en curso del
> camino chat — priorizarlo aunque el resto se recorte.

## Contexto
Los tres informes (posición, sectorial, agenda) viven aislados y con render
pobre: el sectorial se muestra como texto plano con los símbolos de markdown a
la vista (`###`, `**`, `-`), y el detalle de cada posición no aprovecha la
información que ya existe en agenda y sectorial. El dueño pide: (a) formato
legible, (b) que los informes "permeen" en las posiciones.

## Parte 0 — FIX CRASH: render de `proximosEventos` (obligatoria, primero)

En `AnalisisIASection` (src/vistas/Patrimonio.tsx ~línea 1605):
```tsx
{analisis.resultado.proximosEventos.map((e, i) => (
  <div key={i} ...>· {e}</div>
))}
```
`EventoProximo` es una unión `{ cuando: string|null; evento: string } | string`
(formato estructurado F9.95+ o string retrocompat). Cuando el evento es objeto
—que es lo que producen la API y los prompts del camino chat—, React lanza
"Objects are not valid as a React child" y **la solapa Tenencias entera se cae**
al expandir esa posición. Hoy no se nota solo porque los análisis viejos no
cargan (incidente permission-denied); en cuanto se importe un análisis vía chat
con eventos estructurados, crash.

Fix: render discriminado —
```tsx
const evTxt = typeof e === 'string' ? e : `${e.cuando ?? 's/f'} — ${e.evento}`;
```
con `cuando` como chip de fecha (mismo estilo que CalendarioCard) cuando exista.

Nota: este bug es de la misma familia que F9.99.2 — los `.tsx` no se
typechequean. Si F9.99.2 parte 2 (tsconfig) ya se aplicó, `tsc --noEmit` debería
haber marcado esta línea; verificar que quede limpio.

## Parte 1 — Formato del panorama sectorial

Hoy (~línea 1936): `whiteSpace: pre-wrap` sobre el markdown crudo.

1. Implementar un renderer markdown liviano PROPIO (sin dependencias nuevas,
   ~50-70 líneas), suficiente para el subset que produce el prompt:
   - `## Título` → header de sección (estilo de los títulos de Card existentes)
   - `**negrita**` → `<strong>`
   - Líneas `- ` / `· ` → bullets con el estilo de riesgos ya usado
   - Líneas que empiezan con "Si " y contienen "→" (escenarios) → bloque
     destacado tipo card gris, mismo patrón visual que queHariaEnCadaCaso
   - Párrafos separados por línea en blanco
   Sanitizar: es texto pegado por el usuario — NO usar dangerouslySetInnerHTML
   con HTML arbitrario; el renderer construye elementos React, nunca inyecta.
2. Secciones colapsables por sector (expandida la primera), para que el informe
   largo no domine la solapa Research.

## Parte 2 — Estructura predecible del sectorial (backend)

Para que el permeado (parte 3) sea robusto, `buildPromptSectorial` debe exigir
headers canónicos por sector con tag de driver:

```
## Energía AR [driver: energia_ar]
## Macro y pesos AR [driver: cer_pesos]
## Soberano USD [driver: soberano]
## Cripto [driver: cripto]
## Global [driver: tech_global]
```

- Instrucción en el prompt: "cada sector es una sección que empieza EXACTAMENTE
  con `## <Nombre> [driver: <driver>]`, usando solo estos drivers: ...".
- `validarResultadoImportado('sectorial')`: además de ≥200 chars, exigir al
  menos un header `## ... [driver: ...]`; si no hay ninguno, rechazar con
  mensaje claro ("el texto no tiene secciones por sector; regenerá con el
  prompt actual").
- Parser compartido `splitSectorialPorDriver(texto): { driver, titulo, cuerpo }[]`
  en el cliente (src/datos/patrimonioIA.ts) usado por Research y por posiciones.
- Actualizar también el prompt de uso standalone si existe en docs (U-02).

## Parte 3 — Permeado de informes en las posiciones

En el detalle expandido de cada posición (debajo del Análisis IA), dos bloques
nuevos alimentados por datos YA cargados (sin fetch extra):

1. **"De la agenda"**: los próximos 2-3 eventos de `agenda` cuyo `driver`
   coincida con el driver de la posición, ordenados por fecha. Requiere helper
   `tickerADriver(ticker, sectorDisp): Driver` (mapeo determinista:
   energía/gas/utilities AR → energia_ar; BTC/ETH/AAVE/UNI → cripto; GD* →
   soberano; Lecap/CER → cer_pesos; ACN/GLOB/CVX/VZ/B → tech_global; default
   → otro). Colocarlo en un módulo de datos, no inline, porque F9.99.1 (lote)
   lo va a reusar.
2. **"Panorama del sector"**: la sección del sectorial cuyo driver coincide,
   colapsada por defecto (solo título + primera línea), expandible inline,
   renderizada con el renderer de la parte 1.

Reglas:
- Si no hay agenda o sectorial generados, los bloques no se muestran (sin
  placeholders que ensucien).
- Prop drilling: `TenenciasTab` recibe `agenda` y `sectorial` como props desde
  `Patrimonio()` — MISMO patrón prop que F9.99.2 (nunca referenciar estado del
  padre desde componentes de módulo).
- Mostrar la antigüedad del informe fuente ("panorama de hace 12d") para que
  la información vieja no parezca fresca.
