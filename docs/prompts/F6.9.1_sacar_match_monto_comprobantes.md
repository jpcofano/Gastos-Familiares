# F6.9.1 — Sacar el match por monto del flujo de comprobantes (anti-autovínculo)

## Problema
En `matchComprobante`, cuando F6.9 (`reconciliarPorPayee`) devuelve 0 y `matchPorDestino` devuelve null,
cae a `calcularPropuesta(datos, movs, ...)`, cuyo `matchConMovimientos` matchea por **monto ±5%
(`TOLERANCIA_MONTO`) + mismo mes o ±7d, SIN mirar el payee**. Con un único candidato → `rama 1 single`,
y el cliente la **autoconfirma sin UI** (`useEffect` en `Comprobantes.tsx`). Resultado: facturas y pagos
nuevos se autovinculan en silencio a cualquier mov de monto parecido en el mismo mes.

Verificado en vivo: todos los comprobantes cayeron en `→ rama 1` (pelado, sin "reconciliación por payee"),
incluida una `factura_c`. F6.9 corre pero no es la que matchea.

## Decisión cerrada
En el flujo de comprobantes, **lo único que puede autoconfirmar solo es**:
- **rama 0** (dedup por hash), y
- **F6.9** (reconciliación por payee).

El match por **monto+mes se saca** del flujo de comprobantes. Todo lo demás **propone y abre AltaMovimiento**
para que el usuario valide.

## Cambio (una línea, server-side)
`functions/src/index.ts`, dentro de `matchComprobante`:

```ts
// ANTES
const propuesta = calcularPropuesta(datos, movs, items, mesComp);

// DESPUÉS — sin movs: se elimina el rama-1 por monto+mes (autovínculo silencioso).
// La reconciliación de pagos ya la hace F6.9 por payee (arriba); el dedup lo hace rama 0.
// matchConEsperados (rama 2) usa `items`, no `movs`, así que el linkeo a esperado se conserva.
const propuesta = calcularPropuesta(datos, [], items, mesComp);
```

No tocar nada más. En particular:
- `matchPorDestino(datos, movs, mesComp)` sigue recibiendo `movs` reales (clasificación / `yaEnMes`): **no cambia**.
- El bloque F6.9 sigue recibiendo `movs` reales: **no cambia**.
- El cliente **no se toca**: rama 1 con `movimientoId` ahora solo viene de F6.9 (auto-confirm correcto) o dedup.

## Efecto por tipo de comprobante (resultado esperado)
- **Pago con obligación abierta del mismo payee** → F6.9 → `rama 1 (reconciliación por payee)` → autoconfirma, sin duplicar.
- **Pago nuevo** (payee sin obligación) → F6.9 da 0 → clasifica (destino/texto) → **abre AltaMovimiento**, no autovincula.
- **Factura** → no es `esPago` → clasifica + **abre AltaMovimiento** para crear la obligación. Nunca autovincula por monto.
- **Re-subida del mismo PDF** → rama 0 dedup → vincula (correcto).

## Tradeoff conocido y aceptado
Si cargaste un mov **a mano** (sin `destinoCuit`/`destinoCbu`) y después subís su pago, F6.9 no lo encuentra
→ se crea un mov nuevo en vez de autovincular. Es el comportamiento deseado: preferimos que **abra y decida
el usuario** antes que vincular solo por monto.

## Build + test
- `cd functions && npm run build`.
- **Run A (reconciliación):** Clear all data → `01_factura_colegio.png` → `02_pago_colegio.png`.
  Esperado: el pago loguea `→ rama 1 (reconciliación por payee, mov ...)`, **no** se crea segundo mov,
  la obligación pasa a `confirmadoPago:true`.
- **Run B (anti-autovínculo):** Clear all data → `01_factura_colegio.png` → `04_pago_colision.png`
  (Ferretería, mismo monto $220.000, payee distinto, SIN el pago del colegio en el medio).
  Esperado: F6.9 da 0 → **abre AltaMovimiento** (no `→ rama 1` pelado), el mov del colegio **queda intacto**
  (sin pagar). Deben quedar dos movs distintos.
- **Run C (pago nuevo único):** `03_pago_nuevo_unico.png` ($383.700) → abre AltaMovimiento, no vincula a nada.

## Criterio de cierre
Ningún comprobante se autovincula por monto. Solo el pago que reconcilia por payee (Run A) y las re-subidas
(dedup) confirman solos; todo lo demás abre la UI.
