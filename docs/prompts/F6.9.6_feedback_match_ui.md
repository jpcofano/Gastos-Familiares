# F6.9.6 — Feedback de match en la UI: distinguir "pagó factura" / "mov nuevo" / "ya cargado"

## Problema
Hoy el resultado del match se comunica genérico: rama 0 y rama 1 muestran ambas **"Ya cargado"**, y
ramas 2/3 solo muestran un botón **"Revisar y cargar"**. El usuario no distingue tres situaciones que son
semánticamente distintas y que **ya vienen resueltas en `propuestaMatch`**:

1. **Pagó una factura** — un pago saldó una obligación abierta preexistente
   (`reconciliarPorPayee` → rama 1 con `origenReconciliacion:true`). No se crea movimiento nuevo;
   se marca `confirmadoPago:true` en la obligación.
2. **Movimiento nuevo** — el comprobante genera un movimiento (rama 2 = cumple un gasto esperado;
   rama 3 = movimiento libre). Pasa por AltaMovimiento.
3. **Ya cargado** — dedup por hash (rama 0): este archivo ya generó un movimiento; no hay nada nuevo.

Mapa verificado del flujo (`functions/src/index.ts` + `matchLogica.ts`): con `movs=[]` (F6.9.1) la
**rama 1 sale exclusivamente de la reconciliación por payee** (siempre `origenReconciliacion:true`);
`matchPorDestino` solo devuelve rama 2/3. Por eso el mapeo UI es directo y no hay caso "rama 1 por monto".

Además, re-subir el mismo archivo ya se detecta a nivel entrante (`duplicado:true`), pero el cartel
("Ya existe: …") no deja claro que **no se procesa nada nuevo**.

Riesgo: bajo. Es puro cliente salvo una sincronización de tipos (compile-time, sin redeploy).
