# F9.97.1 — CAFCI: parser con estructura confirmada + seed de fondos y mapeo

> Prompt de feature para Code. Va a `docs/prompts/`.
> Origen: la estructura real del JSON de CAFCI quedó confirmada por el código
> AppsScript del sistema anterior (`10_Fondos_CAFCl.js`), que corrió en producción
> contra `api.pub.cafci.org.ar` y pobló las hojas del workbook con datos reales
> (365 filas, control total_share = 100.0 por fondo). Esto reemplaza TODOS los
> supuestos fail-soft del parser actual.

## Contexto

`sincronizarCafci` (functions/src/index.ts) fue escrito con supuestos porque la API
daba 403 fuera del browser. El sistema AppsScript anterior SÍ accedió a la API y su
parser revela la estructura real. El parser actual está mal en tres cosas
estructurales que producirían datos silenciosamente incorrectos (una posición por
fondo en vez de la cartera completa).

## Estructura REAL confirmada del JSON

```
GET https://api.pub.cafci.org.ar/fondo/{fondoId}/clase/{claseId}/ficha

{
  "success": true,
  "data": {
    "model": {
      "nombre": "<nombre de la CLASE>",
      "fondo": { "nombre": "<nombre del FONDO>" }
    },
    "info": {
      "semanal": {
        "fechaDatos": "<fecha de la cartera>",     // ← acá, NO dentro de carteras[i]
        "carteras": [
          { "nombreActivo": "YPF - D", "share": 17.902 },   // ← cada item ES una posición
          { "nombreActivo": "Grupo Fciero Galicia - B", "share": 17.708 },
          ...
        ]
      }
    }
  }
}
```

Correcciones al parser actual, punto por punto:
1. **`carteras[]` es la lista de posiciones directamente.** NO existe sub-array
   `posiciones`/`tenencias`/`items`, NO hay múltiples corridas: es la cartera
   semanal vigente. Eliminar la lógica de `carteras[0]` + sub-array + fallback.
2. **Campo especie = `nombreActivo`** (string, formato "Empresa - Clase").
   Eliminar los candidatos `especie`/`nombreEspecie`/`instrumento`/`descripcion`
   de `parsePosicionCafci` (o dejarlos como fallback DESPUÉS de `nombreActivo`).
3. **Campo peso = `share`** (número, porcentaje ya calculado; la suma por fondo
   da 100.0 ± redondeo — verificado en 14 fondos reales). Eliminar/relegar
   `porcentaje`/`peso`/`porcentajeFondo`/`participacion`/`pct`.
4. **`fechaDatos` está en `data.info.semanal.fechaDatos`**, no dentro de cada
   elemento de `carteras[]`.
5. **Nombres**: usar `data.model.fondo.nombre` (fondo) y `data.model.nombre`
   (clase) para mostrar, con fallback al nombre configurado.
6. **Validación de respuesta**: exigir `json.success === true` y
   `data.info.semanal` presente; si falta, error claro (no continue silencioso).
7. **Headers anti-CloudFront** — usar EXACTAMENTE los que funcionaron en
   producción desde Apps Script (nota el `www.`; el código actual usa
   `https://cafci.org.ar` sin www):
   ```
   User-Agent: Mozilla/5.0
   Accept: application/json, text/plain, */*
   Origin: https://www.cafci.org.ar
   Referer: https://www.cafci.org.ar/
   ```
   Si aun así CloudFront bloquea a la Function en producción, mantener el plan B
   ya previsto (pegado manual de JSON) — no inventar workarounds nuevos.
8. **Control de integridad**: tras parsear, si `sum(share)` del fondo queda fuera
   de [98, 102], marcar el fondo con advertencia visible en la UI (mismo espíritu
   que la hoja Fondos_Control del sistema anterior).

## Criterios de aceptación
1. `sincronizarCafci` con un fondo real (p.ej. 216;1634) trae N posiciones
   (N ≈ 20-40), no 1; suma de share ≈ 100.
2. Especies conocidas quedan mapeadas a ticker automáticamente vía seed
   (GGAL, YPFD, PAMP, etc.); FCI/cauciones/ctas ctes quedan como LIQUIDEZ sin
   pedir mapeo humano.
3. BenchmarkTab muestra la comparación cartera-vs-fondos con al menos 2 fondos.
4. Si la API devuelve 403 desde la Function, el error es claro y el camino de
   pegado manual sigue disponible.
5. Push al cerrar (regla vigente).
