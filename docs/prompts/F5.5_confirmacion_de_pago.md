# F5.5 — Pagado por confirmación (revisa la acción "Registrar" de F5.2)

> Cambio de modelo: el match deja de significar "pagado" automáticamente. El match solo indica que
> apareció un movimiento candidato; **el pagado lo confirma el usuario** con un botón que NO crea
> movimientos — marca el movimiento existente como confirmado.
>
> Esto introduce un dato persistido (`confirmadoPago` en el movimiento), corrigiendo el principio
> "no se materializa" SOLO para esta confirmación. Revisa la acción "Registrar" de F5.2 (que abría
> el alta y creaba un movimiento) — eso se elimina.

## Asunción (vetar si no aplica)
Meses **cerrados/pasados** con match → **pagado** sin pedir confirmación (no se confirma la historia a
mano). La confirmación aplica solo al **mes en curso** (y futuro). Sin migración de datos.

## Cambios

### 1. Tipos
- `Movimiento`: agregar `confirmadoPago?: boolean` (default ausente = false).
- `EstadoItem`: agregar `'por_confirmar'`. Conjunto: `pagado | por_confirmar | parcial | pendiente | vencido | automatico | programado | no_registrado | no_aplica`.

### 2. `estadoItem` (Resumen.tsx)
```ts
if (!aplicaEnMes(item, mes)) return 'no_aplica';

if (matches.length > 0) {
  if (mes < mesActualStr) return 'pagado';              // meses cerrados: match = pagado (asunción)
  const confirmados = matches.filter(m => m.confirmadoPago);
  if (confirmados.length > 0) {
    const montoConf = confirmados.reduce((s,m)=>s+Math.abs(m.monto),0);
    if (item.montoEsperado != null && montoConf < item.montoEsperado * 0.99) return 'parcial';
    return 'pagado';
  }
  return 'por_confirmar';                                // hay match, falta tu confirmación
}
// sin match:
if (item.pagoAutomatico) return 'automatico';
if (mes > mesActualStr) return 'programado';
if (mes < mesActualStr) return 'no_registrado';
if (item.diaVencimiento && item.diaVencimiento < new Date().getDate()) return 'vencido';
return 'pendiente';
```

### 3. Botón "Confirmar pago"
- Visible SOLO en estado `por_confirmar`. (Pendiente no tiene botón: pagado nace del match.)
- Al click → `confirmarPagoEsperado(item, matches)`: en batch, sobre los movimientos matcheados de
  ese esperado/mes, set `{ confirmadoPago: true, itemEsperadoId: item.id, actualizadoEn: serverTimestamp() }`.
  No crea movimientos. Re-fetch al terminar.
- En estado `pagado` (mes en curso), ofrecer un link chico "deshacer" → `desmarcarPago` (set `confirmadoPago:false`).
- **Eliminar de Resumen.tsx el flujo de F5.2**: estado `registrandoItem`, import/render de `AltaMovimiento`,
  y la apertura del alta. El botón ya no abre formulario.

### 4. `src/datos/movimientos.ts`
- `confirmarPagoEsperado(item, matches): Promise<Resultado<void>>` → `writeBatch` de updates.
- `desmarcarPago(matches)` → batch update `confirmadoPago:false`.
- (Mantener `itemEsperadoId?` en `NuevoMovimiento`/`crearMovimiento` — sigue siendo válido para alta directa.)

### 5. `firestore.rules` — habilitar update admin
- `movimientos`: cambiar `allow update, delete: if false;` por:
  `allow update: if esAdmin() && request.resource.data.creadoPor == resource.data.creadoPor && request.resource.data.creadoEn == resource.data.creadoEn && request.resource.data.actualizadoEn == request.time;`
  `allow delete: if false;` (el borrado sigue deferido).
- (Dependientes no confirman en v1 — confirmar es admin. Anotar como futuro si se quiere confirmar propios.)

### 6. Contador + sort + CSS
- `cubierto = estado === 'pagado' || estado === 'automatico'`. `por_confirmar` NO cuenta como cubierto.
- Sort: `vencido(0) < pendiente(1) < por_confirmar(2) < parcial(3) < no_registrado(4) < programado(5) < automatico(6) < pagado(7) < no_aplica(8)`.
- CSS: badge `por_confirmar` con color propio (info/azul), distinto de pagado (verde) y pendiente (ámbar).

### 7. `docs/CLAUDE.md`
- State machine: `pagado` ahora requiere confirmación en el mes en curso; meses cerrados con match = pagado (asunción).
- `confirmadoPago` (flag en `Movimiento`); el botón confirma, no crea.
- Revisa F5.2: la acción "Registrar" (creaba movimiento) queda reemplazada por "Confirmar".

## Definición de hecho (en `/resumen`, mes en curso)
- Un esperado con movimiento matcheado figura **Por confirmar** con botón "Confirmar pago".
- Click → pasa a **Pagado**, NO se crea ningún movimiento (verificar que no aparece doble en la lista).
- En Firestore, el movimiento confirmado tiene `confirmadoPago:true` + `itemEsperadoId`.
- Un esperado de un mes pasado con match figura **Pagado** sin acción.
- "Deshacer" sobre un pagado del mes en curso lo devuelve a Por confirmar.
- El contador cuenta pagado + automatico.
- Un dependiente no puede `update` movimientos por SDK directo (debe fallar).
