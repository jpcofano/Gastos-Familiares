# F6.9.7 — Destino-sin-item no debe secuestrar el match por esperado (P2)

## Problema
`matchPorDestino` (`functions/src/index.ts:401`) tiene tres salidas:
- destino **con** `itemEsperadoId`, no pagado ese mes → `{ rama: 2, itemEsperadoId, origenDestino:true }`.
- destino **con** `itemEsperadoId`, ya pagado ese mes → `{ rama: 2, esAdicional:true, … }`.
- destino **sin** item, solo categoría aprendida → `{ rama: 3, origenDestino:true, categoriaPrellena, … }`.

El orquestador (`index.ts:357`) corta con `return` ante **cualquier** retorno no-nulo, **antes** de
`calcularPropuesta` (que es quien corre `matchConEsperados` por texto). Consecuencia: un destino aprendido
**sin** item (rama 3, típico de un movimiento que en su momento cayó libre) **tapa para siempre** el match
por esperado. Aunque después se corrija el `matchTexto.incluye` del esperado (P1), la próxima factura del
mismo payee cae sobre el destino-sin-item, corta en rama 3 y nunca llega a `matchConEsperados`. El eslabón
1 del ciclo virtuoso (primer match por esperado → linkea `itemEsperadoId` → `aprenderDestino` lo upsertea)
no prende nunca.

## Objetivo
Un destino **con** item (rama 2, incluido `esAdicional`) sigue ganando directo. Un destino **sin** item
(rama 3 solo-categoría) **deja de cortar el flujo**: se prueba `matchConEsperados` por texto; si engancha,
gana la rama 2 del esperado (y el movimiento nace con `itemEsperadoId`, que `aprenderDestino` después
upsertea sobre el destino). Si los esperados tampoco enganchan, recién ahí cae a rama 3, **conservando** la
categoría/subcategoría/etiqueta aprendidas del destino para no perder el prefill.

Precedencia final: `destino-con-item (rama 2)` → `esperado por texto (rama 2)` → `rama 3 con prefill de destino`.

Riesgo: medio-bajo. Cambia el **orden de prioridad** del matcher solo para el caso destino-sin-item; los
caminos rama 0/1 (dedup, reconciliación por payee) y destino-con-item quedan intactos. Server-side: requiere
`cd functions && npm run build` + redeploy/emulator de Functions.

## Cambio único — orquestador de `matchComprobante` (`functions/src/index.ts`)

BUSCAR:
```
    // Rama destino: match por CBU/alias/nombre aprendido (prioridad sobre texto)
    const propuestaDestino = await matchPorDestino(datos, movs, mesComp);
    if (propuestaDestino) {
      await ref.update({
        propuestaMatch: { ...propuestaDestino, calculadoEn: FieldValue.serverTimestamp() },
        actualizadoEn:  FieldValue.serverTimestamp(),
      });
      console.log(`[matchComprobante] ${hashActual} → rama destino (item=${propuestaDestino.itemEsperadoId ?? 'sin item'}, adicional=${propuestaDestino.esAdicional ?? false})`);
      return;
    }

    // sin movs: elimina el rama-1 por monto+mes (autovínculo silencioso).
    // Reconciliación de pagos → F6.9 por payee (arriba); dedup → rama 0.
    const propuesta = calcularPropuesta(datos, [], items, mesComp);

    await ref.update({
      propuestaMatch: {
        ...propuesta,
        calculadoEn: FieldValue.serverTimestamp(),
      },
      actualizadoEn: FieldValue.serverTimestamp(),
    });

    console.log(`[matchComprobante] ${hashActual} → rama ${propuesta.rama}`);
```

REEMPLAZAR:
```
    // Rama destino: match por CBU/alias/nombre aprendido (prioridad sobre texto)
    const propuestaDestino = await matchPorDestino(datos, movs, mesComp);

    // F6.9.7 (P2) — un destino CON itemEsperadoId (rama 2, incluido adicional) gana directo.
    // Un destino SIN item (rama 3, solo categoría aprendida) NO corta el flujo: dejamos que
    // matchConEsperados pruebe por texto. Si nada engancha, cae a rama 3 conservando el prefill.
    if (propuestaDestino && propuestaDestino.rama === 2) {
      await ref.update({
        propuestaMatch: { ...propuestaDestino, calculadoEn: FieldValue.serverTimestamp() },
        actualizadoEn:  FieldValue.serverTimestamp(),
      });
      console.log(`[matchComprobante] ${hashActual} → rama destino (item=${propuestaDestino.itemEsperadoId}, adicional=${propuestaDestino.esAdicional ?? false})`);
      return;
    }

    // sin movs: elimina el rama-1 por monto+mes (autovínculo silencioso).
    // Reconciliación de pagos → F6.9 por payee (arriba); dedup → rama 0.
    const propuesta = calcularPropuesta(datos, [], items, mesComp);

    // F6.9.7 (P2) — si los esperados por texto no engancharon (rama 3) pero el destino había
    // aprendido una categoría, conservamos ese prefill en la rama 3 (no perdemos lo aprendido).
    const propuestaFinal =
      propuesta.rama === 3 && propuestaDestino?.rama === 3
        ? {
            ...propuesta,
            origenDestino:        true,
            categoriaPrellena:    propuestaDestino.categoriaPrellena    ?? null,
            subcategoriaPrellena: propuestaDestino.subcategoriaPrellena ?? null,
            etiquetaPrellena:     propuestaDestino.etiquetaPrellena     ?? null,
          }
        : propuesta;

    await ref.update({
      propuestaMatch: {
        ...propuestaFinal,
        calculadoEn: FieldValue.serverTimestamp(),
      },
      actualizadoEn: FieldValue.serverTimestamp(),
    });

    console.log(`[matchComprobante] ${hashActual} → rama ${propuestaFinal.rama}${(propuestaFinal as { origenDestino?: boolean }).origenDestino && propuestaFinal.rama === 3 ? ' (cat. de destino)' : ''}`);
```

> Por qué la condición `propuesta.rama === 3 && propuestaDestino?.rama === 3`: con `movs=[]`,
> `calcularPropuesta` nunca devuelve rama 1 (no hay movimientos que matchear por monto); sale rama 2
> (esperado por texto enganchó → gana, y trae su `itemEsperadoId`) o rama 3 (nada). El prefill del destino
> solo se reinyecta cuando **ambos** fallan, sin pisar la rama 2 del esperado.
