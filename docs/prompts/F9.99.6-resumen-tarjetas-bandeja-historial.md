## Parte 1 — Bandeja: descarte de entrantes desde la UI (reemplaza al script)

### Contexto
Hay un entrante `ruteado` trabado en la bandeja (el huérfano del incidente de
créditos, F9.99.5). Doble hueco confirmado en el código:
- El filtro de bandeja (F9.56, Comprobantes.tsx ~línea 779) mantiene visible un
  ruteado cuyo doc destino no existe o no está confirmado → si el destino
  muere, el ítem queda para siempre.
- `EntranteCard` no tiene descarte para estado `ruteado`/`error`
  (`descartarEntrada` existe pero solo se usa para comprobantes).
- `scripts/limpiarOrfanoF9995.ts` borra el doc destino y el PDF pero NO el
  registro de `entrantes` → correrlo dejaría el ítem trabado permanentemente.

### Decisión del dueño
NO usar el script. La limpieza debe poder hacerse desde la UI, porque el caso
puede repetirse (falta de créditos, errores de extracción, etc.).

### Implementación
1. Nueva Function onCall `descartarEntranteCompleto` (solo dueño/admin):
   - Input: `hash` del entrante.
   - Lee el entrante; si tiene `destino` y el doc destino existe y NO está
     vinculado/confirmado: borra el doc destino y su PDF en Storage
     (`refStoragePdf`), mismo criterio de seguridad que el script F9.99.5
     (JAMÁS borrar destinos vinculados/confirmados — en ese caso abortar con
     error claro).
   - Borra (o marca `descartado`) el registro en `entrantes` — seguir la
     convención que ya use `descartarEntrada`.
   - Idempotente: si el PDF o el destino ya no existen, warn y seguir.
2. UI en `EntranteCard`: botón "Descartar" (admin) visible para estados
   `ruteado` y `error`, con confirmación ("Se borra el archivo y su documento
   destino no confirmado. No afecta datos ya vinculados."). Spinner + error
   local como ya hace `resolver()`.
3. Marcar `scripts/limpiarOrfanoF9995.ts` como deprecado en su header
   (la UI lo reemplaza); no borrarlo.

## Parte 3 — Tarjeta "Falta cubrir (USD)" → "Cobertura del mes"

### Semántica NUEVA (redefinida por el dueño — reemplaza la de F9.55)
- Cálculo: `faltanteArs = gastosArsEq del mes − pesosDisponibles`
  (los mismos dos números que ya muestran las cards: Gastos y Pesos disponibles).
- `faltanteArs ≤ 0` → **"Cubierto"** en verde.
- `faltanteArs > 0` → **"Sin cubrir"** en rojo + el monto faltante convertido a
  USD (`fmtUsdEq(faltanteArs / tc)`), p.ej.: "Sin cubrir · −U$S 1.797".
  El monto es la cantidad de pesos que faltan para cubrir los gastos con los
  pesos disponibles, expresada en USD.
- Eliminar de ESTA tarjeta la lógica de checklist/esperados: `faltaCubrirUsd`
  basado en `montoEsperado`, `nSinMonto`, el estado parcial y el asterisco
  desaparecen de acá. (El checklist y los montos esperados siguen viviendo en
  Gastos Fijos / pendientes — no tocar eso.)
- Actualizar el comentario F9.55 en el código y registrar el cambio de
  semántica en el doc de contrato que corresponda (era esperados-vs-disponibles,
  pasa a gastos-reales-vs-disponibles).

Nota de borde a resolver explícitamente en código: `gastosArsEq` incluye gastos
en USD convertidos. Si el dueño paga gastos USD con dólares (no con pesos), la
tarjeta puede sobreestimar el faltante. Dejar el cálculo como lo definió el
dueño (gastos totales ArsEq) y documentar el supuesto en el comentario.

4. Tarjeta de cobertura: con los datos actuales (gastos 9.063.637, disponibles
   6.363.162) muestra "Sin cubrir" + faltante ≈ U$S equivalente de 2.700.475.
5. Historial con >5 comprobantes: colapsado por defecto, expande y contrae.
6. `tsc --noEmit` limpio en archivos tocados; `npm run build` y build de
   functions OK.
7. Push al cerrar, mostrando la salida de `git push` y
   `git log --oneline origin/main -1`.
