# F9.99.9 — Conciliación manual con agenda unificada (picker de comprobantes)

> Auditado contra `origin/main` = `910074b` (2026-07-17). Implementar DESPUÉS
> de F9.99.8.1 (depende del módulo compartido `src/datos/agenda.ts` que ese
> hotfix extrae). Guardar en `docs/prompts/F9.99.9-picker-agenda-unificada.md`.

## Problema (evidencia de auditoría)

Cuando un comprobante cae como "Movimiento nuevo" (rama 3) o incluso como
"Gasto esperado" (rama 2), el control manual del usuario es limitado:

1. **Gate restrictivo** (`Comprobantes.tsx:~322`): el botón "Conciliar con
   gasto esperado" (F9.82) solo aparece si `esAdmin && esPagoDoc`
   (`tipoDocumento ∈ {transferencia, comprobante_pago}`). Facturas, tickets y
   recibos en rama 3 no tienen asignación manual.
2. **Lista de candidatos plana**: `itemsGasto = items.filter(activo &&
   tipo==='Gasto' && moneda)` — todas las plantillas activas, sin acotar al
   mes, sin orden, sin estado (pagado/pendiente/vencido).
3. **Futuros sueltos invisibles**: los candidatos son solo `ExpectedItem`s.
   Un gasto futuro sin plantilla (los "sueltos" de F9.99.8) no puede elegirse.
4. **`buscarObligacionesAbiertas`** (`src/datos/comprobantes.ts:198`) filtra
   `itemEsperadoId == item && confirmadoPago == false && mes >= mesComp`:
   (a) nunca encuentra sueltos (no tienen `itemEsperadoId` del ítem);
   (b) `mes >= mesComp` excluye vencidos de meses anteriores — inconsistente
   con la ventana de match [mes−1..mes+3] y con la card Hoy.
5. **Candidatos rama 2 tipo 'esperado' nunca se renderizan**:
   `calcularPropuesta` (`functions/src/matchLogica.ts:~236`) los devuelve
   cuando `itemsMatch.length > 1`, pero la UI (`Comprobantes.tsx:186`) filtra
   solo `tipo === 'movimiento'` → se toma `itemsMatch[0]` a ciegas.

## Decisiones del dueño (cerradas, no re-litigar)

- **Alcance temporal de candidatos**: agenda del **mes actual + vencidos no
  cubiertos**. NO incluir sueltos de mes+1..mes+3 en el picker (la ventana
  amplia queda para el match automático de rama 1).
- **Rama 2**: el picker de re-asignación debe estar **disponible pero sin
  cortar el flujo** — el camino feliz (confirmar la propuesta automática)
  sigue siendo el primario y sin fricción; la re-asignación es un botón
  secundario/ghost colapsado que el usuario abre solo si quiere corregir.
- Marcar un suelto como saldado **confirma el movimiento existente, NO crea
  otro** (regla F9.99.8.1 punto 3; riesgo de duplicado F9.99.7).

## Cambios requeridos

### 1. Fuente de candidatos = agenda unificada

El picker consume la agenda construida por el módulo compartido
`src/datos/agenda.ts` (extraído en F9.99.8.1): checklist de esperados del mes
∪ sueltos futuros del mes ∪ vencidos no cubiertos, con dedupe vía
`movimientosDelItem()` (mismo dedupe de F9.99.8 — un suelto que ya matchea
una plantilla no aparece dos veces).

Filtros del picker: `tipo === 'Gasto'` y `moneda === d.moneda` (igual que hoy).

### 2. Dos tipos de candidato, dos caminos

- **Esperado (plantilla)** → flujo actual: elegir obligación abierta
  (`confirmarRama1`) o crear movimiento vinculado
  (`cargarMovimientoDesdeComprobante` con `itemEsperadoId`).
- **Suelto (futuro o vencido sin plantilla)** → confirmar el movimiento
  existente: `pagado = true` + `confirmadoPago = true` por el camino de
  edición, propagando `origenComprobanteId`/`hashPdf` para trazabilidad.
  **Prohibido crear un movimiento nuevo en este camino.**

### 3. Presentación de candidatos

- Orden por día de vencimiento ascendente; vencidos primero con "Venció día X".
- Estado visible por candidato reusando la semántica de la card Hoy:
  check pagado (al fondo, no seleccionable), reloj pendiente, vencido.
- Sueltos con badge "Sin plantilla" (mismo badge de `SueltoAgendaCard`).
- Cada candidato muestra monto y moneda como hoy (`fmtMonto`).

### 4. Ampliar `buscarObligacionesAbiertas`

Cambiar el piso de `mes >= mesComp` a `mes >= mesComp − 1` (coherente con la
ventana de match [mes−1..mes+3]), manteniendo `orderBy('mes')` y el mes
visible en cada opción. Verificar que el índice compuesto de Firestore
existente soporte la query (si requiere índice nuevo, incluir el deploy de
`firestore.indexes.json` en el cierre).

### 5. Ampliar el gate del picker

- **Rama 3**: picker disponible para TODO documento de gasto (no solo
  transferencia/comprobante_pago). Se mantiene `esAdmin`.
- **Rama 2**: agregar botón secundario colapsado "Asignar a otro gasto"
  (ghost, debajo de la propuesta) que abre el mismo picker. No interrumpe ni
  agrega pasos al camino feliz.

### 6. Renderizar candidatos rama 2 tipo 'esperado'

Cuando `pm.candidatos` trae `tipo === 'esperado'` (múltiples matches de
texto), mostrar la elección al usuario (radio como los candidatos de
movimiento de rama 1) en vez de tomar `itemsMatch[0]` en silencio. Con el
picker del punto 5 disponible, alcanza con que la elección use el mismo
componente.

## Fuera de alcance

- Motor de match automático (ramas 0–3, scores, ventana): intacto.
- Dedupe con extracto de tarjeta: sigue diferido.
- `matchLogica.ts` solo se toca si el punto 6 lo requiere para pasar
  descripciones/montos de candidatos esperado (hoy solo manda `id`).

## Criterios de aceptación

1. Comprobante rama 3 (cualquier tipo de doc de gasto): picker visible, lista
   = agenda del mes + vencidos, ordenada por día, con estados y badges.
2. Elegir un suelto → el movimiento existente queda `pagado` +
   `confirmadoPago` (verificar en Firestore que NO hay doc nuevo) y el
   comprobante pasa a `vinculado`.
3. Elegir un esperado → flujo actual intacto (obligación abierta o alta
   vinculada).
4. Vencido del mes anterior de una plantilla aparece y es saldable.
5. Rama 2: propuesta automática confirmable igual que hoy (cero fricción
   agregada); botón "Asignar a otro gasto" colapsado funciona.
6. Rama 2 con múltiples esperados: el usuario elige, no se asigna a ciegas.
7. `tsc --noEmit` sin errores nuevos (base 41) + `npm run build` manual.
8. **Cierre obligatorio**: salida de `git push` +
   `git log --oneline origin/main -1`.
