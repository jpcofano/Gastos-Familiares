# F9.24 — Consistencia persona → memberId (+ backfill del seed)

> Reglas: doc-first · una PR por fase. **Toca datos**: incluye backfill de docs del seed.
> Separar en sub-pasos si conviene (migración primero, lectura después).

## Problema
El campo `movimientos.persona` es **inconsistente**:
- El flujo de tarjetas ya resuelve persona a **memberId** (`resolverNombreMiembro` en
  `confirmarResumenTarjeta`).
- Pero la **semilla/legacy** guarda `persona` por **nombre** ("Juan", "María",
  "Federico", "Sofía").
- `movimientosDelMes(mes, persona)` filtra por ese campo mixto → el filtro por persona
  puede no matchear (tildes, mayúsculas, variantes).

## Tarea
1. **Canonizar a memberId.** Definir que `movimientos.persona` guarda **siempre el
   memberId** del doc de `usuarios`. (Mantener, si se quiere, un `personaNombre`
   denormalizado para mostrar, pero el campo de filtro/relación es el id.)
2. **Backfill** (script de migración, una sola corrida):
   - Recorrer `movimientos` con `persona` por nombre.
   - Resolver nombre → memberId vía `usuarios` (reusar `resolverNombreMiembro`/tabla de
     la familia). Mapa esperado: Juan, María (admin) · Federico, Sofía (dependientes).
   - Escribir `persona = memberId`. Registrar los que no resuelven (loguear, no romper).
3. **Lecturas**: `movimientosDelMes` y cualquier query/filtro por persona pasan a usar
   memberId. La UI que muestra el nombre resuelve memberId → nombre (no al revés).
4. **Escrituras nuevas**: confirmar que `crearMovimiento` y el flujo de tarjetas ya
   escriben memberId (el de tarjetas sí; revisar alta manual).

## Aceptación
- Todos los `movimientos` tienen `persona` = memberId (o quedan logueados los no
  resueltos para revisión manual).
- Filtrar Resumen/Dashboard por persona funciona sin fallar por tildes/typos.
- Alta manual y confirmación de tarjeta escriben memberId.
- El backfill es idempotente (re-correrlo no rompe ni duplica).

## Notas
- Es el paso que destraba el filtro por persona de forma confiable.
- Hacerlo **antes** de la migración a Firestore prod (Etapa B) para que el import ya
  llegue canonizado. Ver `docs/plan-maestro-firestore-pwa.md`.
