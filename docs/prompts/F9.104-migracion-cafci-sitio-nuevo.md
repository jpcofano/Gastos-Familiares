# F9.104 — Migración de la ingesta CAFCI al sitio nuevo

## Contexto

Los 403 de `sincronizarCafci` **no eran un bloqueo de CloudFront contra las
Cloud Functions**. CAFCI migró su sitio y dio de baja la API vieja. Las
hipótesis de F9.102.1/.2 (reputación de IP/ASN, fingerprint TLS de Node,
región de la CF) quedan todas descartadas por evidencia directa.

### Evidencia recolectada (21/07/2026)

1. `https://api.pub.cafci.org.ar/fondo/344/clase/569/ficha` → **403** desde
   navegador residencial (Chrome, barra de direcciones y `fetch` desde el
   origen `www.cafci.org.ar`), no sólo desde la CF.
2. El **propio sitio de CAFCI** se pega 403 contra su propia API. En la
   consola de `ficha-fondo.html?q=344;569`, el bootstrap de Angular falla:
   `GET https://api.pub.cafci.org.ar/fondo/tipo-clase → 403`, disparado desde
   `angular.js` en `doBootstrap`, antes siquiera de pedir la ficha. Después
   `fichaFondo.js:403` imprime `----` porque se quedó sin datos.
3. Los errores de CORS que aparecen en consola son **ruido**: CloudFront no
   adjunta `Access-Control-Allow-Origin` en sus respuestas 403, y el browser
   reporta CORS antes que el 403 subyacente. Le pasa igual al sitio oficial.
4. `https://api.pub.cafci.org.ar/pb_get` (planilla diaria XLSX) **sigue
   respondiendo 200**, incluso desde datacenter. Es otro behavior de la misma
   distribución y sobrevivió a la baja.

### El sitio nuevo

Host nuevo: `estadisticas.cafci.org.ar`. Stack nuevo (Rails
server-rendered, `authenticity_token` y CSRF en los metas, assets con
fingerprint, imágenes viejas bajo `/assets/legacy/`). El Angular de
`ficha-fondo.html` sigue publicado pero apunta al backend muerto.

**El esquema de IDs se conserva.** La ficha nueva es:

```
https://estadisticas.cafci.org.ar/fondos/{fondoId}?clase={claseId}
```

Verificado con tres casos, incluido el borde:

| URL | Resultado |
|---|---|
| `/fondos/344?clase=569` | Alpha Pesos — mismo fondo que la API vieja |
| `/fondos/798?clase=1982` | Mercado Fondo Clase A, datos al 20/07/2026 |
| `/fondos/15?clase=15` | Pionero Acciones Plus Clase B (`fondoId === claseId` no rompe) |

Responde 200 desde datacenter, sin bot detection. Una Cloud Function le puede
pegar de frente.

### Formato del dato: la cartera está en un atributo del canvas

Ya no hay JSON de API. La composición viene embebida en el HTML:

```
div#cartera .divContent > div#pie-chart > canvas[data-controller="pie-chart"]
  data-pie-chart-items-value = '[{"nombre":"...","porcentaje":21.3}, ...]'
```

Cambian los nombres de campo: **`nombre` / `porcentaje`**, contra
`nombreActivo` / `share` de la API vieja.

### El truncamiento — el punto crítico de esta spec

El pie chart es un componente de UI: muestra los primeros N activos y agrupa
la cola en un item literal **`"Resto de Activos"`**. Medido en producción:

| Fondo | Items | Suma | `Resto de Activos` |
|---|---:|---:|---:|
| Alpha Pesos (money market) | 12 | 99,8 | **10,1 %** |
| Galileo Acciones (615/2249) | 14 | 99,9 | **1,6 %** |

N es dinámico, no un top-11 fijo. La cola gorda del money market es un
artefacto del tipo de fondo (treinta plazos fijos parecidos); los fondos de
acciones AR están concentrados y el residuo es chico.

**El modo de falla a evitar**: la validación de integridad actual
(`functions/src/index.ts:3253-3254`) chequea que `totalPct` caiga en
`[98,102]`. Ambos casos de arriba pasan ese chequeo — incluido el que tiene
**10 % de cartera ciega**. Un dato truncado que se hace pasar por completo es
peor que un dato ausente: entra al benchmark marcado como íntegro y sesga
justamente contra las posiciones chicas, que son las que interesan para medir
concentración.

Por eso `Resto de Activos` se persiste como **bucket explícito de
desconocido**: nunca prorrateado entre las demás posiciones, nunca descartado
en silencio, y la cobertura identificada viaja hasta el reporte.

Nota menor: los porcentajes vienen redondeados a un decimal. Con 13 fondos
ponderados es ruido tolerable; queda anotado.

**Auditar antes de tocar**: verificar líneas y firmas citadas antes de
implementar. F9.102/F9.102.1/F9.102.2 movieron renglones y las referencias
pueden estar corridas. Si algo difiere, reportarlo primero.

## Alcance

### 1. Fetch: host y formato nuevos

En `sincronizarCafci` (`functions/src/index.ts:3288-3297`), reemplazar la URL
y el manejo de respuesta:

```ts
const url = `https://estadisticas.cafci.org.ar/fondos/${fondo.fondoId}?clase=${fondo.claseId}`;
const res = await fetch(url, {
  headers: {
    'User-Agent': 'Mozilla/5.0',
    'Accept': 'text/html,application/xhtml+xml',
  },
});
```

- Sacar `Origin` y `Referer`: eran spoofeo para la API vieja y ya no aplican.
- La respuesta es **HTML**, no JSON. `await res.text()`.

### 2. Logueo real de errores (deuda de esta sesión)

El bloque actual (`functions/src/index.ts:3298-3305`) mapea cualquier 403 a un
mensaje fijo que **afirma** "CloudFront bloqueó la request" y **descarta el
body**. Esa suposición escrita en el código costó tres frentes de
investigación: el body real habría mostrado desde el primer día que era un
error de aplicación, no de WAF.

Reemplazar por logueo de lo observado, sin interpretarlo:

```ts
if (!res.ok) {
  const body = (await res.text()).slice(0, 300);
  console.error(`[sincronizarCafci] ${fondo.nombre}: HTTP ${res.status} — ${body}`);
  throw new Error(`HTTP ${res.status}: ${body.slice(0, 150)}`);
}
```

Es el mismo criterio que ya usaba el Apps Script legacy
(`10_Fondos_CAFCl.js:213`). Aplicar el mismo tratamiento en
`importarCafciManual`.

### 3. Extracción del atributo

Nuevo helper puro y exportado (testeable sin red):

```ts
export function extraerItemsCartera(html: string): Array<{ nombre: string; porcentaje: number }>
```

- Localizar `data-pie-chart-items-value` por regex sobre el HTML. **No agregar
  cheerio ni jsdom**: es un solo atributo y `functions/package.json` no tiene
  parser de HTML; sumar dependencia por esto no se justifica.
- **Decodificar entidades HTML antes de `JSON.parse`.** El atributo viene
  escapado (`&quot;` y potencialmente `&amp;`, `&#39;`, `&lt;`, `&gt;`).
  Omitir este paso hace fallar el parseo — es el error más probable de esta
  implementación.
- Si el atributo no está, o el JSON no parsea, o el array queda vacío:
  lanzar error con mensaje explícito. **No devolver array vacío en silencio.**

### 4. `fechaDatos`

Es la clave del `docId` (`cafciCarteras/{fondoId}_{fechaDatos}`), así que no
puede quedar librada a un default.

La ficha muestra la fecha de cartera como `Valores al 26/06/2026`, **distinta
de** la fecha de valores del fondo (`información al 20/07/2026`). Tomar la de
**cartera**, no la del encabezado.

No tengo el selector exacto — el extractor de HTML no me devolvió ese nodo.
**Auditar el HTML real** y reportar el selector antes de fijarlo. Fallback
aceptable: regex por `/Valores al (\d{2}\/\d{2}\/\d{4})/` acotado al bloque
`div#cartera`, normalizando a `YYYY-MM-DD`.

Si no se puede extraer, **fallar el fondo con error explícito**. Caer a
`fechaFetch` generaría un `docId` con fecha equivocada y ensuciaría la serie
histórica de forma difícil de detectar después.

### 5. Bucket de desconocido y cobertura identificada

En `parsearFichaCafci` (`functions/src/index.ts:3180`):

- Detectar el item de cola por `/^resto de activos$/i` sobre `nombre`.
- Persistirlo como posición con `categoria: 'RESTO'`, `ticker: null`, y **no**
  contarlo como pendiente de mapeo.
- Calcular `coberturaIdentificada = totalPct − pesoResto`.
- **Reemplazar** `advertenciaIntegridad` (hoy `totalPct < 98 || totalPct > 102`)
  por dos chequeos independientes:
  - `advertenciaSuma`: `totalPct` fuera de `[98,102]` — detecta HTML roto o
    parseo incompleto.
  - `advertenciaCobertura`: `coberturaIdentificada < 95` — detecta
    truncamiento severo.
- Guardar `coberturaIdentificada` y `pesoResto` en el doc de `cafciCarteras`
  (`functions/src/index.ts:3315-3325`), y propagarlos al tipo de retorno
  `ParseoFichaCafci`.

El umbral 95 es una elección de diseño, no una constante mágica: define
qué cartera es suficientemente conocida para entrar al benchmark. Dejarlo como
constante nombrada y comentada.

### 6. `parsePosicionCafci`

`functions/src/index.ts:3135-3148` ya acepta `porcentaje` como fallback de
peso. **Sólo falta agregar `nombre`** a la lista de alias de especie
(línea 3141), delante de `nombreActivo`. El resto del helper no se toca.

Conservar la retrocompatibilidad con `nombreActivo`/`share`: hay docs ya
persistidos con el formato viejo y `importarCafciManual` puede seguir
recibiendo JSON legacy.

### 7. Revertir el experimento de región

Cerrado por evidencia: la región nunca fue la causa.

- `functions/src/index.ts:3259-3266`: eliminar `region: 'us-central1'` y el
  bloque de comentario que documenta la hipótesis. La función vuelve a
  `southamerica-east1` vía `setGlobalOptions` (línea 24).
- `src/firebase.ts:32-36`: eliminar `functionsUsCentral` y su comentario.
- `src/firebase.ts:54`: eliminar el `connectFunctionsEmulator` asociado.
- Actualizar los llamados en `src/vistas/Patrimonio.tsx` que usen esa
  instancia, para que vuelvan a `functions`.
- Después del deploy, verificar con `firebase functions:list` que
  `sincronizarCafci` queda **una sola vez, en `southamerica-east1`**.

### 8. `PegarJsonModal`

`src/vistas/Patrimonio.tsx:2282` muestra la URL de la API muerta. El modal
sigue teniendo sentido como fallback, pero apuntando al host correcto.

Como el formato dejó de ser JSON, el pegado manual ya no puede ser "copiá el
JSON de la ficha". Cambiar a: mostrar la URL nueva
(`https://estadisticas.cafci.org.ar/fondos/{fondoId}?clase={claseId}`) e
instruir a pegar el **snippet de consola** que extrae los items:

```js
copy(document.querySelector('[data-pie-chart-items-value]').dataset.pieChartItemsValue)
```

`importarCafciManual` sigue recibiendo un string JSON y compartiendo
`parsearFichaCafci`; lo que cambia es la forma de obtenerlo. Ajustar el texto
del modal y la validación de entrada para aceptar el array de items directo
(sin el envoltorio `{success, data:{info:{semanal}}}` de la API vieja),
manteniendo compatibilidad con el formato viejo.

Si el fetch automático queda estable tras el deploy, evaluar en una spec
posterior si el modal sigue justificándose.

## Criterios de aceptación

1. `sincronizarCafci` sobre los 13 fondos configurados: reportar cuántos
   sincronizaron y, para los que fallen, el status HTTP y el body real —
   no un mensaje interpretado.
2. Para Galileo Acciones (615/2249), el doc en `cafciCarteras` tiene:
   14 posiciones, `totalPct ≈ 99,9`, `pesoResto ≈ 1,6`,
   `coberturaIdentificada ≈ 98,3`, y una posición con `categoria: 'RESTO'`.
3. `fechaDatos` del doc coincide con la fecha de **cartera** que muestra la
   ficha, no con la de valores del fondo.
4. Un fondo money market (p. ej. 344/569) dispara `advertenciaCobertura` y
   **no** dispara `advertenciaSuma`. Es el caso que hoy pasaría silencioso y
   la razón de ser del cambio de criterio.
5. `firebase functions:list` muestra `sincronizarCafci` una sola vez, en
   `southamerica-east1`.
6. `npx tsc --noEmit` sin errores nuevos sobre el baseline. **Fijar el
   baseline al empezar y anotarlo en el commit** — se movió de 48 a 41 entre
   sesiones sin registro, y mientras no esté fijado el control "0 errores
   nuevos" no significa nada.
7. Tests de `extraerItemsCartera` con: HTML con entidades escapadas, atributo
   ausente, JSON inválido, array vacío. Los cuatro casos deben fallar o pasar
   de forma explícita.
