# F9.93 — Patrimonio: análisis IA por posición + sectorial (bajo demanda)

## Contexto

Motor de **juicio** del sistema (el determinístico ya está en F9.90/91).
Una sola Cloud Function con dos modos cubre el análisis por posición y el
panorama sectorial. Decisiones del dueño: **bajo demanda con caché**; lote
completo solo como **acción manual explícita** (control de costos); **toggle**
global activar/desactivar.

Filosofía innegociable: describe, contextualiza y muestra riesgos.
**NUNCA recomienda comprar/vender/mantener.** Sin campos de acción sugerida.

## 1) Cloud Function `analizarConIA`

- Callable (auth: solo dueño, verificar en la Function además de rules).
- SDK de Anthropic ya presente en `functions/`. Modelo: `claude-sonnet-4-6`
  con la herramienta `web_search` habilitada (necesita info actual).
- Input: `{ modo: 'posicion' | 'sectorial', ticker?, contexto }` donde
  `contexto` = datos determinísticos que la app ya tiene (para modo posicion:
  ticker, nombre, tipo, sector, país, % del total, cuentas donde está, driver,
  top-N en que cae; para sectorial: composición agregada por sector/driver).
- **Toggle:** doc `configPatrimonio/ia` → `{ habilitado: bool }`, editable
  con un switch en la app (card de configuración, solo dueño). Si está off,
  la Function rechaza con mensaje claro y la UI muestra el switch.
- Guardar SIEMPRE el resultado (caché):
  - modo posicion → `analisisPosiciones/{ticker}`:
    `{ ticker, generadoEn, modeloUsado, resultado }` (sobrescribe el previo).
  - modo sectorial → `analisisSectorial/{autoId}` con `generadoEn`.

## 2) Salida estructurada (prompt interno de la Function)

Pedir al modelo **JSON estricto** con este shape (modo posicion):

```
{
  "queEs": "1-2 frases: qué es el instrumento/empresa y de qué depende su valor",
  "situacionActual": "3-5 frases con lo relevante HOY (resultados, regulación,
                      noticias, precio vs historia) — usar web_search",
  "riesgos": ["3 a 5 riesgos específicos de ESTE papel, concretos"],
  "rolEnCartera": "1-3 frases usando el contexto provisto: peso, con qué otras
                   posiciones comparte driver, qué le aporta o concentra",
  "proximosEventos": ["cupones, vencimientos, resultados, revisiones
                       tarifarias con fecha aproximada si se conoce"],
  "senalesAVigilar": ["2-4 señales observables que indicarían que la tesis
                       del papel cambió"],
  "fuentes": ["urls o medios consultados"]
}
```

Modo sectorial: mismo espíritu con bloques por sector relevante de la cartera
(energía AR, macro/CER AR, soberano AR, cripto, tech global) — panorama,
riesgos y eventos por bloque, con fuentes.

Instrucciones fijas del prompt interno: español rioplatense; prohibido
recomendar acciones ("comprar", "vender", "conviene"); si no hay información
confiable de algo, decirlo en vez de inventar; máx ~250 palabras por posición.

## 3) UI

- **Detalle de posición (Tenencias → tap):** arriba lo determinístico ya
  existente; abajo sección "Análisis" →
  - sin caché: botón "Generar análisis" (deshabilitado + hint si toggle off);
  - con caché: render de las secciones + "generado hace X días" + botón
    "Regenerar". Nunca se regenera solo.
- **Agregación por ticker:** un análisis por ticker (TRAN una vez aunque esté
  en 2 cuentas; GLOB una vez agregando CEDEAR + manual). El contexto que se
  envía ya viene agregado.
- **Lote manual:** en la card de configuración, botón "Analizar toda la
  cartera" → modal de confirmación: "~N tickers, puede tardar varios minutos
  y consume API" → corre secuencial con barra de progreso; errores por ticker
  no frenan el lote (se listan al final).
- **Sectorial:** botón "Generar panorama sectorial" en la solapa Research
  (crearla si no existe; si el design system ya la trae, usarla). Muestra el
  último con su fecha.

## 4) Costos y límites

- Sin llamadas automáticas en ningún flujo (ni al confirmar corridas, ni al
  abrir vistas, ni al generar PDF — el PDF solo lee caché).
- `max_tokens` acotado (~1500 posicion / ~3000 sectorial).
- Registrar en cada doc `modeloUsado` y `generadoEn` para trazabilidad.

## Criterios de aceptación

1. Toggle off → ningún camino llega a la API; UI lo explica.
2. Generar TRAN produce JSON válido renderizado en secciones, cacheado; al
   reabrir la posición se ve el caché con antigüedad, sin nueva llamada.
3. El resultado no contiene recomendaciones de acción (verificar wording).
4. El lote pide confirmación, muestra progreso y tolera fallos parciales.
5. El PDF (F9.92) incluye los análisis cacheados con su fecha.
6. Nada del lado gastos se toca; reglas solo-dueño en las colecciones nuevas.
