# F9.51 — Etapa C: montar ShareLanding en la app real (`/comprobantes?share=1`)

> Reglas: **NO modela datos nuevos, NO toca `sw.js` ni el manifest** (eso fue F9.49). Solo
> monta una **pantalla de recepción** sobre el flujo de share-target que YA existe, encadenada
> al confirm real. Diseño aprobado: ver `ui_kits/mobile/ShareLanding.jsx` (referencia visual y
> de copy — **no portear los timers ni la mock data**, ver §3).

## Diagnóstico (auditado contra el repo)
- `public/sw.js` recibe el `POST /share-target`, guarda el File en IndexedDB y redirige a
  **`/comprobantes?share=1`** (F9.49).
- `src/datos/shareTargetIdb.ts` expone `leerYBorrarArchivoCompartido()` (lee-y-borra el File).
- Hoy, cuando Comprobantes detecta `?share=1`, salta directo al picker/confirm con el archivo
  ya cargado: **el arranque en frío queda en blanco** mientras se lee IDB + se clasifica +
  se extrae (la extracción Anthropic puede tardar segundos). Falta la pantalla que cubre eso.

## Tarea — montar `ShareLanding` real sobre el arranque de Comprobantes

### 1. Disparador
En la vista **Comprobantes**, cuando `new URLSearchParams(location.search).get('share') === '1'`:
montar `ShareLanding` a pantalla completa (overlay, `z-index` alto) **antes** de resolver el
archivo, y desmontarlo al encadenar al confirm real (`onReady`) o al cancelar (`onClose`).
Limpiar el `?share=1` de la URL (`history.replaceState`) al cerrar, para que un refresh no
re-dispare.

### 2. Cableado de fases a la async REAL (esto es el corazón del prompt)
La pantalla del kit corre 5 fases con `setTimeout` y mock data. En la app, **cada fase la
dispara el paso async real** — nada de timers fijos:

| Fase kit | Evento real que la activa |
|---|---|
| 0 `recibido`     | `leerYBorrarArchivoCompartido()` devolvió un File (nombre + peso reales). |
| 1 `leyendo`      | File leído / subido a Storage (el paso que ya hace el flujo de carga). |
| 2 `clasificado`  | La extracción ya determinó **tipo** (factura/comprobante vs resumen de tarjeta) → badge. |
| 3 `extrayendo`   | Llamada a la function de extracción Anthropic en curso. |
| 4 `listo`        | Volvió el resultado: monto + campos + **match** (gasto esperado vs movimiento nuevo). |
| → `onReady()`    | Encadenar al confirm real **ya existente** (NO crear un confirm nuevo). |

- Si no hay File en IDB (usuario refrescó, o entró a `?share=1` a mano): cerrar el landing en
  silencio y mostrar Comprobantes normal. No romper.
- Si la extracción **falla** (timeout/error de la function): el landing cae a un estado de
  error con botón "Cargar manual" que abre el confirm con lo que se haya extraído (o vacío).
  No dejar el spinner colgado para siempre.

### 3. Qué portear del kit y qué NO
- **Portear:** todo el layout, copy, badges, la rama factura (campos + badge de destino) y la
  rama resumen (split este-mes/deuda-futura de F9.21), la barra de progreso, los estados de
  texto del pie. El componente ya está en `ui_kits/mobile/ShareLanding.jsx` (TSX-friendly).
- **Reemplazar:** `SL_FACTURA`/`SL_RESUMEN` (mock) → datos reales del File + resultado de la
  extracción. El `useEffect` de timers (`durs = [...]`) → la fase la setea cada paso async.
  El count-up del monto se mantiene, pero arranca cuando llega el monto real (fase 4).
- El **tipo** (`factura`|`resumen`) ya no es prop forzada: sale de la clasificación real.

### 4. Datos reales que alimentan el landing
- **Factura:** `monto`, `comercio`, `fecha/vence` desde la extracción; **badge de destino** =
  resultado del match que ya corre (`Gasto esperado · <nombre> · <período>` si matchea contra
  un esperado; `Movimiento nuevo` si no). No re-implementar el match: leer el que ya existe.
- **Resumen de tarjeta:** `total`, `consumos`, `enCuotas`, y el split `esteMes`/`deudaFutura`
  derivado por la lógica F9.21 (este-mes vs deuda-futura). Si el parsing de líneas no está
  disponible aún, mostrar total + N consumos y omitir el split (no inventar números).

## Aceptación
- Compartir un PDF a la PWA instalada (Android) abre Comprobantes con el **landing cubriendo
  el arranque**; las fases avanzan **siguiendo la async real** (no en tiempos fijos), y al
  terminar encadena al **confirm existente** con los datos precargados.
- Factura → muestra monto + comercio + vence + badge correcto (esperado/nuevo). Resumen →
  muestra total + consumos + split este-mes/deuda-futura.
- Cancelar cierra el landing, limpia `?share=1` y deja Comprobantes utilizable.
- Refrescar en `?share=1` sin File en IDB **no** rompe (cae a Comprobantes normal).
- Error de extracción → estado de error con salida a carga manual, sin spinner colgado.
- iOS: como no hay Web Share Target (F9.49), el landing no se dispara por share; el file
  picker in-app de Comprobantes sigue siendo el fallback. Sin cambios ahí.
