# F8.2 — Editor de Destinos (payees aprendidos → ítem esperado)

## Objetivo

Activar el ítem **Destinos** del grupo "Clasificación y aprendizaje · admin" (creado
en F8.1) con un editor admin sobre la colección `destinos`: los payees aprendidos por
`aprenderDestino` que vinculan un CBU/CUIT/alias/nombre con un **ítem esperado** y/o una
categoría.

`destinos` **no tiene `match` en `firestore.rules`** → el cliente hoy no lo lee ni escribe.
Esta fase:

1. Abre **lectura admin** de `destinos` en rules (write sigue cerrado).
2. Agrega callables **`upsertDestino`** y **`eliminarDestino`** (mutación server-side,
   normalización + validación del FK `itemEsperadoId`).
3. Crea el wrapper de cliente y el editor `Destinos.tsx`, y **enciende** el ítem del menú.

Deploy: toca `functions/` + rules + hosting → build de functions **obligatorio** antes.

## Contexto verificado (origin HEAD `df090fd`)

**Shape de `destinos/{id}`** (lo escribe `aprenderDestino` en `functions/src/index.ts`):
```
destinoNorm    : string        // clave normalizada
tipo           : 'cbu'|'cuit'|'alias'|'nombre'
itemEsperadoId : string        // FK a itemsEsperados — OPCIONAL (ausente si solo hay categoría)
categoria      : string|null
subcategoria   : string|null
etiqueta       : string|null
confianza      : number        // arranca 0.8; +0.1 por corrección; cap 1.0
creadoPor      : string
actualizadoEn  : serverTimestamp
```
- **id del doc** = `idDestinoNorm(norm)` = `sha256(norm).slice(0,24)` (helper en `index.ts`).
- **Normalización**: `normalizarDestino(raw)` vive en `functions/src/matchLogica.ts`
  (exportada). Devuelve `{ tipo, norm } | null`. CBU=22 díg, CUIT=11 díg, alias=6–20
  `[a-z0-9._-]`, nombre=NFD+lower+colapsa espacios, mín 3 chars. **Solo server** — el
  cliente NO normaliza (manda raw; server resuelve). Mantiene "server autoritativo".
- **Umbral del matcher**: `matchPorDestino` ignora destinos con `confianza < 0.7`. No hay
  campo `activo` → "desactivar" = bajar confianza < 0.7 (o borrar). **NO inventar `activo`**
  (el matcher no lo lee → sería no-op silencioso).
- **FK crítico**: un destino con `itemEsperadoId` colgado (item inexistente) rompe el ciclo
  de aprendizaje. La callable **debe validar** que el item exista antes de escribir el link.
- Patrón de callable a espejar: `resolverEntranteAmbiguo` / `guardarTaxonomia` (misma
  cabecera de auth: `request.auth` → `email` → `autorizados/{email}.rol === 'admin'`).
- Patrón de wrapper de cliente: `src/datos/configFamilia.ts` (`httpsCallable(functions, ...)`).
- Fuente de ítems esperados en el cliente: `useItemsEsperados()` (real-time).

### 2a. `upsertDestino`

Acepta **crear** (con `destinoRaw`) o **editar** (con `id` existente).

```ts
export const upsertDestino = onCall(
  { region: 'southamerica-east1' },
  async (request) => {
    if (!request.auth) throw new HttpsError('unauthenticated', 'No autenticado');
    const email = request.auth.token.email?.toLowerCase();
    if (!email) throw new HttpsError('unauthenticated', 'Email no disponible');
    const autSnap = await db.collection('autorizados').doc(email).get();
    if (!autSnap.exists || autSnap.data()?.rol !== 'admin') {
      throw new HttpsError('permission-denied', 'Se requiere rol admin');
    }

    const {
      id, destinoRaw,
      itemEsperadoId, categoria, subcategoria, etiqueta, confianza,
    } = (request.data ?? {}) as {
      id?: string; destinoRaw?: string;
      itemEsperadoId?: string | null;
      categoria?: string | null; subcategoria?: string | null; etiqueta?: string | null;
      confianza?: number;
    };

    const cat  = (categoria ?? null) || null;
    const item = (itemEsperadoId ?? null) || null;

    // Debe quedar util: al menos categoría o item (mismo criterio que aprenderDestino)
    if (!cat && !item) throw new HttpsError('invalid-argument', 'Se requiere categoría o ítem esperado');

    if (confianza != null && (typeof confianza !== 'number' || confianza < 0 || confianza > 1)) {
      throw new HttpsError('invalid-argument', 'confianza fuera de rango [0,1]');
    }

    // Guard del FK: si hay item, debe existir en itemsEsperados
    if (item) {
      const itemSnap = await db.collection('itemsEsperados').doc(item).get();
      if (!itemSnap.exists) throw new HttpsError('invalid-argument', `itemEsperadoId inexistente: ${item}`);
    }

    let ref: FirebaseFirestore.DocumentReference;
    let base: Record<string, unknown> = {};

    if (id) {
      // EDITAR
      ref = db.collection('destinos').doc(id);
      const snap = await ref.get();
      if (!snap.exists) throw new HttpsError('not-found', 'Destino no encontrado');
    } else {
      // CREAR — normalización server-side
      if (!destinoRaw || typeof destinoRaw !== 'string') {
        throw new HttpsError('invalid-argument', 'destinoRaw requerido para crear');
      }
      const parsed = normalizarDestino(destinoRaw);
      if (!parsed) throw new HttpsError('invalid-argument', 'destinoRaw no normalizable');
      const nid = idDestinoNorm(parsed.norm);
      ref = db.collection('destinos').doc(nid);
      base = {
        destinoNorm: parsed.norm,
        tipo:        parsed.tipo,
        creadoPor:   email,
      };
    }

    const update: Record<string, unknown> = {
      ...base,
      categoria:     cat,
      subcategoria:  (subcategoria ?? null) || null,
      etiqueta:      (etiqueta ?? null) || null,
      actualizadoEn: FieldValue.serverTimestamp(),
    };
    // itemEsperadoId: set explícito (permite vincular o desvincular con null)
    update.itemEsperadoId = item;
    if (confianza != null) update.confianza = confianza;
    else if (!id) update.confianza = 0.8; // default en creación

    await ref.set(update, { merge: true });
    console.log(`[upsertDestino] ${ref.id} → cat=${cat ?? '-'} item=${item ?? '-'} (por ${email})`);
    return { ok: true, id: ref.id };
  },
);
```

### 2b. `eliminarDestino`

```ts
export const eliminarDestino = onCall(
  { region: 'southamerica-east1' },
  async (request) => {
    if (!request.auth) throw new HttpsError('unauthenticated', 'No autenticado');
    const email = request.auth.token.email?.toLowerCase();
    if (!email) throw new HttpsError('unauthenticated', 'Email no disponible');
    const autSnap = await db.collection('autorizados').doc(email).get();
    if (!autSnap.exists || autSnap.data()?.rol !== 'admin') {
      throw new HttpsError('permission-denied', 'Se requiere rol admin');
    }
    const { id } = (request.data ?? {}) as { id?: string };
    if (!id) throw new HttpsError('invalid-argument', 'id requerido');
    await db.collection('destinos').doc(id).delete();
    console.log(`[eliminarDestino] ${id} (por ${email})`);
    return { ok: true };
  },
);
```

> Verificar que `FieldValue`, `idDestinoNorm` y `normalizarDestino` estén en scope (ya lo
> están: `FieldValue` se usa en todo el archivo, `idDestinoNorm` es local, `normalizarDestino`
> viene del import de `./matchLogica`). No reimportar.

## Cambio 3 — Wrapper de cliente `src/datos/destinos.ts` (NUEVO)

```ts
import { collection, getDocs } from 'firebase/firestore';
import { httpsCallable } from 'firebase/functions';
import { db, functions } from '../firebase';

export interface DestinoDoc {
  id: string;
  destinoNorm: string;
  tipo: 'cbu' | 'cuit' | 'alias' | 'nombre';
  itemEsperadoId: string | null;
  categoria: string | null;
  subcategoria: string | null;
  etiqueta: string | null;
  confianza: number;
}

export async function listarDestinos(): Promise<DestinoDoc[]> {
  const snap = await getDocs(collection(db, 'destinos'));
  return snap.docs.map(d => {
    const x = d.data();
    return {
      id: d.id,
      destinoNorm: x.destinoNorm ?? d.id,
      tipo: x.tipo ?? 'nombre',
      itemEsperadoId: x.itemEsperadoId ?? null,
      categoria: x.categoria ?? null,
      subcategoria: x.subcategoria ?? null,
      etiqueta: x.etiqueta ?? null,
      confianza: typeof x.confianza === 'number' ? x.confianza : 0,
    } satisfies DestinoDoc;
  });
}

export interface UpsertDestinoInput {
  id?: string;             // editar
  destinoRaw?: string;     // crear (server normaliza)
  itemEsperadoId?: string | null;
  categoria?: string | null;
  subcategoria?: string | null;
  etiqueta?: string | null;
  confianza?: number;
}

export async function upsertDestino(input: UpsertDestinoInput): Promise<{ ok: boolean; id: string }> {
  const fn = httpsCallable(functions, 'upsertDestino');
  const res = await fn(input);
  return res.data as { ok: boolean; id: string };
}

export async function eliminarDestino(id: string): Promise<void> {
  const fn = httpsCallable(functions, 'eliminarDestino');
  await fn({ id });
}
```

Espejar la estética de `Diccionario.tsx` (F8.1) / `Categorias.tsx`. Spec:

**Lectura**

**Listado (cada fila)**
- `destinoNorm` (fuerte) + chip `tipo` (cbu/cuit/alias/nombre).
- Línea secundaria: si `itemEsperadoId` → `🔗 {nombre del item}`; si no → `categoria › subcategoria`.
- Chip `confianza` en %. Si `confianza < 0.7`: chip en tono warning + tooltip/helper
  **"Bajo umbral — el matcher lo ignora"**.
- Acciones: `pencil` (editar), `trash-2` (borrar con confirm).
- Buscador arriba: filtra por `destinoNorm` y por nombre de item/categoría.

**Alta (bottom-sheet "Nuevo destino")**
- `destinoRaw` (text, requerido) — helper: "CBU (22 díg), CUIT (11), alias o nombre.
  Se normaliza automáticamente."
- `itemEsperadoId` (select de `items`, opción "— sin vínculo —" → null).
- `categoria` (select, opcional → null), `subcategoria`, `etiqueta` (opcionales).

**Edición (bottom-sheet)**
- NO editar `destinoRaw`/`destinoNorm`/`tipo` (la clave es inmutable — cambiarla = borrar y
  crear otro). Mostrarlos read-only.
- Editables: `itemEsperadoId`, `categoria`, `subcategoria`, `etiqueta`, `confianza`
  (número 0–1 o slider; label del umbral 0.7 visible).
- Submit → `upsertDestino({ id, itemEsperadoId, categoria, subcategoria, etiqueta, confianza })`.
- Manejar `HttpsError` (p.ej. `invalid-argument` por FK inexistente) mostrando el `message`.

**Nota UX**: al vincular un item, dejar claro que el destino gana como **rama 2** en el

## Regression / smoke tests (post-deploy)

1. **Rules:** con admin, `/perfil/destinos` lista destinos reales de producción. Con
   dependiente, el ítem no aparece y la ruta redirige a `/`.
2. **Lectura FK:** un destino con `itemEsperadoId` muestra el **nombre** del ítem esperado
   (no el hash). Uno sin item muestra `categoria › subcategoria`.
3. **Umbral:** un destino con `confianza < 0.7` se ve con chip warning "bajo umbral".
4. **Crear (alias):** nuevo destino con `destinoRaw` = un alias de prueba (p.ej.
   `test.destino.gf`) + vincular a un **itemEsperadoId real** (usar el de Expensas u otro
   ítem vigente de `itemsEsperados`). Guardar → aparece en la lista con el item resuelto.
   Verificar en consola Firestore: `tipo:'alias'`, `destinoNorm` en minúsculas, `confianza:0.8`,
   `creadoPor` = tu email, id = 24 hex.
5. **Guard FK:** intentar crear/editar con un `itemEsperadoId` inexistente (inventado) →
   la callable rechaza con `invalid-argument` y el editor muestra el mensaje. **No** se
   escribe doc colgado.
6. **Editar confianza:** bajar la confianza del destino de prueba a 0.5 → chip warning.
   Subirla a 0.9 → sale del warning.
7. **Desvincular:** editar el destino de prueba, poner item "— sin vínculo —" pero dejar
   una categoría → guarda OK (queda como rama 3, prefill de categoría).
8. **Validación "vacío":** editar dejando sin item y sin categoría → rechazo
   `invalid-argument` ("Se requiere categoría o ítem esperado").
9. **Efecto en matching (opcional, si hay comprobante de ese payee):** cargar un comprobante
   cuyo `destinoNombre`/CBU normalice al destino de prueba vinculado → debe prellenar el
   ítem esperado (rama 2). Confirma el ciclo end-to-end.
10. **Borrar:** eliminar el destino de prueba (confirm) → desaparece. Verificar en Firestore.
11. **No regresión:** Diccionario (F8.1) sigue funcionando; el resto del grupo intacto.
