# F3.3 — Resolución de memberId y autorización

**Fase:** F3 (Auth + shell PWA) · **Sub-paso:** 3.3
**Prerequisito:** F3.2 cerrado (login/logout con Google funcionando contra emulador).
**Scope:** solo emulador. Nada toca producción.

---

## Contexto

El login ya funciona, pero cualquier cuenta puede entrar. Este paso agrega la autorización: resolver QUIÉN es el usuario logueado dentro de la familia, leyendo `/config/familia` con el SDK del cliente (decisión cerrada: sin Cloud Functions, para evitar costos).

Datos confirmados por la auditoría de tipos (F3.1.2) sobre `/config/familia`:
- `miembros` es `Record<string, FamiliaMiembro>` donde **la clave es el nombre de la persona** ("Juan", "María", etc.). Esa clave ES el `memberId`, y coincide con el campo `persona` de los movements.
- Cada `FamiliaMiembro` tiene: `nombre: string`, `emails: string[]`, `rol: 'admin' | 'dependiente'`, `activo: boolean`.

---

## Qué hace F3.3

1. **`src/familia.ts`** (o ubicación coherente con la estructura):
   - `resolverMiembro(email: string, config: FamiliaConfig): { memberId: string, miembro: FamiliaMiembro } | null`
   - Lógica: recorrer `Object.entries(config.miembros)` y buscar la entrada cuyo array `emails` contenga el email del user. Comparación **case-insensitive** (lowercase ambos lados) y con `trim()`.
   - Si el miembro encontrado tiene `activo === false` → tratar como NO autorizado (devolver null o estado explícito, a criterio, pero documentado).
   - Caso borde: si un email matchea más de un miembro (no debería pasar), tomar el primero y loguear warning en consola.

2. **Lectura de `/config/familia`:**
   - One-shot con `getDoc` (NO `onSnapshot` — regla del proyecto: listeners solo donde el realtime aporta valor; la config familiar no cambia en vivo).
   - Conversión del doc crudo a `FamiliaConfig`: el campo `actualizadoEn` llega como Timestamp de Firestore → convertir a `Date` en la lectura. Esta es la PRIMERA conversión Timestamp→Date del proyecto; dejarla en una función helper clara (`docAFamiliaConfig` o similar) porque el patrón se va a repetir en F4.

3. **Hook `useMiembro()`** (compone con `useFirebaseUser` de F3.2):
   - Estados expuestos: `{ estado: 'cargando' | 'noAutenticado' | 'noAutorizado' | 'autenticado', memberId: string | null, miembro: FamiliaMiembro | null, firebaseUser: User | null }`.
   - Flujo: sin user → `noAutenticado`. Con user → leer config → resolver → si matchea y activo → `autenticado`; si no → `noAutorizado`.
   - La lectura de config se hace UNA vez por sesión de login (no en cada render). Si falla la lectura (offline, rules), estado de error visible en consola y tratar como `noAutorizado` con mensaje claro, no crash.

4. **UI provisoria en `App.tsx`** (F3.4 la reemplaza):
   - `cargando` → "Cargando…"
   - `noAutenticado` → botón "Ingresar con Google"
   - `noAutorizado` → "Tu cuenta (email) no pertenece a la familia." + botón "Salir"
   - `autenticado` → "Hola {miembro.nombre} ({rol})" + botón "Salir"

5. **Datos de prueba en el emulador:** documentar cómo probar los dos caminos: crear en el popup fake una cuenta cuyo email SÍ esté en `miembros[*].emails` del doc sembrado (ver el emulator UI para los emails reales seedeados) → `autenticado`; y una cuenta con email inventado → `noAutorizado`.

## Criterio de cierre

- Login con email whitelisteado (del seed) → "Hola {nombre} ({rol})" correcto, con el `memberId` resuelto a la clave del Record.
- Login con email NO whitelisteado → `noAutorizado`, con opción de salir.
- Miembro con `activo: false` (editarlo a mano en el emulator UI para probar) → `noAutorizado`.
- Refresh: el estado se resuelve solo, sin parpadeos entre estados.
- `npm run validate` sigue en 8 OK / 0 FAIL.
