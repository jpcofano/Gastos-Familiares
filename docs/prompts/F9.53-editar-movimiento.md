# F9.53 — Editar / eliminar un movimiento ya cargado (buscador + editor, admin-only)

> Reglas: **CRUD de un `movimientos/{id}` existente vía callable admin-only** (mismo patrón
> que F9.36–41: el server fuerza invariantes, el cliente no escribe directo). Suma una
> pantalla **buscador** colgada de Configuración + el editor precargado, y hace tappables las
> filas de movimiento de Dashboard/Resumen para abrir el mismo editor. **No** toca el modelo ni
> los agregados (son derivados, F9.25). Ref visual: `ui_kits/mobile/{BuscarMovimiento,EditarMovimiento}.jsx`.

## Diagnóstico
- Hoy un movimiento mal clasificado (ej. quedó con subcat equivocada) **no se puede corregir**
  desde la app: hay que tocar Firestore a mano. Falta un buscador + editor.
- Alta manual ya escribe `movimientos`; **editar/eliminar uno existente no existe**.
- Decisiones que aplican: `persona` = **memberId** (vacío = familiar, F9.24) · medio se
  guarda **canónico** (Efectivo→MP, F9.23) · `categoria`/`subcat` = **label string** (F9.38) ·
  TC se aplica **por fecha** (`tcParaFecha`, decisión TC) · agregados **derivados, nunca
  persistidos** (F9.25 → no hay nada que recalcular al editar).

## Tarea

### 1. Callables admin-only (Cloud Functions) — patrón F9.36
Toda edición pasa por el server (auth → `/autorizados` esAdmin → validar → escribir). El
cliente **no** escribe `movimientos` directo para estas operaciones.

**`editarMovimiento({ id, cambios })`** — campos editables: `descripcion`, `monto`, `fecha`,
`tipo` (Gasto/Ingreso), `moneda` (ARS/USD), `categoria`, `subcat`, `persona`, `medio`.
Invariantes que **fuerza el server** (no confía en el payload):
- `persona`: vacío o un memberId que exista en `config.miembros`. Nunca un nombre suelto.
- `medio`: pasar por `medioCanonico()` (Efectivo→Mercado Pago).
- `categoria`: debe existir en la taxonomía; `subcat` opcional, validada contra las subcats de
  esa categoría (si no aplica, se limpia — no inventar).
- TC: si `moneda === 'USD'`, recomputar `tcUsdArs = tcParaFecha(fecha)`; si pasa a ARS,
  `tcUsdArs = null`. Recomputar **siempre** que cambie `fecha` o `moneda`.
- `itemEsperadoId`: si el movimiento estaba vinculado a un esperado y la edición no cambia
  identidad (persona/categoría/monto dentro de tolerancia), **mantener** el vínculo; si cambia
  sustancialmente, desvincular (no dejar un match inválido).
- Devolver el doc actualizado.

**`eliminarMovimiento({ id })`** — borra el doc. **Guardrails server-side:**
- Si el movimiento es un **pago de tarjeta consolidado** (`ExcluirDash=1` / `TarjetaPago`) o
  está atado a un `resumenesTarjeta`, **bloquear** con mensaje claro (borrarlo descuadra la
  conciliación) — se edita/borra desde el resumen, no acá.
- Resto: borrar y devolver ok.

### 2. UI — buscador (Configuración) + editor
- **Configuración familiar · admin → "Buscar / editar movimiento"** (item nuevo): abre la
  pantalla buscador. Search por texto (descripción / categoría / subcat / persona) + filtros
  rápidos (chips de persona + período). Lista de movimientos del período, tappable.
- **Editor** = patrón FullModal/Hero/Drawer (ver kit): precargado, todos los campos editables,
  **categoría → subcategoría en cascada** (al cambiar categoría se limpia la subcat que ya no
  aplica). **Aviso ámbar** si el movimiento viene de tarjeta/cuotas (editables igual, con
  advertencia — decisión del usuario). Guardar → `editarMovimiento`; Eliminar → confirmación →
  `eliminarMovimiento`.
- **Tap directo en Dashboard/Resumen:** las filas de movimiento abren el **mismo** editor
  (mismo componente, misma callable). No duplicar lógica.
- **Solo admin:** la entrada de Configuración y los callables. Un dependiente no ve el item ni
  puede invocar las funciones (server lo rechaza igual).

### 3. Importante (mapeos kit→real)
- El kit usa `persona` como **nombre** y aplana consumos de tarjeta en el buscador para la
  demo. En la app real: `persona` = **memberId**, y el buscador lista la colección
  `movimientos`. **Las líneas dentro de `resumenesTarjeta` (consumos parseados) NO son
  `movimientos` editables por esta vía** — quedan fuera de alcance (surface aparte).
- El monto admite decimales (F9.44). El picker de persona/medio/categoría puede reusar el
  bottom-sheet del kit (`Sheet`) o selects nativos — a criterio, respetando el patrón vigente.

## Aceptación
- Admin: Configuración → "Buscar / editar movimiento" → busca, abre uno, **corrige la subcat**
  (caso real: un movimiento con la subcategoría equivocada) y guarda → el cambio persiste y el
  Dashboard/Resumen lo reflejan (recalculan en vivo, sin agregados persistidos).
- Cambiar `fecha`/`moneda` recomputa `tcUsdArs` por fecha; cambiar `persona` guarda memberId
  (vacío = familiar); `medio` queda canónico.
- Eliminar un movimiento común funciona con confirmación; eliminar un **pago de tarjeta
  consolidado** queda **bloqueado** con mensaje.
- Tap en una fila de Dashboard/Resumen abre el mismo editor.
- Dependiente: no ve la entrada; invocar el callable a mano es rechazado (admin-only).

## Notas
- No persistir agregados (F9.25): tras editar, las vistas recalculan solas. No escribir totales.
- No re-litigar taxonomía: `categoria`/`subcat` siguen siendo **label string** (F9.38). Editar
  un movimiento cambia su label; no dispara el rename en cascada (eso es `guardarTaxonomia`).
- **Documentar en el repo:** registrar en `docs/CLAUDE.md` (callables `editarMovimiento`/
  `eliminarMovimiento` admin-only, invariantes forzados, guardrail de borrado de pago de
  tarjeta) y la verificación en `docs/RUNBOOK-migracion-prod.md`.
