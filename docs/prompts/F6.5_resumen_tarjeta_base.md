# F6.5 — Pipeline de resumen de tarjeta (base)

> Modo B: base congelado. Cambios → addendum. Requiere F6.4.5 (clasificador) cerrado — F6.5 lo consume.
> Emulador-only. Cierre toca docs/CLAUDE.md en el mismo commit.

## Motivo
Un `resumen_tarjeta` NO es un pago único: es un estado de cuenta con N consumos. Genera 1 doc
`resumenesTarjeta` + N movimientos (consumos) + rellena los 2 totales (ARS/USD) en los esperados de
tarjeta existentes. Porta la extracción multi-línea del viejo (47_Tarjetas_PDF.gs), que ya resolvía
cuotas, percepciones, reversos y multi-moneda.

## Decisiones cerradas
- **Q5 — Branch por `tipoDocumento`** en la function de extracción (F6.2): si `resumen_tarjeta`, corre el
  **prompt multi-línea dedicado** (portado del viejo) y escribe un draft de resumen, NO `datosExtraidos`.
  Una function, dos prompts.
- **Q6 — Preview del lote con deselección por línea**: la extracción escribe el resumen como **draft**
  (header + N líneas). La UI muestra las N líneas; el usuario revisa y puede deseleccionar (ej. una línea
  ya registrada). Al **confirmar el lote** → se crean los N movimientos + se rellenan los totales + se marca
  el resumen confirmado. **No auto-crea.** (Mismo principio que F6.3, en batch.)
- **Q7 — Genera consumos Y los 2 totales**, con flags anti-doble-conteo (ver abajo). Confirmado: NO dobla
  porque cada uno vive en una vista distinta.
- **Clasificación**: cada consumo pasa por el helper de F6.4.5 (`clasificar(descripcionRaw, { tarjeta })`)
  → sugiere cat/subcat (canónica), editable en el preview. Al confirmar, el diccionario aprende.
- **tarjetaCodigo del contenido** del PDF (no del nombre de archivo), con el catálogo como referencia.
- **Dedup doble**: SHA-256 del archivo (F6.1) bloquea re-subir el mismo PDF; `resumenTarjetaId =
  tarjeta+período` **avisa** (advisory, patrón F6.4) si ya hay resumen de esa tarjeta+período.

## Flags de cada movimiento creado (CRÍTICO — anti-doble-conteo)
- **Consumos** (N compras): `excluirDash: false` (cuentan en gasto por categoría), `incluirResumenMes:
  false` (no es salida de caja del mes, es crédito).
- **Los 2 totales** (ARS/USD): `excluirDash: true` (no son gasto nuevo, no doblan el dashboard),
  `incluirResumenMes: true` (es la plata real que sale del banco al pagar).
- Dashboard = consumos; Caja del mes = totales. Nunca se suman juntos.

## PASO 2 — Implementación (con aprobación)
### Extracción (branch resumen)
- En la function de F6.2, si `tipoDocumento === 'resumen_tarjeta'` → prompt multi-línea dedicado →
  escribir draft de resumen (header + líneas) + `estado` de draft. NO `datosExtraidos`.
- Detectar `tarjetaCodigo` del contenido.

### Preview + confirmación (cliente)
- Vista del resumen draft: header + tabla de N líneas, cada una con su cat/subcat **sugerida** por F6.4.5
  (editable), checkbox de inclusión (default on), tipo (consumo/cuota/percepción/reverso).
- Dedup advisory de entidad (`resumenTarjetaId`): si ya existe, avisar y permitir continuar.
- **Confirmar lote**:
  - Crear N movimientos (solo los seleccionados) con `resumenTarjetaId`, `origen: 'resumen'`, cat/subcat
    final, flags consumo (`excluirDash:false`, `incluirResumenMes:false`).
  - Rellenar los 2 totales: confirmar los esperados `tarjetaCodigo+moneda` con el total ARS y el total USD,
    creando los movimientos de pago con flags total (`excluirDash:true`, `incluirResumenMes:true`).
  - Aprendizaje del diccionario por cada consumo confirmado (F6.4.5).
  - Marcar el resumen confirmado.
- onSnapshot (F5.3) refleja todo sin refetch.
