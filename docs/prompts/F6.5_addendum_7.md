# F6.5 — Addendum 7 (estado autodescriptivo, fix del confirmado fantasma)

> Un resumen aparecía `confirmado` sin que nadie lo confirmara. Causa: el deserializador inventa
> `estado: 'confirmado'` cuando el documento no trae el campo, y el seed nunca escribe `estado`.
> Rompe el invariante "propone → confirmás". Fix: el documento persistido es la única fuente de
> verdad del estado.

## Diagnóstico (verificado en el repo)
- `src/datos/resumenesTarjeta.ts:35` → `estado: data.estado ?? 'confirmado'`. El fallback de lectura
  fabrica el estado terminal `confirmado` para cualquier doc sin el campo.
- `scripts/seed/transformers/cardStatements.ts` → los docs sembrados se escriben SIN `estado`
  (setean `confirmadoEn` / `confirmadoPor` cuando `EstadoImport === 'aplicado'`, pero nunca `estado`).
  Por eso TODOS los resúmenes sembrados caen en el `?? 'confirmado'` y se muestran confirmados.
  El "2025-05 $1.439.062" es uno de ellos — no es un fantasma puntual, son todos los sembrados.
- El único escritor legítimo de `estado: 'confirmado'` es la acción de usuario en
  `resumenesTarjeta.ts:367` (la que setea `confirmadoPor: memberId`).

## Decisión (A-over-B)
- **A (elegida):** el documento persistido es la única fuente de verdad de `estado`. El seed lo escribe
  explícito; el deserializador, si falta, cae al estado más conservador, **nunca** a uno terminal.
- **B (rechazada):** dejar `?? 'confirmado'` en el mapper. B mete cualquier doc sin el campo
  (seed, migración, doc viejo) directo al estado terminal por una puerta trasera de lectura, y acopla
  la corrección del estado a un fallback de presentación.

## Cambios

### 1. Seed escribe `estado` explícito
`scripts/seed/transformers/cardStatements.ts`, dentro del `.map(...)`, agregar el campo:
```ts
estado: r.EstadoImport === 'aplicado' ? 'confirmado' : 'parseado',
```
- `aplicado` en el legacy = ya estaba confirmado allá; `confirmadoEn` / `confirmadoPor` ya lo acompañan
  → `confirmado` legítimo y consistente.
- El resto trae datos parseados (totales, periodo) pero nunca se confirmó → `parseado`
  (no `subido`: el doc sembrado ya tiene contenido extraído).

### 2. Deserializador: default conservador
`src/datos/resumenesTarjeta.ts:35`:
```diff
- estado: data.estado ?? 'confirmado',
+ estado: data.estado ?? 'subido',
```
Después del cambio 1, ningún doc sembrado llega sin `estado`. Un doc sin el campo pasa a ser una
anomalía real, y `subido` (el más conservador del union) la muestra como "sin procesar" en vez de
esconderla como confirmada.

### 3. Re-sembrar
Re-correr el seed contra el **emulador** para que los docs ya sembrados tomen el `estado` correcto.
`writeBatch(db, 'resumenesTarjeta', docs)` reescribe por `ResumenID`, así que pisa el payload viejo
con el campo nuevo.
Para producción: el seed pisa por id; como ningún resumen fue confirmado por usuario todavía
(el final del pipeline nunca se probó), es seguro, pero correrlo con el guard `--i-am-sure` de siempre.

## No tocar
- El union de `estado` NO cambia acá (`'subido' | 'parseado' | 'confirmado' | 'error'`; si ya corriste
  el addendum 6, incluye además `'requiere_tarjeta'`). El 7 solo cambia un valor de default, no el tipo
  — por eso es independiente del orden respecto del 6.
- La acción de confirmar (`resumenesTarjeta.ts:367`) queda igual: sigue siendo el único camino a
  `confirmado` en runtime.

## Cierre

- Verificar que NINGÚN resumen aparezca `confirmado` sin `confirmadoPor` seteado.
- Subir un PDF nuevo y confirmarlo a mano → debe pasar `subido → parseado → confirmado` con
  `confirmadoPor: <vos>` (esto cierra de paso el item 1 pendiente: el final del pipeline nunca se probó).
