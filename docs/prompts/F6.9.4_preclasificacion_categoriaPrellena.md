## PASO 1 — F6.9.4: preclasificación
### Problema
El preload del alta (`src/vistas/Comprobantes.tsx`) arma la categoría solo con `clasificar(comercioRazonSocial)`
(diccionario de texto) y **nunca lee `pm.categoriaPrellena`** que produce el match por destino. Para una
transferencia, `comercioRazonSocial` suele ser "Mercado Pago"/vacío → el texto no engancha → el form abre
con categoría **vacía**, aunque el destino esté aprendido. De ahí el "no preclasifica nunca".

### Cambio (cliente, `src/vistas/Comprobantes.tsx`)

**Cambio 1 — preferir la prellena del destino sobre el texto (líneas 166-168).**
BUSCAR:
```
    categoria:           sugerenciaValida?.categoria    ?? undefined,
    subcategoria:        sugerenciaValida?.subcategoria ?? undefined,
    etiqueta:            sugerenciaValida?.etiqueta     ?? undefined,
```
REEMPLAZAR:
```
    categoria:           pm.categoriaPrellena    ?? sugerenciaValida?.categoria    ?? undefined,
    subcategoria:        pm.subcategoriaPrellena ?? sugerenciaValida?.subcategoria ?? undefined,
    etiqueta:            pm.etiquetaPrellena     ?? sugerenciaValida?.etiqueta     ?? undefined,
```

**Cambio 2 — el badge "Pre-clasificado" también cuando vino por destino (línea 195).**
BUSCAR:
```
      {sugerenciaValida && <span className="cmp-preclasificado">Pre-clasificado</span>}
```
REEMPLAZAR:
```
      {(pm.categoriaPrellena || sugerenciaValida) && <span className="cmp-preclasificado">Pre-clasificado</span>}
```

Notas:
- El override de rama 2 (categoría desde el esperado) **no se toca**: sigue ganando para items esperados.
- `pm.categoriaPrellena` ya viaja desde la capa de datos (`src/datos/comprobantes.ts`); solo faltaba consumirla.
- En la primera ocurrencia de un payee nuevo (nada aprendido) sigue abriendo vacío — es correcto. La
  preclasificación aparece a partir de la segunda, cuando el destino ya está en `destinos/{id}`.
