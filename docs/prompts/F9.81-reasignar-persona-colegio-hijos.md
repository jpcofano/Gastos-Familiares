# F9.81 — Migración: reasignar persona de "colegio Fede/Sofia" a un padre (ocultar a los hijos)

> Script de migración en `scripts/seed/` sobre la colección **`movimientos`** (mismo patrón que el backfill F9.24:
> dry-run por defecto → `--apply` para escribir; lo corre el usuario: emulador → prod). **No** es cambio de UI.

---

## Por qué

Los hijos (Fede, Sofía) son **dependientes**: la app les muestra solo sus movimientos vía
`where('persona','==', memberId)` (`movimientosDelMes`). Los gastos de **colegio de cada hijo** hoy tienen
`persona = <memberId del hijo>`, así que **los ven**. Reasignando `persona` a un **padre** (Juan o María), esos
gastos dejan de aparecerles (el padre sí los ve; el filtro por persona los excluye del hijo).

## §1 · Script `scripts/seed/reasignarPersonaColegio.ts`

- **Selección**: `movimientos` donde
  1. `persona` ∈ `IDS_HIJOS`, **y**
  2. normalizado (sin acentos, lowercase) de `descripcion` **o** `subcategoria` **o** `etiqueta` **o** `notas`
     **contiene "colegio"**.
  (Restringir a los hijos evita tocar gastos que ya son de un padre o familiares.)
- **Acción**: `persona = PADRE_DESTINO` + `actualizadoEn = serverTimestamp()`. Nada más (monto/categoría/fecha intactos).
- **Dry-run por defecto**: listar cada doc candidato (`id`, `fecha`, `descripcion`, `persona` actual → nueva) y el total.
  Con `--apply`: escribir en **batches de ≤450**.
- Reusar la conexión admin y el patrón de `backfillPersonaMemberId.ts`.

```ts
// ⚠️ CONFIRMAR estos IDs contra config.miembros (los 4 legacy usan nombre-como-id).
const IDS_HIJOS = ['Fede', 'Sofia'];        // o 'Federico'/'Sofía' según el id real
const PADRE_DESTINO = 'Juan';               // TODO a Juan (los dos hijos → Juan)

const norm = (s?: string) => (s ?? '').normalize('NFD').replace(/[\u0300-\u036f]/g, '').toLowerCase();
const esColegio = (m: any) =>
  ['descripcion', 'subcategoria', 'etiqueta', 'notas'].some(k => norm(m[k]).includes('colegio'));
```

## §2 · Evitar que se repita a futuro (regla deseada: colegios → padre)

El backfill solo arregla lo existente. Regla confirmada por el usuario: **los colegios siempre se asignan a un padre**
(a Juan). Para que los **nuevos** gastos de colegio caigan en el padre:

- **Ítems esperados**: si "Colegio Fede" / "Colegio Sofía" son `itemsEsperados` con `persona = <hijo>`, cambiarles
  `persona = 'Juan'` (así el prellenado y el alta manual sugieren el padre).
- **Diccionario**: si hay una regla que mapea "colegio…" → `persona = <hijo>`, actualizarla a `persona = 'Juan'`.
  Si no existe, crear la regla `contiene "colegio"` → `persona = 'Juan'`.

## §3 · Caveats

- **Tarjeta**: los consumos de tarjeta toman `persona` **según el dueño de la tarjeta** (no por este script). Como las
  tarjetas son de los padres, un colegio pagado con tarjeta ya queda en el padre → sin acción. Solo habría que revisar
  si alguna tarjeta estuviera a nombre de un hijo (no es el caso hoy).
- Reasignar `persona` **no** cambia montos ni agregados de la familia (Dashboard admin/Resumen ven todo igual); solo
  cambia **quién** es el dueño y, por ende, la visibilidad del dependiente.
- La callable `editarMovimiento` fuerza `persona = memberId`; el script corre con admin SDK (bypassa Rules), que es el
  camino correcto para bulk. Si se prefiere por-doc, usar `editarMovimiento` con `{ persona: PADRE_DESTINO }`.

## Test / corrida (usuario)
1. `dry-run` en **emulador** con seed: revisar la lista de candidatos (deberían ser solo los colegios de Fede/Sofía).
2. `--apply` en emulador → loguearse como Fede/Sofía: esos gastos **ya no aparecen**; como Juan/María, sí.
3. Repetir en **prod** (dry-run → `--apply`).
