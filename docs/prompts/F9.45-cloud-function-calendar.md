# F9.45 — Cloud Function `crearEventosVencimiento` (Canal B · Google Calendar)

> Reglas: doc-first · una PR. Implementa el **Canal B** de F9.43: crea/actualiza eventos de
> vencimiento en el único calendario de pagos (`juanymaypagos@gmail.com`) vía OAuth con un
> refresh token. Depende de que el seed capture `Config!B4` (Paso 0, incluido acá).

## Credencial (ya resuelta con el usuario)
- Calendario único: **`juanymaypagos@gmail.com`** (cuenta dedicada, en `Config!B4`).
- Credencial: **OAuth, un refresh token** (no service account — la cuenta es @gmail, no
  Workspace). Secrets de Functions (los carga el usuario, NO van a git ni al código):
  - `GOOGLE_OAUTH_CLIENT_ID`
  - `GOOGLE_OAUTH_CLIENT_SECRET`
  - `GOOGLE_OAUTH_REFRESH_TOKEN`
- ⚠️ El refresh token debe generarse con la app de OAuth **en Producción** (en Testing
  caduca a los 7 días — `refresh_token_expires_in`). Si la Function empieza a fallar con
  `invalid_grant`, es eso: republicar y regenerar el token.

## Paso 0 — capturar `Config!B4` en el seed
- `scripts/seed/readExcel.ts`: leer la hoja **`Config`** (hoy no se lee) y exponer al menos
  `B4`. Con `XLSX.utils.sheet_to_json` no alcanza para una celda suelta — leer la hoja como
  matriz (`{ header: 1 }`) y tomar `[3][1]` (fila 4, col B, 0-indexed), o por referencia de
  celda `ws['B4'].v`.
- `scripts/seed/transformers/config.ts`: agregar `calendarEmail: <Config!B4>` al doc
  `config/familia`. Si B4 viene vacío, dejar el campo sin setear (no inventar).

## Toggle global (admin) — `config.familia.calendarSync`
El Canal Calendar se prende/apaga con **un solo flag global** (es un calendario compartido,
lo configura el admin): `config/familia.calendarSync: boolean` (default `false` hasta que el
admin lo active). UI ya en el kit: `Perfil › Notificaciones` → card "Eventos en Google
Calendar" con switch (solo admin lo mueve; dependiente ve el estado Activo/Inactivo).
- Editable vía callable admin-only (mismo patrón que las configs F9.36–41).
- **La Function chequea el flag primero**: si `calendarSync !== true`, no crea/actualiza nada
  (y opcionalmente borra eventos futuros ya creados, con sus `calendarEventId`). El push por
  usuario (Canal C) es aparte y va post-PWA.

## Función `crearEventosVencimiento`
`functions/src/index.ts`. Patrón igual a `actualizarTCDiario` (F9.30): `onSchedule` diario
+ una **callable** gemela para disparo manual/test (admin-only, mismo gate `/autorizados`).

1. **Auth**: `google-auth-library` → `OAuth2Client(clientId, clientSecret)` con
   `setCredentials({ refresh_token })`; `googleapis` `calendar({version:'v3', auth})`. El
   `calendarId` es `config/familia.calendarEmail` (fallback: no correr y loguear si falta).
   **Antes de nada, chequear `config/familia.calendarSync === true`** — si está off, salir sin
   tocar el calendario.
2. **Recolectar vencimientos próximos** (ventana: hoy → hoy+N, N configurable, default 14):
   - `itemsEsperados` activos → `diaVencimiento` + `periodicidad` → fecha de vencimiento del
     período en curso. Monto esperado si existe.
   - `resumenesTarjeta` del período → `fechaVencimiento` + total a pagar (mismo total que usa
     el footer, F9.40).
   - `config.familia.tarjetas[]` → `venceDia` como respaldo si no hay resumen del período.
   - Dedup: si una tarjeta ya tiene resumen con `fechaVencimiento`, ese gana sobre `venceDia`.
3. **Crear/actualizar evento** por cada vencimiento:
   - `summary`: ej. `Vence: <label> · $<monto>` (o sin monto si no hay).
   - `start`/`end`: **all-day** en la fecha de vencimiento (`date`, no `dateTime`),
     timezone `America/Argentina/Buenos_Aires`.
   - `reminders`: popup 1 día antes + el mismo día (overrides).
   - **Idempotencia**: guardar el `calendarEventId` por (item × período) en el propio doc de
     origen (`itemsEsperados`/`resumenesTarjeta`, campo `calendarEventId` + `calendarPeriodo`).
     Si ya existe para ese período → `events.update`; si no → `events.insert` y guardar el id.
     Así re-correr no duplica.
4. **Tolerante a fallos**: un error en un evento loguea y sigue con los demás; un error de
   auth global (refresh token vencido) loguea claro y corta (no rompe el resto de Functions).

## Aceptación
- Seed escribe `config/familia.calendarEmail` desde `Config!B4`.
- La callable de test crea eventos all-day en el calendario de `juanymaypagos` para los
  vencimientos de los próximos N días, con recordatorios.
- Re-correr **no duplica** (update por `calendarEventId`).
- Sin secrets en código/git; build (`tsc` functions) verde.
