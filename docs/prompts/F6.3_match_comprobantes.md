> **Modo de trabajo**: investigá y reportá ANTES de escribir. Esperá aprobación explícita.

## Arquitectura aprobada (no reabrir)

- **El match corre server-side**, encadenado tras la extracción: trigger Gen 2
  `onDocumentUpdated('comprobantes/{hash}')`, region southamerica-east1, que dispara cuando un
  comprobante **pasa a `'extraido'`**.
- **El server PROPONE, no muta.** Calcula el match y escribe una `propuestaMatch` en el comprobante.
  **NO** crea movimientos, **NO** confirma pagos, **NO** toca esperados. Respeta el principio vigente:
  *la extracción nunca se auto-confirma — propone → confirmás* (y F5.5: el match no significa pagado).
- **El humano confirma desde la UI** y esa confirmación ejecuta la escritura **client-side**, reusando
  el confirmar-pago de F5.5 (rama 1) o el alta (ramas 2/3). El estado final del comprobante es `'vinculado'`.

### Guard anti-loop (importante)
El trigger escribe `propuestaMatch` con un `update`, que re-dispara `onDocumentUpdated`. Para no
loopear: correr **solo si** `before.estado !== 'extraido' && after.estado === 'extraido'`. La
auto-escritura deja `estado === 'extraido'` en ambos lados → guard false → no re-entra. Confirmar
(→`'vinculado'`) y errores tampoco re-disparan. Sin secret de Anthropic acá: el match es cómputo puro.

## PASO 2 — Implementación (recién con aprobación, ajustada a hallazgos)

### Tipo `Comprobante` (dispara ciclo seed/validate)
Agregar `propuestaMatch?`:
```ts
propuestaMatch?: {
  rama: 0 | 1 | 2 | 3;       // 0 = ya vinculado (movimiento con este hashPdf ya existe)
  movimientoId?: string;     // ramas 0 y 1
  itemEsperadoId?: string;   // ramas 1 y 2
  candidatos?: Array<{ tipo: 'movimiento' | 'esperado'; id: string; score?: number }>;
  calculadoEn: Date;
};
```

### Compartir lógica (opción A aprobada)
- Funciones nuevas en `functions/src/matchLogica.ts` (puro, sin SDK): `matchComprobanteConMovimiento`
  y `matchComprobanteConEsperado`. NO reusa el matcher de F5.5 (eje inverso).
- **Lo único que espeja a F5.5** es la evaluación de `matchTexto`: extraerla a un helper con nombre
  `evaluarMatchTexto(texto, matchTexto)` (lowercase `includes` por token de `incluye`, exclusión por
  `excluye`). Comentar que espeja la lógica del cliente; nota en `CLAUDE.md` de que se sincronizan a mano.

### Decisiones de match (cerradas)
- **Monto**: `Math.abs(Movement.monto)` vs `montoTotal` (siempre positivo, = 1er venc). Tolerancia
  **±5%**. Match **exacto puntúa más alto** que dentro de tolerancia (vía `score`). **Fallback (F6.2.2)**:
  probar también cada `datosExtraidos.vencimientos[].monto` (por si se pagó el 2º venc con recargo);
  el match por `montoTotal` puntúa más alto, los demás vencimientos son candidatos de menor score.
- **Temporal**: candidato si **mismo `mes` OR fecha ±7 días** (unión, NO corte duro). La proximidad de
  fecha solo afina el `score`. (Evita duplicar movimientos en servicios con gap emisión↔pago > 7 días.)
- **Texto**: `comercioRazonSocial.toLowerCase()` vía `evaluarMatchTexto`. Esperado sin `matchTexto` →
  match solo por moneda+tipo = **candidato débil** (va a `candidatos[]`, no como principal).
- **Link canónico** comprobante↔movimiento = `Movement.hashPdf`. Rama 1 lo estampa al confirmar;
  ramas 2/3 lo setean al crear. No crear un campo de "ref" nuevo.

### Trigger `matchComprobante` (server-side, sin secret)
```
onDocumentUpdated('comprobantes/{hash}', region southamerica-east1):
  - GUARD: before.estado !== 'extraido' && after.estado === 'extraido' (sino return)
  - leer datosExtraidos (moneda, montoTotal, fecha, comercioRazonSocial, tipoDocumento)
  - RAMA 0 (dedup por pago, primero): ¿existe movimiento con hashPdf === este hashPdf?
      → sí: propuestaMatch { rama:0, movimientoId }, y dejar el comprobante 'vinculado'. return.
  - match contra movimientos: candidatos por moneda dura + tipo + (mismo mes OR fecha ±7d) + monto ±5%
      + texto comercio vs descripción → ordenar por score (exacto de monto + fecha cercana = más alto)
  - match contra esperados: moneda+tipo duros + evaluarMatchTexto(comercio, matchTexto); sin matchTexto
      → candidato débil
  - rutear:
      hay movimiento candidato         → rama 1 (+ itemEsperadoId si el mov ya lo tiene o el esperado matchea)
      sin movimiento, esperado matchea → rama 2
      nada                             → rama 3
  - update: { propuestaMatch: {...} }   // estado SIGUE 'extraido'; NO toca otras colecciones
```
- Si hay varios candidatos, cargá todos en `candidatos[]` y que la UI deje elegir; no adivines uno solo.

### UI `/comprobantes` — confirmación (client-side, reusa F5.5)
Mostrar la propuesta por rama y un botón que ejecuta la escritura al confirmar:
- **Rama 0 — ya vinculado** → sin acción; el comprobante ya quedó `'vinculado'` desde el trigger.
  Mostrar a qué movimiento está atado (`movimientoId`). No volver a proponer.
- **Rama 1 — movimiento ya existe** → adjuntar comprobante (estampar `hashPdf` del comprobante en el
  movimiento) + confirmar pago **reusando el mecanismo de F5.5** (`confirmadoPago:true` + `itemEsperadoId`).
  **NO crea movimiento.** Comprobante → `'vinculado'`.
- **Rama 2 — solo esperado matchea** → crear movimiento **pre-linkeado** (`itemEsperadoId` +
  `confirmadoPago:true` + `hashPdf`, con los `datosExtraidos` precargados) tras confirmación del humano.
  Comprobante → `'vinculado'`.
- **Rama 3 — nada matchea** → abrir alta de movimiento **precargada** con `datosExtraidos` para revisión
  manual. Al guardar, estampar `hashPdf`. Comprobante → `'vinculado'`.
- "Deshacer" disponible donde aplique, igual que F5.5.
