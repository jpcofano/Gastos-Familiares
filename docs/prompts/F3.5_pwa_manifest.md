# F3.5 — Manifest PWA

**Fase:** F3 · **Sub-paso:** 3.5 (cierra la fase F3)
**Prerequisito:** F3.4 cerrado (AppShell con 4 estados funcionando).
**Scope:** solo emulador / local.

---

## Contexto

Último paso de F3: hacer la app instalable como PWA, igual que la app de comidas. SOLO instalabilidad — el offline real (persistencia de Firestore, caching) es compromiso de fases F4-F5, NO entra acá.

---

## Qué hace F3.5

1. **Manifest** (`manifest.webmanifest` o el approach que use el blueprint de comidas — replicar lo que ya funcionó ahí):
   - `name`: "Gastos Familiares" / `short_name`: "Gastos"
   - `display: standalone`, `start_url: /`, orientación y colores de tema coherentes con el estilo del shell.
   - Íconos en los tamaños estándar (192, 512, maskable). Si comidas tiene un set de íconos generado, usar el mismo pipeline; si no, generar íconos simples placeholder y dejar anotado que el ícono definitivo se diseña después.

2. **Registro:** link del manifest en el `index.html` + meta tags mínimos (theme-color, viewport ya debería estar).

3. **Service worker:** SOLO si el blueprint de comidas ya incluye uno mínimo para instalabilidad (algunos navegadores lo piden). Si se agrega, que NO cachee nada de Firestore ni intercepte llamadas — registro vacío o precache solo del shell estático. El offline de datos es F4-F5.

---

## Restricciones

- **NO** implementar `enablePersistentLocalCache` ni offline de Firestore (compromiso F4-F5).
- **NO** caching agresivo en el service worker si lo hay.
- **NO** tocar `scripts/seed/`, Rules, índices.

---

## Criterio de cierre

- Lighthouse o el navegador reconocen la app como instalable (ícono de instalar en la barra, o "Agregar a inicio" en mobile).
- La app instalada abre en modo standalone y el flujo de login funciona igual que en el navegador.
- `npm run validate` sigue en 8 OK / 0 FAIL.
- **Cierre de fase F3:** con esto, F3 completo. Actualizar `docs/CLAUDE.md`: estado de F3 → cerrado (mostrar diff antes de aplicar, como siempre).
