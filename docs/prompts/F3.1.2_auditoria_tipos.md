# F3.1.2 — Auditoría de tipos: emulador vs `src/types/index.ts`

**Fase:** F3 · **Sub-paso:** 3.1.2 (verificación, post 3.1.1)
**Scope:** solo lectura del emulador + reporte. NO modificar nada todavía.

---

## Contexto

F3.1.1 generó `src/types/index.ts` con interfaces puras del dominio. Al inspeccionar el emulador a mano aparecieron discrepancias de nombres entre lo que el seed (F2) escribió en Firestore y lo que declaran los tipos del cliente. Ejemplos ya detectados:

- `/config/familia` tiene el campo `actualizadoEn` (castellano), pero `index.ts` no lo declara y el resto de las interfaces usan `createdAt` / `updatedAt` (inglés).
- Sospecha de inconsistencia `pdfHash` (en `Movement`) vs `hashPDF` (en `CardStatement`).

La decisión cerrada del proyecto es **castellano camelCase para TODOS los campos y colecciones**. Donde el tipo esté en inglés y el seed haya escrito en castellano, el tipo está mal.

El objetivo de este paso es que NO haya divergencia entre los nombres reales en Firestore (lo que escribió el seed) y los tipos del cliente, porque cualquier divergencia produce lecturas `undefined` silenciosas en F4.

---

## Qué hacer

1. **Conectarse al emulador de Firestore por código** (Admin SDK contra el host del emulador, igual que corre el seed — `FIRESTORE_EMULATOR_HOST`). NO contra producción.

2. **Leer un documento de muestra (1–3 docs) de cada colección** que tenga interfaz declarada en `src/types/index.ts`:
   - `movements`
   - `cardStatements`
   - `expectedItems`
   - `config/familia`
   - (y cualquier otra colección que ya tenga tipo en `index.ts`)

3. **Extraer el set real de nombres de campo de cada doc**, incluyendo tipo de dato observado (string / number / boolean / array / map / timestamp / null). Para `config/familia`, expandir la estructura de `miembros` (¿es map keyed por memberId? ¿qué campos tiene cada miembro? ¿qué strings exactos aparecen en `rol`?).

4. **Comparar campo por campo contra cada interfaz de `index.ts`** y producir un reporte con tres listas por interfaz:
   - **Coinciden** (nombre exacto presente en ambos).
   - **En Firestore pero NO en el tipo** (campos que el cliente no declara).
   - **En el tipo pero NO en Firestore** (campos que el cliente declara y no existen → candidatos a nombre incorrecto o campo inventado).
   - Marcar explícitamente los pares sospechosos de ser el mismo campo con distinto nombre (ej. `updatedAt` ↔ `actualizadoEn`, `pdfHash` ↔ `hashPDF`).

5. **Para `config/familia` específicamente**, reportar la forma exacta de `miembros` y los valores literales de `rol` encontrados — esto es lo que F3.3 va a usar para resolver `memberId` y autorización.
