# F6.8 — Match por cuenta destino (aprendido) + dedup informativo (base prompt)

> Para transferencias y pagos, la **cuenta destino** es el dato de match más confiable, mejor que la
> descripción. Hoy `matchConEsperados` matchea por texto del comercio, y un duplicado solo dispara un
> flag de hash sin contexto. F6.8: el destino es clave de match de primera clase, **aprendida** (no
> configurada), y los duplicados informan a qué corresponden.

## Contexto (verificado en el repo)
- `matchComprobante` (onDocumentUpdated, index.ts:198) hace rama 0 (dedup por `hashPdf`,
  index.ts:219-236) y `matchConEsperados` (matchLogica.ts:119) por `matchTexto.incluye` sobre el comercio.
- La extracción saca `cuentaDebito` (origen), `numeroCuenta` (cliente), `comercio`, `numeroCliente`.
  **No** saca el destino estructurado de una transferencia.
- El aprendizaje de categorías es server-side on-write (`aprenderMovimientoCreado/Actualizado`,
  index.ts:400/410) con normalización antes de insertar e idAprendido (index.ts:308).

## Decisiones fijadas (de Juan)
- **Mapa aparte** `destinos/{idNorm}` (no campo en itemEsperado), al estilo del diccionario.
- El "adicional" es un **movimiento separado** con la categoría/etiqueta del item, no suma al item.
- Extraer el destino **para todos los comprobantes** (no solo transferencias).

## Modelo de datos

`destinos/{idNorm}` (mapa aprendido):
```ts
interface Destino {
  destinoNorm: string;          // clave normalizada (CBU dígitos / alias lower / nombre normalizado)
  tipo: 'cbu' | 'alias' | 'nombre';
  itemEsperadoId?: string;      // si el destino está asociado a un esperado
  categoria?: string;
  subcategoria?: string;
  etiqueta?: string;
  confianza: number;            // misma mecánica que el diccionario (umbral 0.7, +0.1 al corregir)
  creadoPor: string;
  actualizadoEn: Timestamp;
}
```

Extracción del comprobante — agregar (todos los tipos, no solo transferencia):
- `destinoCbu?` (CBU/CVU, solo dígitos), `destinoAlias?` (alias), `destinoNombre?` (receptor/beneficiario).

## Cambios

### 1. Extracción (prompt del comprobante)
Agregar al prompt y al JSON los campos `destinoCbu`, `destinoAlias`, `destinoNombre`. Para
transferencia = la contraparte receptora; para factura/servicio = CBU/cuenta de pago del emisor si
figura. null si no aplica.

### 2. Aprendizaje (extender el trigger existente, NO crear uno paralelo)
En `aprenderMovimientoCreado/Actualizado`: cuando el movimiento confirmado tiene un destino y está
vinculado a un item (o tiene categoría/etiqueta), hacer upsert en `destinos/{idNorm}`:
- **Normalizar antes de insertar** (CBU = dígitos; alias = lower/trim; nombre = via `normalizador`).
  (Pitfall conocido del repo: guardar el string crudo crea entradas que nunca vuelven a matchear.)
- Misma mecánica de confianza que el diccionario (+0.1 al corregir).
- Así la **primera vez** que confirmás una transferencia contra un esperado, se aprende el destino;
  no se precargan CBUs.

### 3. Match por destino (rama nueva, prioridad sobre el texto)
En `matchComprobante`, antes del match por texto: normalizar el destino del comprobante, buscar en
`destinos`. Si hay hit con `itemEsperadoId`:
- esperado **impago** ese período → proponer match + `confirmadoPago: true`.
- esperado **ya pagado** ese período → proponer **movimiento separado** con `categoria/subcategoria/
  etiqueta` del item prellenadas (el "adicional"), sin re-marcar el item.
Si el hit tiene solo categoría (sin item) → prellenar categoría/etiqueta (estilo adicional), sin vínculo.
El match por destino es más fuerte que el de texto: si ambos disparan, gana destino.

### 4. Dedup informativo
- **Rama 0 (hash):** además de detectar, escribir en el doc del comprobante un `dedupInfo`
  `{ movId, item?, mes, monto }` leyendo el movimiento existente, para que la UI muestre
  "ya cargado: <item> de <mes>, $<monto>" con link — en vez de un flag pelado.
- **Advisory semántico:** si `destino + monto + período` coincide con un movimiento existente (archivo
  distinto, mismo gasto) → marcar advisory "posible ya cargado como <X>" (no bloquea; informa).

## No tocar
- El diccionario de categorías por descripción (sigue para lo que no tiene destino).
- El cuadre, la confirmación de tarjetas, el router de F6.7.

## Criterios de cierre
- Cargar factura del colegio (crea/identifica el esperado) → transferir → subir comprobante de la
  transferencia → matchea por destino → esperado marcado **pagado**.
- Segunda transferencia al mismo destino (esperado ya pagado) → movimiento separado con la etiqueta
  del colegio, no re-marca el esperado.
- Subir el **mismo archivo** ya cargado → la UI dice a qué está vinculado (no un flag mudo).
- Subir **otro archivo** del mismo gasto/período al mismo destino → advisory "posible ya cargado como X".
- Primera vez sin destino aprendido → no rompe: cae al match por texto / queda para confirmar y aprende.
