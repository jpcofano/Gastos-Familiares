# F5.3 — Realtime (onSnapshot + offline)

> **Modo de trabajo**: investigá y reportá ANTES de escribir. Esperá aprobación explícita.
> Emulador-only. Refactor 100% cliente (`src/`). Sin backend, sin cambios de tipos/seed.

## Motivo
El pipeline de comprobantes es asincrónico: la function escribe `extraido`/`error`/`propuestaMatch`
sin que la UI se entere → hoy refrescás a mano para ver cada cambio del trigger. Realtime ataca eso de
raíz. Beneficia además Resumen, Dashboard, Esperados y listas de movimientos.

## Principio clave (no sobre-ingeniería)
**Firestore ya hace latency compensation.** Una escritura local dispara el callback de `onSnapshot`
*al instante*, antes del ack del server, con `snapshot.metadata.hasPendingWrites === true`. Es decir:
con `onSnapshot` bien puesto, el "optimistic update" sale **gratis** — NO hay que mantener estado
optimista a mano salvo un caso puntual que lo justifique. Default: apoyarse en el SDK, no hand-rollear.

## PASO 2 — Implementación (recién con aprobación, ajustada a hallazgos)

### Suscripciones (reemplazar one-shot por onSnapshot)
- Patrón a definir en PASO 1:
  - **Hook por colección** (`useComprobantes`, `useMovimientos`, `useItemsEsperados`): `onSnapshot` en
    `useEffect`, devuelve datos + loading + error, **cleanup del unsubscribe** en el return del effect.
  - **Context compartido** para colecciones consumidas por varias vistas (un solo listener, N consumidores).
- Prioridad de cobertura: `comprobantes` (máximo valor), `movimientos`, `itemsEsperados`. Las vistas
  derivadas (Resumen, Dashboard) recomputan de los datos en vivo — no suscribir un "resumen", suscribir
  las fuentes y derivar.
- Colecciones casi-estáticas (`config`, `diccionario`, `etiquetas`, `subcategorias`): pueden quedar
  one-shot o suscribir con baja prioridad; decidir según el mapa. No malgastar listeners.

### Offline
- Habilitar cache persistente **una vez** en el init, con la API moderna del SDK modular:
  `initializeFirestore(app, { localCache: persistentLocalCache({ tabManager: persistentMultipleTabManager() }) })`.
  (Si el SDK es viejo y solo está `enableIndexedDbPersistence`, reportarlo en PASO 1 antes de tocar.)
- Multi-tab contemplado por `persistentMultipleTabManager`.

### Optimistic
- **Default: ninguno manual.** Apoyarse en latency compensation. Opcional: usar `hasPendingWrites` para
  un indicador sutil de "guardando…" donde sume, sin duplicar estado.

### Limpieza
- Todo `onSnapshot` con su `unsubscribe` llamado en cleanup. Sin listeners colgados al desmontar.
- Quitar los `getDocs` one-shot que queden muertos tras el refactor (no dejar las dos vías).
