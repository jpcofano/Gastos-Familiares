# F9.50 — Fix deploy: subir memoria de las funciones de aprendizaje (128→256 MiB)

> Reglas: doc-first. **Toca SOLO `functions/src/index.ts` (2 líneas).** Sin cambios de
> lógica, modelo ni triggers. Desbloquea el `firebase deploy --only functions`.

## Síntoma
`firebase deploy --only functions` falla SOLO en dos funciones:
```
Could not create or update Cloud Run service aprendermovimientocreado,
Container Healthcheck failed. The user-provided container failed to start and
listen on the port defined provided by the PORT=8080 environment variable...
```
Mismo error en `aprendermovimientoactualizado`. Todas las demás (callables, cron MEP,
`extraerComprobante`, `matchComprobante`, `routearEntrante`, `extraerResumenTarjeta`)
deployaron OK.

## Causa (confirmada leyendo el código)
Las dos funciones de aprendizaje están declaradas con **`memory: '128MiB'`**:
```ts
export const aprenderMovimientoCreado = onDocumentCreated(
  { document: 'movimientos/{id}', memory: '128MiB' },  // ← acá
  ...
export const aprenderMovimientoActualizado = onDocumentUpdated(
  { document: 'movimientos/{id}', memory: '128MiB' },  // ← acá
```
Una función gen2 con Node 22 que carga `firebase-admin` **no arranca en 128 MiB** — se queda
sin memoria antes de escuchar en el puerto 8080 → falla el healthcheck del contenedor. El
mínimo práctico es **256 MiB** (que además es el default que usan todas las demás del proyecto;
por eso ninguna otra falla). Las funciones pesadas que llaman a Anthropic ya usan 512/1024.

## Tarea
En `functions/src/index.ts`, cambiar `memory: '128MiB'` → `memory: '256MiB'` en las dos
definiciones:
- `aprenderMovimientoCreado` (≈ línea 608)
- `aprenderMovimientoActualizado` (≈ línea 622)

Nada más. No tocar la lógica de `aprender()` / `aprenderDestino()` ni los triggers.

## Despliegue (lo corre el usuario)
Las dos funcionaban como recursos rotos (estado FAILED / revisión descartada); ya se borraron
con `gcloud functions delete`. Tras el fix:
```bash
firebase deploy --only functions
```
Deberían crearse OK (trigger Firestore `movimientos/{id}`, 256 MiB). Verificar:
```bash
gcloud functions list --project=gastos-familiares-e6415
```
`aprenderMovimientoCreado` y `aprenderMovimientoActualizado` deben quedar **ACTIVE** con
trigger de Firestore (no `https`, no `FAILED`).

## Aceptación
- `firebase deploy --only functions` termina sin errores (20/20 ACTIVE).
- Las dos `aprender*` quedan ACTIVE, trigger `movimientos/{id}`, 256 MiB.
- Typecheck de `functions/` sin errores nuevos.

## Nota
No bloquea la migración: estas dos son el **feedback loop de aprendizaje** (categoría/destino),
no el camino crítico del import. Si urge avanzar, el resto del runbook (import + validate) corre
sin ellas; este fix se puede aplicar en cualquier momento.
