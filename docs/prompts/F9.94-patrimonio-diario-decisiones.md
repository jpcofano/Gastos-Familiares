# F9.94 — Patrimonio: diario de decisiones (registro + revisión 30/90 días)

## Contexto

Cierra el ciclo "proponer y medir": hasta ahora el sistema mide la cartera;
esto mide las **decisiones del dueño**. Se registra una decisión con la foto
de métricas del momento, y el sistema la trae a revisión a los 30 y 90 días
mostrando qué cambió — sin juzgarla. Feature prevista en el roadmap del
contrato (`docs/patrimonio/CLAUDE-PATRIMONIO.md`).

Principio innegociable: **el sistema NO puntúa decisiones** (ni "acierto",
ni colores buenos/malos sobre el resultado). Muestra deltas y pregunta;
el juicio es del dueño. La revisión evalúa proceso, no solo resultado.

## 1) Colección `decisionesPatrimonio/{autoId}`

```ts
{
  creadaEn: Timestamp,
  titulo: string,                 // "Recorte TRAN a ~10%"
  razon: string,                  // por qué la tomo (texto libre)
  tickers: string[],              // involucrados, puede ser []
  opcionReferencia: string | null,// 'A'|'B'|'C'|null si nace de una opción del Plan
  metricasAlCrear: {              // foto al registrar
    totalInvertibleUsd, energiaArPct, paisArPct, criptoPct,
    top1Ticker, top1Pct, hhi,
    valoresTickers: { [ticker]: valorUsd }   // solo los de `tickers`
  },
  revisiones: [{                  // se agregan al revisar
    tipo: '30d' | '90d',
    fecha: Timestamp,
    notas: string,                // qué pasó / la volvería a tomar
    metricasAlRevisar: { ...misma forma... }
  }],
  estado: 'abierta' | 'revisada30' | 'revisada90' | 'cerrada'
}
```

Reglas Firestore: `allow read, write: if esDueno();` (patrón existente).

## 2) Registrar decisión

- **Desde una opción del Plan:** botón "Registrar decisión" en cada card de
  opción → modal con `opcionReferencia` y `tickers` precargados, campos
  titulo/razon. Al guardar, capturar `metricasAlCrear` de las métricas
  vigentes (reusar el cálculo existente; no recalcular distinto).
- **Libre:** botón "+ Decisión" en la sección Diario (ver §3) sin precarga.
  Una decisión válida puede ser "mantener todo como está" — el modal no debe
  exigir tickers.

## 3) Sección "Diario de decisiones" (al pie de la solapa Plan)

- Lista de decisiones desc por fecha: titulo · fecha · chips de tickers ·
  badge de estado. Tap → detalle expandido: razon, métricas al crear, y las
  revisiones que existan (cada una con su delta y notas).
- **Aviso de revisión pendiente:** si `creadaEn + 30d` pasó sin revisión 30d
  (o +90d sin la de 90d), badge ámbar "revisión pendiente" en la fila, y un
  contador discreto en el tab Plan (ej. dot en el label). Chequeo al cargar
  la vista — sin notificaciones push, sin schedulers.

## 4) Flujo de revisión

- Desde la decisión con revisión pendiente → pantalla/modal "Revisión 30d"
  (o 90d):
  - **Tabla entonces → hoy** de las métricas capturadas + valores de los
    tickers involucrados, con el delta en USD y pp. Sin colores de
    bueno/malo — delta en tipografía neutra.
  - Dos preguntas de texto libre: "¿Qué pasó desde entonces?" y
    "¿La volverías a tomar? ¿Por qué?" (van juntas a `notas`).
  - Guardar → agrega la revisión, actualiza `estado`.
- La revisión 90d se habilita aunque la 30d no se haya hecho (no bloquear;
  marcar la 30d como omitida visualmente si se salteó).

## 5) Informe PDF (F9.92) — sección nueva

Agregar sección "Decisiones registradas" (entre Opciones y Análisis IA):
decisiones abiertas y las revisadas en los últimos 90 días, cada una con
titulo, fecha, razon (resumida), y si tiene revisiones, el delta principal +
las notas. Si no hay decisiones: omitir sección limpiamente (patrón de las
secciones IA).
