## Ajustes obligatorios (los 3 puntos trabados)

### 1. Interfaces puras, sin `Timestamp` de ningún SDK
- `src/types/` NO debe importar `Timestamp` ni del admin SDK ni del cliente. Las interfaces deben quedar 100% libres de SDK.
- Los campos de fecha/timestamp se representan como tipo de dominio propio: usar `Date` en el dominio.
- La conversión `Timestamp ↔ Date` es trabajo de F4 (capa de mapeo al leer de Firestore). **NO la implementes ahora.**
- Mantener los nombres de campo del modelo F1 exactos: `tcUsdArs`, `fechaConsumoOriginal`, `mes`, `parentId`, `persona`, `moneda`, etc. (castellano camelCase).

### 2. Config de prod por env vars + gitignore
- Rama de prod de `firebase.ts` lee de `import.meta.env.VITE_FIREBASE_*`.
- Crear `.env.example` documentando las claves necesarias (sin valores reales).
- Confirmar que `.env` y `secrets/` estén en `.gitignore`. Si no lo están, agregarlos.

### 3. `types/index.ts` acotado a lo firme
- Tipar SOLO lo que F3 necesita más lo que esté 100% claro del modelo F1.
- Mínimo requerido para F3: el tipo de `/config/familia` (con `miembros[*].emails`, `persona`, rol) — es lo que F3.3 leerá para resolver el `memberId`.
- `Movement` / `CardStatement` pueden entrar si los campos están firmes.
- Si hay duda sobre algún campo, **marcalo y preguntá** en vez de adivinar. No inventes campos.
