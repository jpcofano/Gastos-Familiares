# F9.42 — Seed: adjuntar líneas (consumos + cuotas) a resumenesTarjeta

> Reglas: doc-first. **Toca seed + (mínimo) viewer/validador.** Arregla de raíz el
> "Revisar (0 líneas)", el FAIL de cuadre del gate y la ausencia de cuotas — **todo es la
> misma causa**: el seed nunca adjunta los consumos al doc del resumen.

## Diagnóstico (auditado contra la planilla `2026-06-19_sheet_snapshot.xlsx`)
- `scripts/seed/transformers/cardStatements.ts` crea cada `resumenesTarjeta/{ResumenID}` con
  el **header** (`totalARS`, `totalUSD`, cierre, vence, estado…) pero **sin
  `movimientosParseados`** → todos los resúmenes quedan con **0 líneas**.
- Las **líneas existen** en la hoja **`Tarjetas_Movimientos`** (no en `Tarjetas_Resumen`):
  **1009 consumos** repartidos en **17 resúmenes**, **con `CuotaActual`/`CuotaTotal`
  poblados** (cuotas reales, ej. "REX 2/3", "LA ESQUINA V SRL 4/9").
- Por eso: "Revisar (0 líneas)" en el viewer, el **cuadre del gate FALLA** (Σ líneas = 0 vs
  total) y **no se dibujan cuotas**. Una sola raíz, tres síntomas.

### Mapa de columnas — `Tarjetas_Movimientos`
`0:MovimientoResumenID · 1:ResumenID · 3:FechaConsumo · 4:Descripción · 5:Moneda ·
6:Monto · 8:Tarjeta · 9:Categoría · 11:TC_USDARS · 18:FechaImputacion · 19:MesImputacion ·
20:Persona · 22:Subcategoria · 23:Etiqueta · 25:CuotaActual · 26:CuotaTotal`

## El modelo NO cambia — el seed debe escribir la forma exacta
La colección ya está bien y es **única**: `resumenesTarjeta.movimientosParseados:
MovimientoParseado[]`. El runtime (subir PDF → `confirmarResumenTarjeta`) y el viewer
(`docACardStatement`, `calcularCuadre`) usan **ese mismo campo y tipo**. El parseo futuro
queda bien tal cual; **no hay cambio de esquema**. Lo único: el **seed debe escribir cada
línea con la forma EXACTA de `MovimientoParseado`** (no una forma propia), si no el cuadre
sigue fallando.

### ⚠️ Crítico — `calcularCuadre` salteа líneas mal formadas
`calcularCuadre` hace `if (!l.incluir || l.monto <= 0) continue;` y `tipoDeLinea(l)` lee
`l.tipoLinea`. Entonces, sin `incluir: true` y sin `tipoLinea`, **la suma da 0 y el cuadre
FALLA igual aunque adjuntes las líneas**. Hay que respetar el tipo completo.

## Tarea
1. **`readExcel.ts`**: cargar la hoja `Tarjetas_Movimientos` (hoy solo se lee
   `Tarjetas_Resumen`). Exponerla como `data.tarjetasMovimientos`.
2. **`cardStatements.ts`**: agrupar los consumos por `ResumenID` y adjuntarlos como
   **`movimientosParseados: MovimientoParseado[]`** en el doc del resumen, con la **forma
   exacta del tipo** (`src/types/index.ts`). Mapeo desde `Tarjetas_Movimientos`:
   ```
   {
     seq:               <índice 1..N dentro del resumen>,
     tipoLinea:         CuotaTotal > 1 ? 'cuota' : 'consumo',
     fechaConsumo:      <FechaConsumo como 'YYYY-MM-DD' string, NO Date>,
     descripcionRaw:    Descripción (col 4),
     nroCupon:          '',
     cuotaActual:       Number(CuotaActual) || 1,      // col 25
     cuotaTotal:        Number(CuotaTotal) || 1,       // col 26
     moneda:            Moneda (col 5),
     monto:             Math.abs(Number(Monto)),       // SIEMPRE positivo (col 6)
     personaDetectada:  Persona (col 20) || '',
     esBonificacion:    false, esReverso: false, esImpuesto: false,
     personaConfirmada: Persona (col 20) || null,
     categoria:         Categoría (col 9) || null,
     subcategoria:      Subcategoria (col 22) || null,
     incluir:           true,                           // ← imprescindible para el cuadre
   }
   ```
3. **NO crear movimientos nuevos.** Los movimientos de tarjeta ya los siembra el
   `movements` transformer (desde Histórico). `movimientosParseados` es el **snapshot
   parseado del PDF** embebido en el resumen — adjuntarlo NO debe duplicar movimientos
   (si no, doble conteo en Dashboard/Resumen).
4. **Coherencia de totales**: con `incluir:true` + `tipoLinea` correctos,
   `calcularCuadre(movimientosParseados, totalARS, totalUSD)` debe dar balance (±1) para los
   17. Si alguno no cierra, loguear (no romper).
5. **Viewer**: confirmar que `TarjetaFace`/visor leen `movimientosParseados` y dibujan el
   split de cuotas (F9.21) + "Revisar (N líneas)" con N real.

## El resumen sin líneas — `BBVA-VISA-SIG_1414251`
- Tiene header en `Tarjetas_Resumen` pero **0 consumos** en `Tarjetas_Movimientos` **y** en
  `Historico`. No es un bug del seed: **los consumos no existen en la planilla** (re-seedear
  no los crea). Decisión:
  - **(a)** dejarlo como resumen **header-only** → el viewer muestra "sin detalle de
    consumos" + el total; el validador lo **saltea** del cuadre (0 líneas = no aplica ±1), o
  - **(b)** si esos consumos importan, hay que **agregarlos a la planilla** (vienen del PDF
    original) y re-seedear.
  Recomendado (a) salvo que el usuario tenga el detalle de ese resumen.

## Validador (gate F9.31)
- El cuadre `resumenesTarjeta (±1)` debe pasar para los **17** una vez adjuntas las líneas.
- Para resúmenes con **0 líneas** (el 1 header-only): **skip + WARN**, no FAIL (no hay
  detalle contra qué cuadrar). Así el gate queda **21/21** sin enmascarar un cuadre real.

## Aceptación
- Los 17 resúmenes muestran sus consumos (N líneas reales) y el split de cuotas con
  `CuotaActual/CuotaTotal` de la planilla.
- El cuadre del gate pasa para los 17; `BBVA-VISA-SIG_1414251` queda skip+WARN (o resuelto
  si se cargan sus consumos).
- Sin duplicar dato: las líneas se adjuntan desde `Tarjetas_Movimientos`, no se inventan.
- Re-seed del emulador requerido para que aplique (lo corre el usuario).

## Notas
- Esto **destapa las cuotas de verdad** (la data estaba, no se cargaba). Mismo origen que el
  "FAIL" del gate y el "Revisar (0 líneas)". Ver `docs/pulido-kit-app.md` (ítems 5, 11) y
  `CLAUDE.md`.
