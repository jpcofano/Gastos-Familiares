# F5.4 — Página de configuración de gastos/ingresos esperados (ABM, app-owned)

> Objetivo: `itemsEsperados` pasa a editarse desde la app (admin). El Excel queda como seed de
> dev/migración; la página es el editor de producción. Ítems creados en la app usan **id automático**
> (las colisiones dejan de existir; el matcher es id-agnóstico, no se rompe nada).
>
> Tipos ya listos: `matchTexto` es objeto (F4.2.1.1), `periodicidad`/`pagoAutomatico` existen (F5.2).
> Esta fase agrega la vista, las escrituras y las Rules.

## Regla de trabajo
Investigá y reportá antes de escribir; esperá OK. Confirmá:
1. Fuente de `categorias`, `subcategorias` y `tarjetas` en `config/familia` (¿hay lista de subcategorías? hay transformer `subcategorias.ts`).
2. El bloque actual `match /itemsEsperados/{id}` en `firestore.rules`.
3. Cómo se navega/rutea hoy (React Router) para enganchar la ruta admin.

## Alcance (todo junto: editar + crear + baja)

### 1. Datos — `src/datos/itemsEsperados.ts`
Agregar:
- `crearItemEsperado(data): Promise<Resultado<string>>` → `addDoc` (id automático). Devuelve el id.
- `actualizarItemEsperado(id, data): Promise<Resultado<void>>` → `updateDoc` + `actualizadoEn = serverTimestamp()`.
- `desactivarItemEsperado(id)` / `reactivar` → `updateDoc({ activo })` (soft-delete, default).
- `eliminarItemEsperado(id)` → `deleteDoc` (baja dura, solo con confirmación explícita).
- `itemsEsperadosTodos()` → lista TODOS (activos + inactivos) para el ABM (la actual `itemsEsperadosActivos` filtra `activo==true`; no tocarla, se usa en Resumen).

### 2. Vista — `src/vistas/ConfigEsperados.tsx` (+ ruta admin-only + link en el shell)
- Dos pestañas: **Gastos** / **Ingresos** (misma colección, filtra por `tipo`).
- Lista agrupada con los campos clave + badge activo/inactivo. Inactivos atenuados.
- Botones: crear, editar (abre form), desactivar/reactivar, eliminar (confirm).
- Guard admin: si no `esAdmin`, no se renderiza (mismo patrón `ResumenAdmin`).

### 3. Formulario (crear/editar) — campos
- `tipo` (Gasto/Ingreso), `activo` (toggle)
- `categoria` / `subcategoria` → dropdowns desde `config/familia` (subcategoría según la fuente que confirmes)
- `persona` → dropdown de miembros + opción "(ninguna)"
- `moneda` ARS/USD
- `tarjetaCodigo` → dropdown desde `config.tarjetas` + "(ninguna)"
- `montoEsperado` (number, opcional), `diaVencimiento` (1–31, opcional)
- `periodicidad` (mensual/bimestral/trimestral/anual/unico)
- `pagoAutomatico` (toggle) — **este es el que hoy no tenías dónde setear**
- `matchTexto.incluye` / `matchTexto.excluye` → editor de **chips** (agregar/quitar strings, se guardan en minúsculas)
- `banco`, `etiqueta`, `notas`, `autoCalendario` (opcionales)
- id: en crear, automático (`addDoc`). En editar, NO se toca el id (estable; editar categoría no huerfana movimientos linkeados por `itemEsperadoId`).

### 4. Aviso de duplicados al crear
- Si ya existe un ítem **activo** con misma tupla `(tipo, categoria, subcategoria, persona, moneda)` → `window.confirm` "ya existe un esperado parecido", permitir continuar.

### 5. Rules — `firestore.rules`
- `match /itemsEsperados/{id}`: `allow read: if esMiembro();` `allow create, update, delete: if esAdmin();`
  Validar en create/update: `tipo in ['Gasto','Ingreso']`, `moneda in ['ARS','USD']`, `activo is bool`,
  `periodicidad in ['mensual','bimestral','trimestral','anual','unico']`, `pagoAutomatico is bool`.
- **Aprovechá esta pasada de Rules para aplicar el hotfix P0** (`F5.1.0_hotfix_autorizados_P0.md`):
  seed transformer `autorizados`, `authDoc()` por `request.auth.token.email.lower()`,
  `/autorizados` `write: if false`, eliminar el `setDoc` de `useMiembro.ts`. Evita una segunda pasada de Rules + re-seed.

### 6. Migración / seed — sin cambios destructivos
- El seed de `itemsEsperados` (desde Excel) se mantiene como fixture de dev. `MATCH_TEXTO_OVERRIDES`
  queda en el transformer (siembra el `matchTexto` de los ítems migrados); la UI lo edita de ahí en más.
- Nota para el validator: el check "count itemsEsperados == filas activas Excel" solo vale **inmediatamente
  después de un seed limpio**. Si usás export/import del emulador y creás ítems en la app, ese conteo va a
  superar las filas de Excel — es esperado, no es FAIL.

### 7. Ciclo de seed limpio
`rmdir /s /q emulator-data` → emulators → `npm run seed` → `npm run validate` → verde
(incluido el check de autorizados del P0).

## Definición de hecho
- Como admin: crear un esperado nuevo (id automático), editarlo, desactivarlo/reactivarlo, eliminarlo.
- Togglear `pagoAutomatico` en un ítem y verlo pasar a **Automático** en `/resumen` sin re-seed.
- Editar `matchTexto.incluye/excluye` con chips y ver el match cambiar.
- Un dependiente NO ve la página / no puede escribir `itemsEsperados` (probar con SDK directo: create debe fallar).
- P0: un cliente no puede escribir `autorizados/*` (create debe fallar).
- `npm run validate` verde tras seed limpio.

## Recomendación operativa (no parte del código)
Empezá a usar `firebase emulators:start --import=./emulator-data --export-on-exit` para que las ediciones
de la página persistan entre sesiones sin re-seed. Reservá el `rmdir` para cuando quieras resetear a cero.
