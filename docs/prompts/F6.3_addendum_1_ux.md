# F6.3 — Addendum 1 (UX: match invisible, alta pre-clasificada)

> Modifica el flujo de PropuestaCard en src/vistas/Comprobantes.tsx. No cambia la lógica server-side
> del match (el trigger sigue calculando propuestaMatch igual); cambia SOLO cómo se le presenta al usuario.

## Principio
El usuario NO ve "match / sin match / rama N". Para él el flujo es siempre el mismo:
**subo → se abre el alta PRE-CLASIFICADA (descripción + categoría + subcategoría + persona del diccionario)
→ reviso → confirmo.** Si por detrás matcheó un esperado o no, es mecánica interna. El usuario se entera
de los matches con esperados en la **hoja Resumen**, no acá.

## Qué cambia en PropuestaCard
La lógica de ramas sigue por detrás (estampar `itemEsperadoId`, `hashPdf`, `confirmadoPago` según
corresponda), pero la PRESENTACIÓN se unifica:

- **Eliminar el lenguaje de plomería**: nada de "Sin match detectado", "Movimiento detectado",
  "Ya vinculado al movimiento", "candidatos — seleccioná". El usuario no lee la palabra "match".
- **Indicador único "pre-clasificado"**: cuando el diccionario sugirió cat/subcat, mostrar un sello sutil
  tipo "Pre-clasificado" (que el sistema completó campos), no un veredicto de match.
- **Un solo gesto**: abrir el alta pre-cargada con lo que el sistema pudo completar (extracción + dict),
  editable, y confirmar. Mismo gesto en todas las ramas visibles.

### Por rama (lo que pasa por detrás, invisible al usuario)
- **Rama 0 (ya vinculado)**: el comprobante ya está atado a un movimiento → no re-proponer. Mostrar
  discreto "Ya cargado" sin lenguaje de match. (Único caso sin alta.)
- **Rama 1 (movimiento existe)**: en vez de "Movimiento detectado / Confirmar pago" expuesto, el sistema
  vincula por detrás (estampa `hashPdf` + `confirmadoPago` + `itemEsperadoId`) y lo refleja como cargado.
  Si hay múltiples candidatos, la selección puede quedar (es elección real), pero sin el rótulo "match".
- **Rama 2 (esperado matchea) y Rama 3 (nada matchea)**: para el usuario son IDÉNTICAS → abren el alta
  pre-clasificada. La diferencia (rama 2 pre-linkea `itemEsperadoId`, rama 3 no) es interna e invisible.
  Eliminar "Sin match detectado — alta manual"; reemplazar por abrir el alta directo (o un CTA neutro
  tipo "Revisar y cargar").

## El alta pre-clasificada (depende de F6.4.5 addendum 5 — ya cerrado)
- Abrir `AltaMovimiento` con preload = extracción (monto, fecha, descripción limpia, moneda) + sugerencia
  del diccionario (cat/subcat con umbral 0.7) + persona (quien subió). El preload nace con el dict ya
  cargado (addendum 5), así cat/subcat vienen llenas.
- Editable; al confirmar, aprende (trigger de movimientos, F6.4.5).

## Los esperados se ven en Resumen, no acá
- NO mostrar en Comprobantes si el movimiento confirmó un esperado. Esa información vive en ResumenMes
  (checklist de esperados, F5.5). El comprobante es para cargar el gasto bien categorizado; el estado del
  esperado es asunto del Resumen. Verificar que al confirmar (rama 1/2) el esperado correspondiente quede
  reflejado en Resumen (ya debería, por F5.5).

- Subir uno que SÍ matchea un esperado (ej. un servicio) → el usuario hace el mismo gesto (alta
  pre-clasificada); por detrás se estampa `itemEsperadoId`; verificar que en **Resumen** el esperado
  figura como cargado. El usuario nunca leyó "match".
