# F4.3 — Scope por persona (pantalla de hijos)

**Fase:** F4 · **Sub-paso:** 4.3 (cierra F4)
**Prerequisito:** F4.2 cerrado y verificado en browser.
**Scope:** Rules + índices + capa de datos + UI. Solo emulador.

---

## Contexto

Decisión cerrada: los dependientes (Federico, Sofía) ven SOLO sus movimientos; los admins (Juan, María) ven todo. La infraestructura ya existe: `/autorizados` con `rol` y `memberId`, helpers `esMiembro()`/`esAdmin()`/`miMemberId()` en Rules. NO se usan custom claims (decisión cerrada; plan B en F6 solo si Rules se sienten lentas).

La seguridad va en las RULES, no solo en la UI: un dependiente con la consola del navegador abierta no debe poder leer movimientos ajenos aunque manipule las queries.

---

## Qué hace F4.3

### 1. Rules — scope de lectura por colección

- `movimientos`:
  ```
  allow read: if esAdmin()
           || (esMiembro() && resource.data.persona == miMemberId());
  ```
- `resumenesTarjeta`, `itemsEsperados`, `diccionario`, `reglasNormalizacion`: lectura **solo admin** (los dependientes no las necesitan en F4; se revisa en F5 si su pantalla suma checklist propio).
- `tcDiario`, `subcategorias`, `etiquetas`, `config/familia`: lectura para cualquier `esMiembro()` (el dependiente necesita TC para equivalentes y catálogos para mostrar nombres).
- Escrituras: TODO sigue cerrado (F5).

### 2. Índice compuesto

La query del dependiente es `where('mes','==',X) && where('persona','==',Y)` → requiere índice compuesto `(mes, persona)` en `movimientos`. Agregarlo a `firestore.indexes.json`. OJO: el emulador es laxo con índices — agregarlo igual, porque en producción sin él la query muere. Reportar el índice agregado.

### 3. Capa de datos

- `movimientosDelMes(mes, persona?)`: si recibe `persona`, agrega el `where`. La vista decide según rol:
  - admin → sin persona (todo).
  - dependiente → SIEMPRE con su `memberId` (nunca opcional — si falta, las Rules rechazan la query entera, que es el comportamiento correcto: fail-closed).

### 4. UI

- **Dashboard**: mismo componente para todos. Para dependientes: titulado "Mis movimientos", totales y equivalentes calculados sobre lo suyo. Sin selector de "ver a otros".
- **Navegación**: el link a `/resumen` se oculta para dependientes, y la ruta misma redirige a `/` si un dependiente la fuerza (defensa en UI; las Rules ya niegan los datos igual).
- Nada nuevo para admins: ven lo mismo que hasta ahora.

### 5. Pruebas (las dos direcciones)

- Cuenta fake con email de Federico: ve SOLO sus movimientos (86 en el snapshot actual), totales coherentes con eso.
- Desde la consola del navegador logueado como Federico: intentar `getDocs` de movimientos SIN filtro de persona → **permission denied** (probar y documentar cómo).
- Forzar `/resumen` como Federico → redirige, y un `getDocs` directo a `itemsEsperados` → denied.
- Admin (Juan): todo igual que antes, sin regresión.
- `npm run validate`: 9 OK / 0 FAIL (el seed no se toca en este paso... salvo que la decisión de `mes` pendiente entre acá — ver nota).
