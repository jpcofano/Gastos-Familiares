# F9.88 — Notificaciones: usar los esperados de Resumen (no exigir `diaVencimiento`)

> `src/vistas/perfil/Notificaciones.tsx` → `recordatoriosEsperados` + tipo `Recordatorio` + `RecordatorioRow`.
> Reusa `calcularChecklist` / `cubierto` / `ACCIONABLE` de `src/datos/checklist.ts` (ya compartidos con Resumen).

---

## Problema (confirmado por log)

`itemsGastoConVenc: 1` de 24: casi ningún pago esperado tiene `diaVencimiento`. Hoy `recordatoriosEsperados` hace
`if (item.tipo !== 'Gasto' || item.diaVencimiento == null) continue;` → descarta los 23 sin día y la pantalla queda
casi vacía. **Decisión:** las notificaciones deben reflejar lo mismo que **Resumen › esperados del mes** marca como
pendiente — usando el **estado del checklist**, no el campo `diaVencimiento` (que es opcional y casi nunca está).

## §1 · Nuevo criterio de inclusión

Incluir todo ítem cuyo estado de checklist sea **accionable** (lo que Resumen muestra como pendiente):
`pendiente`, `vencido`, `por_confirmar`, `parcial`. `diaVencimiento` pasa a ser **opcional** (mejora la fila, no la
condición). La ventana de 14 días se aplica **solo a los ítems que sí tienen día**; los sin día son "esperados del mes"
y se muestran igual (como en Resumen).

```ts
import { calcularChecklist, cubierto, ACCIONABLE, mesActualStr } from '../../datos/checklist';

function recordatoriosEsperados(items, movs, hoy, navigate) {
  const checklist = calcularChecklist(items, movs, mesActualStr());
  const out = [];
  for (const { item, estado } of checklist) {
    if (item.tipo !== 'Gasto') continue;
    if (!ACCIONABLE.includes(estado)) continue;          // pendiente | vencido | por_confirmar | (parcial vía ACCIONABLE)
    let fecha = null, estadoRec;
    if (item.diaVencimiento != null) {
      fecha = new Date(hoy.getFullYear(), hoy.getMonth(), item.diaVencimiento);
      const dias = diffDias(fecha, hoy);
      estadoRec = estadoPorDias(dias);
      if (estadoRec === 'proximo' && dias > DIAS_VENTANA) continue;   // fuera de ventana SOLO si tiene día
    } else {
      estadoRec = estado === 'vencido' ? 'vencido' : 'proximo';       // sin día: esperado del mes, siempre visible
    }
    out.push({
      id: `esp-${item.id}`, tipo: 'esperado',
      titulo: [item.categoria, item.subcategoria].filter(Boolean).join(' › ') || item.notas || '(sin categoría)',
      sub: estado === 'por_confirmar' ? 'Pago detectado — falta confirmar' : estado === 'parcial' ? 'Pago parcial detectado' : undefined,
      fecha, estado: estadoRec,
      montos: item.montoEsperado != null ? [{ monto: item.montoEsperado, moneda: item.moneda }] : [],
      onTap: () => navigate('/resumen', { state: { sec: 'fijos' } }),
    });
  }
  return out;
}
```

- `ACCIONABLE` en checklist.ts ya es `['pendiente','vencido','no_registrado','por_confirmar']`. Para notificaciones
  del mes en curso **excluí `no_registrado`** (es de meses pasados) — filtralo aparte o usá una lista local
  `['pendiente','vencido','por_confirmar','parcial']`. Elegí lo que deje la lista igual a lo que muestra Resumen para el mes actual.

## §2 · `Recordatorio.fecha` opcional + fila sin día

```ts
interface Recordatorio { …; fecha: Date | null; … }
```
En `RecordatorioRow`, cuando `r.fecha == null`, en vez del número de día grande mostrar un marcador neutro
(ej. un punto del color del estado + "Este mes", o el ícono de recibo), y el badge sigue: `vencido`→danger,
`proximo`→neutral. Cuando hay fecha, se mantiene el número de día + mes como está.

- El badge de la campana (`contarVencProximos` = vencido + hoy) no se infla con los sin-día (van como `proximo`).
- Ordenar: vencido → hoy → proximo; dentro de proximo, primero los que tienen fecha (por día), después los sin día.

## §3 · Tarjetas — sin cambios

`recordatoriosTarjetas` queda igual (usa `fechaVencimiento` / `venceDia`, que sí existen).

## §4 · (Aparte) el índice de Firestore

Los errores rojos `The query requires an index` en `/comprobantes` y `/perfil` son otra cosa (una query que falla
devuelve 0). Crear el índice desde el link azul de la consola y versionarlo en `firestore.indexes.json`. No bloquea
este cambio, pero conviene cerrarlo.
