# F9.31 — Verificación de datos reales (gate pre-migración)

> Doc + auditoría, no migra nada. Extiende el script de verificación ya existente
> (`npm run validate` → `scripts/seed/validators/checks.ts`) en vez de crear uno paralelo —
> ya era exactamente "script de verificación read-only sobre emulador/seed, NO prod" que
> pedía este gate, solo le faltaban los checks de esta fase.

## Qué se agregó a `checks.ts` (12 → 19 validadores)

Los 12 originales ya cubrían conteos/sumas/`mes==fecha`/`autorizados` contra la planilla
legacy. F9.31 agrega los 7 que el gate pedía explícitamente y que no estaban:

1. **`persona == memberId (100%)`** — todo `movimientos.persona` debe ser una key de
   `config/familia.miembros`. Si encuentra valores no resueltos, lista hasta 5 y apunta a
   `scripts/seed/backfillPersonaMemberId.ts` (F9.24, todavía no corrido).
2. **`banco en set canónico`** — todo `movimientos.banco` debe ser uno de
   `BBVA/Galicia/Personal Pay/Efectivo`. Nota: Efectivo se valida tal cual está guardado —
   el alias a Mercado Pago (F9.23) es solo de *display*, no se espera que el dato cambie.
3. **`categoria en taxonomía conocida`** — toda `movimientos.categoria` debe estar en
   `config/familia.categorias`.
4. **`cuotaActual <= cuotaTotal`** — recorre `resumenesTarjeta[].movimientosParseados[]`.
5. **`resumenesTarjeta cuadre (±1)`** — recalcula Σ líneas (consumo/cuota suman, reverso/
   bonificación/reintegro restan) + ajustes consolidado, y compara contra `totalARS`/
   `totalUSD` con la misma tolerancia ($1/U$S1) que usa `confirmarResumenTarjeta` en
   producción.
6. **`tcDiario en rango razonable (500-5000)`** — sanity check, no exacto (el MEP real
   fluctúa); solo atrapa valores corruptos (0, negativos, texto mal parseado).
7. **`roles: 2 admin + 2 dependiente`** — valida la distribución de roles activos en
   `config/familia.miembros`, no solo el conteo total (que ya validaba el check #8 original).

No agregué un check de cobertura de fechas de `tcDiario` (punto 4 de la espec original) como
validador aparte porque el check #8 original (`movimientos sin TC`) ya cubre el síntoma real
que importa: un movimiento en USD sin `tcUsdArs` resuelto. Verificar "huecos" en `tcDiario`
en abstracto sin que afecten a ningún movimiento no es información que cambie una decisión
de migración.

## Por qué no corrí el script

`npm run validate` necesita el emulador corriendo con el seed ya cargado — un estado que no
puedo verificar ni reproducir yo mismo en este entorno, y corresponde a "vos corrés
seed/emulador manualmente" (regla del repo). El código de los 19 checks está listo y
compila (`tsc --noEmit` sobre el archivo, sin errores nuevos más allá de los que ya tenía
el archivo original por la forma en que tipa `allMovs` — no son de esta fase).

## Cómo correrlo

```
firebase emulators:start --only auth,firestore   # en otra terminal, con --import si hay seed previo
npm run seed          # si el emulador está vacío
npm run validate
```

Salida esperada: `19 OK / 0 FAIL`. Cualquier FAIL imprime el detalle (qué docs, qué
diferencia) — no hace falta abrir Firestore a mano para diagnosticar.

## Datos NUEVOS (no vienen de legacy — no se concilian contra la planilla)

Lista de lo que el sistema nuevo agrega y se valida por su propia regla, no contra Excel:

| Qué es | De dónde sale el valor inicial | Regla de validación propia |
|---|---|---|
| `resumenesTarjeta[].movimientosParseados[].cuotaActual/cuotaTotal/montoCuota` | Extracción Claude del PDF (F9.13/F9.21) | Checks #4 y #5 de arriba (consistencia interna + cuadre) |
| `itemsEsperados[].periodicidad`/`pagoAutomatico` | Default `'mensual'`/`false` al seedear; editable por admin | Sin validación automática hoy — `periodicidad` no-mensual es placeholder (`aplicaEnMes` devuelve `true` siempre, ver `docs/CLAUDE.md`) |
| `destinos/{idNorm}` (aprendizaje de payee) | Se puebla solo al confirmar movimientos con campos `destino*` (F6.8) | No tiene contraparte legacy; no se valida en este gate (no afecta migración de `movimientos`) |
| `diccionario` (aprendizaje de clasificación) | Seed (~479) + trigger de aprendizaje en uso | Check #3 original (`diccionario count >= excel - 15`) ya lo cubre |
| `datos/medios.ts` (metadata de display: color/dominio/tipo por medio) | Curado a mano en código (F9.23), no Firestore | No aplica un check de datos — es código de la app, no un dato a migrar |

Ninguno de estos bloquea la migración de `movimientos`/`resumenesTarjeta`/`itemsEsperados`/
`tcDiario`/`usuarios` (que es lo que migra la Etapa B) — son features del sistema nuevo, no
datos heredados.
